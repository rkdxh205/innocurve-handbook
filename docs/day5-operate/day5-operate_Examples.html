● 초급 Getting Started 내 봇이 스스로 문제를 알리도록 첫 안전장치를 붙이는 단계
G-1 초급 내 봇에 오류 알림 기능 추가하기
지금까지 만든 뉴스 알림 봇은 며칠째 잘 작동하고 있지만, 네이버 뉴스 검색 API 서버에 일시적 문제가 생기거나 코드에 실수가 있으면 조용히 멈춰버릴 수 있습니다. "며칠 전부터 알림이 안 왔는데 이제야 알아챘다"는 상황을 막기 위해, 오류가 나면 스스로 텔레그램으로 알려주는 기능을 추가하려 합니다.
사용 요소
try/except 오류 처리 Secrets 원칙 재확인
단계별 실행
  1. 오류가 날 수 있는 부분을 try로 감싸기

    기존 코드에서 뉴스를 가져와 텔레그램으로 보내는 핵심 로직을 try 블록 안으로 옮깁니다.

    try:
    articles = fetch_news(my_name)
    send_new_articles_to_telegram(articles)
  2. except 블록에 오류 알림 코드 추가하기

    오류가 발생하면 시각, 오류 종류, 오류 메시지를 담아 send_telegram_message로 전송하도록 작성합니다.

    except Exception as e:
    error_time = datetime.now().strftime("%Y-%m-%d %H:%M:%S")
    error_message = (
        f"[뉴스봇 오류 발생]\n시각: {error_time}\n"
        f"오류 종류: {type(e).__name__}\n오류 메시지: {e}"
    )
    send_telegram_message(error_message)
  3. 일부러 오류를 내서 테스트하기

    코드 한 줄을 임시로 바꿔 존재하지 않는 변수를 참조하게 만든 뒤 실행해보고, 텔레그램에 오류 알림이 도착하는지 확인합니다. 확인 후에는 반드시 원래 코드로 되돌립니다.

    # 테스트용으로 잠시 추가 (확인 후 반드시 삭제)
articles = fetch_news(존재하지_않는_변수)
오류가 발생하면 파이썬은 즉시 except 블록으로 실행을 넘깁니다. 이 블록 안에서도 정상적으로 텔레그램 API를 호출할 수 있으므로, "오류가 났다"는 사실 자체를 알림으로 보낼 수 있습니다. 같은 오류는 GitHub Actions 로그에도 남으므로 두 군데에서 교차 확인이 가능합니다.
오류 알림용 코드는 send_telegram_message처럼 별도 함수로 만들어두면, 평소 정상 알림과 오류 알림 모두에서 재사용할 수 있어 편리합니다.
● 중급 Daily Workflow 운영 기록을 점검하고, 발견한 문제를 구체적으로 설명해 개선을 요청하는 단계
D-1 중급 실행 로그 점검 후 Claude Code에게 개선 요청하기
봇을 며칠간 운영한 뒤, GitHub Actions의 실행 이력을 열어 셀프 리뷰 체크리스트 항목을 하나씩 확인해봅니다. 점검 과정에서 "가끔 같은 기사가 두 번 오는 것 같다"는 문제를 발견했습니다. 이 문제를 직접 코드로 고치는 대신, Claude Code에게 상황을 구체적으로 설명하고 개선을 요청하려 합니다.
사용 요소
셀프 리뷰 체크리스트
단계별 실행
  1. GitHub Actions 실행 이력 열어보기

    저장소의 Actions 탭에서 최근 실행 목록을 확인하고, 각 실행의 로그를 열어 어떤 기사가 발송되었는지 확인합니다.

  2. 체크리스트 항목별로 점검하기

    실행 시각, 중복 발송, 동명이인 필터링, 오류 알림, 이 네 항목을 하나씩 대조하며 발견한 문제를 메모해둡니다.

  3. 발견한 문제를 Claude Code에게 구체적으로 설명하기

    현상을 최대한 구체적으로 전달할수록 더 정확한 수정을 받을 수 있습니다.

    최근 3일치 GitHub Actions 로그를 보니
같은 기사가 두 번씩 텔레그램으로 발송된 걸 발견했어.
이미 보낸 기사인지 확인하는 로직이 없는 것 같은데,
기사 링크를 저장해뒀다가 중복이면 보내지 않도록 코드를 고쳐줘.
  4. 수정된 코드 확인 후 다시 실행해 검증하기

    Claude Code가 수정한 코드의 내용을 확인하고, 다음 실행에서 같은 기사가 중복 발송되지 않는지 다시 점검합니다.

Claude Code는 문제 상황을 코드가 아니라 "현상"으로 설명해도 원인을 추론해 관련 코드를 찾아 수정할 수 있습니다. 다만 "며칠 몇 시에 어떤 기사가 어떻게 중복됐는지"처럼 구체적인 사실을 전달할수록 더 정확한 수정을 받을 수 있습니다.
로그나 코드를 Claude Code에게 붙여넣을 때, 텔레그램 봇 토큰이나 API 키 같은 값이 로그에 그대로 노출되어 있지 않은지 반드시 확인한 뒤 전달하세요.
● 고급 Power User 직접 만든 경험을 근거로 조직 전체의 AI 도입 로드맵을 설계하는 단계
P-1 고급 AI 도입 3-6-12개월 로드맵 초안 작성하기
5일간의 실습을 마무리하며, 처음부터 끝까지 직접 만들어본 경험을 근거로 자신이 속한 부처·기관 전체에 AI를 어떻게 도입할지 로드맵을 작성해야 하는 상황입니다. Claude에게 구체적인 프롬프트로 요청해 초안을 받고, 발표를 위해 다듬으려 합니다.
사용 요소
AI 도입 리더십 성찰
단계별 실행
  1. 로드맵 프롬프트를 작성해 Claude에게 요청하기

    내 부처·기관명과 5일간의 실습 경험을 근거로 요청하면 더 구체적인 초안을 받을 수 있습니다.

    너는 공공부문 디지털 전환 전략 자문가야.
이번 5일간 내가 직접 만든 "내 이름 뉴스 알림 텔레그램 봇" 경험을 바탕으로
[부처/기관명]에 AI 활용을 도입하는 3-6-12개월 로드맵을 작성해줘.

- 3개월: 시범 도입할 부서와 업무
- 6개월: 확산 조건과 대상
- 12개월: 정착을 위한 성과지표와 거버넌스
각 단계의 예상 리스크와 대응 방안도 함께 적어줘.
  2. 받은 초안을 내 조직 상황에 맞게 검토하기

    로드맵 초안에서 제시된 시범 업무·확산 조건·성과지표가 실제 우리 부처·기관 상황과 맞는지 검토하고, 맞지 않는 부분은 구체적으로 수정을 요청합니다.

  3. 발표 자료 형태로 다듬기

    3개월·6개월·12개월 표 형태로 정리하고, 각 단계의 리스크와 대응 방안을 함께 넣어 발표 슬라이드나 문서로 정리합니다.

Claude는 일반적인 3-6-12개월 프레임에 내가 입력한 맥락(부처·기관명, 5일간 겪은 구체적 경험)을 결합해 초안을 만듭니다. 로드맵의 설득력은 결국 "내가 직접 해봤다"는 경험에서 나오므로, 프롬프트에 실습 중 겪은 구체적 에피소드를 추가할수록 더 설득력 있는 초안을 받을 수 있습니다.
로드맵 초안은 한 번에 완성되지 않습니다. "리스크 대응 방안을 더 구체적으로 적어줘", "우리 부처의 조직 특성(예: 산하기관이 많음)을 반영해줘"처럼 후속 질문을 이어가며 다듬는 것이 좋습니다.
함께 보기: 매뉴얼 문서의 "조직 차원 고려사항"(데이터 보안, 책임소재, 거버넌스, 조직 변화관리)을 로드맵에 반영하면 더 완성도 높은 발표가 됩니다.
