1. 오류 처리와 알림 설계

try/except — 자동화된 도구가 조용히 멈추지 않게 만들기

GitHub Actions로 매일 자동 실행되는 봇은 사람이 화면 앞에서 지켜보고 있지 않다는 점이 특징입니다. 코드 어딘가에서 오류가 발생하면 실행이 조용히 중단되고, 아무도 눈치채지 못한 채 며칠이 지날 수 있습니다. "자동화된 도구는 조용히 멈추면 아무도 모른다"는 원칙을 기억해야 합니다. try/except 구문은 오류가 발생할 수 있는 코드를 감싸서, 오류가 나더라도 프로그램이 완전히 멈추기 전에 무슨 일이 있었는지를 붙잡아 처리할 기회를 줍니다.

import traceback
from datetime import datetime

try:
    articles = fetch_news(my_name)          # 네이버 뉴스 검색 API 호출
    send_new_articles_to_telegram(articles)  # 새 기사만 골라 텔레그램으로 전송
except Exception as e:
    error_time = datetime.now().strftime("%Y-%m-%d %H:%M:%S")
    error_message = (
        f"[뉴스봇 오류 발생]\n"
        f"시각: {error_time}\n"
        f"오류 종류: {type(e).__name__}\n"
        f"오류 메시지: {e}"
    )
    send_telegram_message(error_message)     # 오류 자체를 텔레그램으로 알림
    print(traceback.format_exc())            # GitHub Actions 로그에도 자세히 남김

오류 알림에 담을 정보 — 시각·종류·메시지, 이 세 가지면 충분합니다

오류 알림 메시지가 너무 길거나 기술적인 내용으로 가득하면 오히려 무엇이 문제인지 파악하기 어려워집니다. 아래 세 가지 정보만 담아도 "언제, 어떤 문제가, 어떻게" 발생했는지 충분히 파악할 수 있고, 필요하면 GitHub Actions 로그를 열어 더 자세한 내용을 확인하면 됩니다.

담을 정보설명예시
시각 오류가 발생한 정확한 날짜와 시간 2026-07-05 09:03:12
오류 종류 파이썬이 자동으로 분류해주는 예외 유형 ConnectionError, KeyError
오류 메시지 파이썬이 알려주는 구체적인 실패 원인 요청 시간 초과(Timeout), 잘못된 요청(401 Unauthorized)

💡 핵심 한 줄: 오류를 완벽하게 막는 것보다, 오류가 났을 때 즉시 알아차릴 수 있게 만드는 것이 더 현실적이고 중요한 목표입니다.

2. 셀프 리뷰 체크리스트

며칠간 봇을 운영해보면 실제로 어떻게 작동하고 있는지 스스로 점검해볼 시점이 옵니다. GitHub Actions 실행 이력과 텔레그램 알림 내역을 직접 열어보며, 아래 항목을 하나씩 확인해보세요.

실행 시각 점검 — 예상한 시간에 실행되었는가

저장소의 Actions 탭에서 워크플로우 실행 이력을 열어보면, cron으로 설정한 시각에 맞춰 실제로 실행되었는지 확인할 수 있습니다. GitHub Actions의 스케줄은 지정한 시각 근처에서 몇 분 정도 지연될 수 있으므로, 정확히 그 분(分)이 아니라 "대략 그 시간대"에 매번 실행되었는지를 기준으로 판단합니다.

중복 발송 점검 — 같은 기사가 반복해서 오지 않는가

하루에 여러 번 실행되는 봇이라면, 이미 보낸 기사인지 판단하는 로직(예: 기사 링크를 저장해두고 비교하는 방식)이 없으면 같은 기사를 여러 번 보내게 됩니다. 텔레그램 대화 내역을 스크롤하며 동일한 제목·링크가 중복되지 않는지 확인합니다.

동명이인 필터링 점검 — 관련 없는 인물의 기사가 걸러지는가

이름만으로 검색하면 동명이인 기사가 섞여 들어올 수 있습니다. Day 3에서 추가한 키워드 필터(소속 기관명 등)가 실제로 관련 없는 기사를 걸러내고 있는지, 최근 알림 내역을 보며 점검합니다.

오류 알림 점검 — 오류 상황에서 알림이 정상 작동하는가

오류 알림 로직은 직접 테스트해볼 수 있습니다. 예를 들어 코드에 일부러 존재하지 않는 변수를 참조하게 만들어 실행해보고, 텔레그램으로 오류 메시지가 도착하는지 확인한 뒤 다시 원래대로 되돌립니다.

점검 항목확인 방법정상 판단 기준
실행 시각 GitHub Actions 실행 이력(Actions 탭) 설정한 cron 시각 근처에 매번 실행됨
중복 발송 여부 텔레그램 대화 내역 스크롤 확인 동일한 제목·링크의 기사가 반복 발송되지 않음
동명이인 필터링 최근 알림 기사 내용 확인 소속·직함이 다른 동명이인 기사가 없음
오류 알림 동작 의도적 오류 테스트 후 텔레그램 확인 오류 발생 시 알림이 정상적으로 도착함

점검 결과는 아래와 같은 상태 표시를 활용해 간단히 기록해두면 이후 관리가 편해집니다.

3. 개인정보·보안 고려사항

Secrets 원칙 재확인 — 토큰과 키는 코드에 절대 직접 쓰지 않는다

Day 4에서 설정한 것처럼, TELEGRAM_BOT_TOKEN, NAVER_CLIENT_ID, NAVER_CLIENT_SECRET 같은 민감한 값은 반드시 GitHub Secrets에만 저장하고, 코드에는 환경변수로 불러오는 방식만 사용합니다. 실수로 코드에 값을 직접 적어 넣고 그대로 커밋하면, 이후에 값을 지워도 커밋 기록에는 남아 있을 수 있으므로 처음부터 하드코딩하지 않는 습관이 중요합니다.

저장소 공개 범위 판단 — Private로 유지할지 Public으로 바꿔도 될지

코드 자체(로직)는 공개해도 무방하지만, 저장소 안에 실행 로그나 캐시 파일 등 민감할 수 있는 부산물이 남아있지 않은지 먼저 점검해야 합니다. 특별한 이유가 없다면 Private 상태를 유지하는 편이 안전합니다.

판단 기준Private 유지Public 전환
개인 신상 관련 코드·로그가 저장소에 존재 권장 비권장
과거 커밋 기록에 민감한 키가 남아있음 반드시 유지 불가
다른 사람이 참고하도록 코드만 공유하고 싶음 - 부산물 제거 후 검토 가능

뉴스 스크랩 결과 속 개인정보 주의 — 검색된 기사 내용에 섞일 수 있는 정보

내 이름을 검색해 나온 기사에는 소속, 직함, 발언 내용 등 개인과 관련된 정보가 자연스럽게 포함됩니다. 이 정보를 텔레그램 대화방 밖으로 캡처해 공유하거나, 관련 없는 제3자와 공유하는 과정에서 의도치 않게 개인정보가 노출될 수 있다는 점을 유의해야 합니다.

봇이 보내주는 알림은 어디까지나 "나만 보는 개인용 알림"이라는 전제로 설계되었습니다. 여러 사람이 보는 채널이나 그룹으로 무심코 연결하지 않도록 주의하세요.

4. AI 도입 리더십 성찰

3-6-12개월 로드맵 프레임 — 직접 만들어본 경험을 조직 전체로 확장하기

지난 5일간 처음부터 끝까지 직접 코드를 작성하고, 오류를 만나고, 고쳐본 경험은 "AI 도구가 실제로 무엇을 할 수 있고 무엇을 못하는지"에 대한 감각을 줍니다. 이 감각을 근거로, 3개월·6개월·12개월 단위로 나눠 부처·기관 전체의 AI 도입 로드맵을 설계할 수 있습니다.

시기핵심 질문예시
3개월 어떤 부서의 어떤 업무부터 시범 도입할 것인가 반복적 문서 분류, 문의 응대 초안 작성 등 정형화된 업무 1~2개 선정
6개월 시범 결과를 다른 부서로 확산할 조건은 무엇인가 오류 처리·점검 체계가 갖춰진 업무부터 순차 확산
12개월 정착을 위한 거버넌스와 성과지표는 무엇인가 업무 처리 시간 단축률, 오류 발생률, 담당 조직 지정

조직 차원 고려사항 — 개인 실습을 넘어설 때 반드시 짚어야 할 것들

개인이 혼자 만든 봇과 달리, 조직 전체에 AI를 도입할 때는 아래와 같은 조직 차원의 고려사항이 함께 다뤄져야 합니다.

  • 데이터 보안·개인정보: 민감정보는 외부 AI 서비스에 직접 입력하지 않도록 명확한 지침이 필요합니다.
  • 책임소재: "AI 초안 → 사람 최종 검수"라는 원칙을 문서화해 모든 구성원이 같은 기준을 갖게 합니다.
  • 거버넌스: 허용 업무와 금지 업무의 범위를 정하고, 성과를 측정·관리할 담당 조직을 지정합니다.
  • 조직 변화관리: AI 도구는 반복 업무를 줄여 구성원이 더 중요한 판단 업무에 집중하도록 돕는 수단이라는 메시지를 일관되게 전달합니다.

💡 핵심 한 줄: 이번 5일 실습에서 직접 겪은 시행착오 하나하나가, 앞으로 부처·기관에 AI를 도입할 때 다른 사람들이 겪을 어려움을 미리 예상하게 해주는 자산입니다.
