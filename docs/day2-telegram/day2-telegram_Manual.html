0. Telegram Desktop 설치 & GitHub 계정 만들기

Telegram Desktop 설치 — 노트북만으로 설치부터 가입까지 끝냅니다

이번 실습은 노트북(데스크톱) 환경만으로 진행합니다. 스마트폰 앱을 따로 설치할 필요는 없지만, 가입 과정에서 인증 코드를 문자(SMS)로 받아야 하므로 본인 명의 휴대폰 번호는 그대로 필요합니다 — 텔레그램은 데스크톱이든 모바일이든 계정을 처음 만들 때는 항상 전화번호로 인증 코드를 보내는 방식이기 때문입니다. 즉 "스마트폰에 텔레그램 앱을 설치"하는 단계만 건너뛸 뿐, 인증 문자를 받을 내 폰은 옆에 두고 진행합니다.

  1. 데스크톱 설치 파일 받기: telegram.org/apps에서 Windows(또는 macOS)용 "Telegram Desktop"을 내려받아 설치합니다. Microsoft Store에서 "Telegram Desktop"을 검색해 설치해도 동일합니다.
  2. 앱 실행 후 전화번호 입력: 설치한 앱을 실행하면 국가 코드와 휴대폰 번호를 입력하는 화면이 뜹니다. 본인 휴대폰 번호를 입력합니다.
  3. 문자로 온 인증 코드 입력: 입력한 번호로 문자 메시지(SMS)가 도착합니다. 이 코드를 데스크톱 앱 화면에 그대로 입력하면 가입이 완료됩니다. (스마트폰에 텔레그램 앱이 없어도 문자 수신만 되면 문제없습니다.)
  4. 이름 설정: 표시될 이름(성·이름)을 입력하면 계정 생성이 끝나고, 곧바로 데스크톱 앱 화면에서 대화를 시작할 수 있습니다.
간혹 "스마트폰으로 먼저 가입하고 데스크톱으로 넘어온다"고 알고 있는 경우가 있는데, 실제로는 데스크톱 앱에서 곧바로 가입까지 끝낼 수 있습니다. 다만 인증 코드 수신을 위한 휴대폰 번호 자체는 데스크톱/모바일 여부와 무관하게 항상 필요합니다.

💡 참고: 오늘 실습은 노트북 화면에서 보기 편하도록 데스크톱 앱으로 진행하지만, 봇이 완성된 뒤에는 같은 계정으로 스마트폰에도 텔레그램 앱을 설치해두면 외출 중에도 편하게 뉴스 알림을 받아볼 수 있습니다. (계정은 전화번호 하나에 연결되어 있어 앱만 추가로 설치하면 기기 간 대화 내용이 그대로 동기화됩니다.)

GitHub 계정 만들기 — 이후 실습을 위해 미리 준비해두는 계정

GitHub는 코드를 저장하고 버전을 관리하는 서비스로, 오늘 10단계에서 이 계정으로 저장소를 만들어 봇의 코드를 올리고 GitHub Actions로 자동 실행시키는 데 사용합니다. 계정 생성 자체가 이메일 인증 등으로 시간이 걸릴 수 있으므로 본격적인 봇 만들기에 앞서 지금 미리 만들어 둡니다.

  1. github.com 접속: 브라우저에서 github.com에 접속해 Sign up 버튼을 클릭합니다.
  2. 계정 정보 입력: 이메일, 비밀번호, username을 입력합니다. username은 영문으로 만들어두면 이후 설정이 수월합니다.
  3. 인증 완료: 화면 안내에 따라 이메일 인증 등 본인 확인 절차를 마치면 가입이 완료됩니다.
GitHub 계정으로 무엇을 언제 하게 될지는 이후 실습 단계에서 자세히 안내합니다. 오늘은 "로그인 가능한 계정 하나를 만들어 둔다"까지만 완료하면 충분합니다.

1. 텔레그램이란, 왜 텔레그램으로 알림을 받는가

Telegram — 전 세계에서 널리 쓰이는 메신저이자 개방형 봇 플랫폼

텔레그램은 카카오톡과 비슷한 메신저 앱이지만, 처음부터 "봇(bot)"을 자유롭게 만들어 붙일 수 있도록 설계된 것이 큰 차이점입니다. 누구나 무료로 계정을 만들고, 자신만의 자동화 프로그램을 봇 형태로 연결해 메시지를 주고받을 수 있습니다. 이번 핸드북에서는 이 봇을 "내 이름이나 관심 키워드가 언급된 뉴스를 알려주는 개인 비서"로 활용합니다.

Bot API — 텔레그램이 공식 제공하는 봇 프로그래밍 인터페이스

Telegram Bot API는 별도의 서버나 복잡한 설정 없이도, 인터넷 요청 한 번으로 메시지를 보내고 받을 수 있게 해주는 공식 인터페이스입니다. 로그인 절차나 승인 심사가 없고, 봇 하나당 발급되는 "토큰" 하나만 있으면 바로 사용할 수 있어 진입 장벽이 매우 낮습니다.

💡 핵심 한 줄: 텔레그램은 API가 공개돼 있고 무료이며, GitHub Actions 같은 자동화 도구와도 궁합이 좋아 "개인 알림 채널"로 안성맞춤입니다.

비교 항목텔레그램카카오톡·문자(SMS)
봇 API 공개 여부 공식적으로 완전히 공개, 무료 개인 개발자용 공개 API가 제한적이거나 유료
알림 자동화 난이도 토큰 하나로 즉시 메시지 발송 가능 별도 심사·계약이 필요한 경우가 많음
이번 핸드북과의 궁합 GitHub Actions의 정기 실행 결과를 그대로 전달하기 적합 해당 없음

2. BotFather로 봇 만들기

@BotFather — 모든 텔레그램 봇을 만들고 관리하는 공식 봇

BotFather는 텔레그램이 공식으로 운영하는 "봇을 만들어주는 봇"입니다. 새 봇을 등록하거나, 이름을 바꾸거나, 프로필 사진을 설정하는 등 봇 관리에 필요한 모든 명령을 대화 형태로 처리합니다.

  1. @BotFather 검색: 텔레그램 앱 상단 검색창에 @BotFather를 입력해 공식 계정(파란 인증 배지가 있는 계정)을 찾습니다.
  2. 대화 시작: BotFather와의 대화방에 들어가 시작(Start) 버튼을 눌러 대화를 엽니다.
  3. /newbot 전송: 새 봇을 만들겠다는 명령을 보냅니다.
  4. 표시 이름 입력: 대화 목록 등에 보일 봇의 이름을 입력합니다. (예: "홍길동 뉴스 알림봇" — 한글도 가능하며 나중에 변경할 수 있습니다.)
  5. username 입력: 봇의 고유 주소가 되는 username을 입력합니다. 아래 규칙을 따라야 하며, 한 번 정하면 바꿀 수 없습니다.

username 규칙 — 한 번 정하면 바꿀 수 없는 봇의 고유 주소

BotFather는 입력한 username이 규칙에 맞고 아직 아무도 쓰지 않은 이름일 때만 봇 생성을 완료합니다.

규칙내용비고
길이 5자 이상 32자 이하 너무 짧으면 거부됩니다
허용 문자 영문 알파벳, 숫자, 밑줄(_)만 가능 한글·공백·특수문자 불가
끝 글자 반드시 bot으로 끝나야 함 예: gildong_news_bot
변경 가능 여부 한 번 정하면 이후 변경 불가 신중하게 정하는 것을 권장합니다
username은 전 세계 사용자와 공유하는 이름 공간이라 이미 다른 사람이 쓰고 있으면 등록되지 않습니다. 여러 후보를 미리 생각해 두는 것이 좋습니다.

3. 봇 토큰이란 무엇인가

Bot Token — 봇의 신원과 권한을 증명하는 비밀 문자열

username 등록이 끝나면 BotFather는 곧바로 "봇 토큰(token)"을 발급해 대화창에 보여줍니다. 이 토큰은 앞으로 프로그램이 "나는 이 봇이다"라고 증명할 때 쓰는 열쇠이며, 토큰만 있으면 누구든 그 봇의 이름으로 메시지를 보내거나 받을 수 있습니다.

토큰은 다음과 같은 형식으로 발급됩니다.

110201543:AAHdqTcvCH1vGWJxfSeofSAs0K5PALDsaw

앞부분의 숫자(110201543)는 봇의 고유 ID이고, 콜론(:) 뒤의 약 35자 문자열이 실제 비밀 값입니다. (※ 위 예시는 텔레그램 공식 튜토리얼에 나오는 예시 형식이며, 실제로 발급받은 토큰은 이와 다른 값입니다.)

봇 토큰은 집 열쇠나 비밀번호와 동일하게 취급해야 합니다. GitHub 등 공개 저장소 코드에 그대로 적어 올리거나, 화면 공유·발표 중에 화면에 노출하지 않도록 각별히 주의하세요. 토큰이 유출되면 즉시 BotFather에서 /revoke 명령으로 재발급받아야 합니다.

💡 핵심 한 줄: 발급받은 토큰은 메모장 등에 안전하게 적어두고, 이후 실습에서 코드나 자동화 워크플로우의 비밀 값(secret)으로 사용하게 됩니다.

4. Telegram Bot API 기초

모든 Bot API 호출은 아래와 같이 토큰을 포함한 하나의 주소(URL) 형태를 따릅니다. 이 주소에 필요한 값(파라미터)을 담아 요청을 보내면, 텔레그램 서버가 그 결과를 돌려줍니다.

https://api.telegram.org/bot<여기에_발급받은_BOT_TOKEN>/METHOD_NAME

HTTPS GET 또는 POST 토큰 필요

sendMessage — 봇이 특정 대화방에 메시지를 보내는 메서드

가장 기본적인 메서드로, 어느 대화방(chat_id)에 어떤 내용(text)을 보낼지 지정하면 그대로 메시지가 전송됩니다. 이번 핸드북의 최종 목표인 "뉴스 알림 보내기"의 핵심 동작이 바로 이 메서드입니다.

파라미터필수 여부설명
chat_id 필수 메시지를 받을 대화방의 고유 번호
text 필수 보낼 메시지 내용
parse_mode 선택 굵게·기울임 등 서식을 적용할지 지정 (예: Markdown, HTML)

getUpdates — 롱 폴링(long polling)으로 대기 중인 메시지를 가져오는 메서드

봇에게 도착했지만 아직 프로그램이 확인하지 않은 메시지들을 가져오는 메서드입니다. "롱 폴링"이란 서버에 접속해서 새 메시지가 올 때까지 짧게 기다렸다가 응답받는 방식으로, 별도의 웹 서버 없이도 메시지 수신이 가능하게 해줍니다.

파라미터기본값설명
offset 없음 이 번호 이후의 새 메시지만 가져오도록 지정 (이미 처리한 메시지를 다시 받지 않기 위함)
limit 100 한 번에 가져올 메시지 최대 개수 (1~100 사이)
timeout 0 새 메시지가 없을 때 몇 초 동안 대기할지 (초 단위)
allowed_updates 모든 종류 받고 싶은 업데이트 종류를 제한 (예: 일반 메시지만)
봇에 웹훅(webhook)이 설정돼 있으면 getUpdates는 동작하지 않습니다. 이번 핸드북에서는 웹훅을 설정하지 않으므로 문제없이 사용할 수 있습니다.

chat_id 알아내기 — 내 대화방 번호를 확인하는 표준 절차

메시지를 보내려면 상대방(자기 자신)의 chat_id를 알아야 합니다. 아래 절차를 따르면 누구나 자신의 chat_id를 쉽게 확인할 수 있습니다.

  1. 봇에게 말 걸기: 텔레그램에서 방금 만든 내 봇을 검색해 대화방을 열고, 아무 텍스트나 한 번 보냅니다. (예: "안녕")
  2. getUpdates 호출: 발급받은 토큰으로 getUpdates 메서드를 호출합니다.
  3. chat.id 확인: 응답 결과(JSON) 안의 message.chat.id 값이 바로 내 chat_id입니다. 이 값을 메모해 두면 잠시 뒤 9단계(완성)에서 실제로 메시지를 보낼 때 계속 사용하게 됩니다.

5. 네이버 오픈 API 가입·등록

네이버 오픈 API — 외부 프로그램이 네이버 서비스 데이터를 요청하는 창구

네이버 오픈 API는 네이버가 보유한 검색·번역·지도 등의 기능을 외부 프로그램에서 호출할 수 있도록 열어둔 서비스입니다. 이번 봇에서 쓸 뉴스검색 API는 이 중 하나로, 특정 검색어가 포함된 뉴스 기사 목록을 JSON 형태로 돌려줍니다. 이 API를 쓰려면 먼저 네이버 개발자센터에서 "애플리케이션"을 하나 등록하고, 그 애플리케이션 전용의 인증 열쇠(Client ID·Client Secret)를 발급받아야 합니다. 이 열쇠를 발급받아 안전하게 보관한 뒤, 바로 이어지는 6~9단계에서 실제 요청·응답 구조와 검색·필터링·전송 로직까지 오늘 안에 모두 완성합니다.

애플리케이션 등록 — Client ID·Client Secret 발급 절차

아래 절차는 developers.naver.com에서 실제로 진행하는 등록 과정입니다. 한 번만 등록해두면 이후 모든 요청에서 같은 열쇠를 계속 사용합니다.

  1. 로그인: developers.naver.com에 접속해 본인의 네이버 계정으로 로그인합니다.
  2. Application 메뉴 진입: 상단 메뉴에서 Application → 애플리케이션 등록을 선택합니다.
  3. 애플리케이션 이름 입력: 40자 이내로 원하는 이름을 입력합니다(예: "뉴스 언급 알림 봇"). 이 이름은 외부에 공개되지 않으며, 관리 목적으로만 쓰입니다.
  4. 이용 API 선택: 제공 API 목록에서 검색(Search)을 체크합니다.
  5. 비로그인 오픈 API 서비스 환경 등록: 환경으로 WEB을 선택하고, 서비스 URL을 입력합니다. 실습 단계에서는 실제 서비스 도메인이 없으므로 http://localhost 계열의 주소가 흔히 허용됩니다.
  6. 등록 완료 및 확인: 등록을 마치면 애플리케이션 상세 화면에서 Client ID와 Client Secret 값을 확인할 수 있습니다.
등록 항목입력 내용비고
애플리케이션 이름 40자 이내 자유 입력 관리용 이름, 외부 비공개
이용 API 검색(Search) 뉴스검색을 포함한 통합 검색 API 카테고리
서비스 환경 WEB 비로그인 오픈 API 방식
서비스 URL 실제 서비스 도메인 또는 http://localhost 로컬 실습 단계에서는 로컬 주소도 흔히 허용됨
Client ID와 Client Secret은 봇 토큰과 마찬가지로 비밀번호와 같은 개인 인증 정보입니다. 코드 파일에 직접 적어두지 말고, .env 파일처럼 GitHub에 올라가지 않는 별도 파일에 저장한 뒤 불러와서 사용하세요.

💡 핵심 한 줄: 봇 토큰과 네이버 Client ID·Secret, 이 두 열쇠를 발급받았다면 준비는 끝입니다 — 이제 이 열쇠들로 실제 뉴스를 검색하고, 내 이름이 맞는지 걸러내고, 텔레그램으로 보내는 로직까지 오늘 안에 끝까지 완성합니다.

6. 실전 뉴스 수집 — 검색 쿼리로 API 호출하기 (이름·관심 키워드·시간 범위)

뉴스검색 API 호출 — 검색어 하나로 최신 기사 목록 받아오기

발급받은 Client ID·Secret을 HTTP 요청 헤더에 담아 아래 주소로 요청을 보내면, 검색어(query)가 포함된 최신 뉴스 기사 목록을 JSON으로 돌려받습니다.

GET https://openapi.naver.com/v1/search/news.json?query=<검색어>&display=20&sort=date
Header: X-Naver-Client-Id, X-Naver-Client-Secret
파라미터설명
query검색할 키워드 (URL 인코딩 필요)
display한 번에 받아올 기사 개수 (최대 100)
sortdate로 지정하면 최신순 정렬

검색어 하나만으로는 원하는 기사를 놓칠 수 있으므로, 실제로는 이름과 소속·지역·직함을 조합한 여러 검색어를 함께 사용합니다. 예를 들어 "홍길동"이라는 이름만 검색하면 동명이인 기사까지 섞이지만, "홍길동 세종시" "홍길동 시의회"처럼 조합하면 더 정확한 후보 기사를 모을 수 있습니다.

queries = ["홍길동 더불어민주당", "홍길동 세종시의회", "홍길동 세종특별자치시의원"]

응답 JSON의 items 배열 안에 기사 하나하나가 들어 있으며, 자주 쓰는 필드는 다음과 같습니다.

필드설명
title기사 제목 (HTML 태그·이스케이프 포함될 수 있음)
description기사 요약
originallink / link원문 링크 / 네이버 뉴스 링크
pubDate기사 발행 시각 (RFC 822 형식 문자열)
title·description에는 검색어가 강조된 <b> 태그와 &quot; 같은 HTML 엔티티가 섞여 옵니다. 화면에 그대로 보내기 전에 태그를 제거하고 엔티티를 원래 문자로 되돌리는 처리가 필요합니다.

관심 키워드 — 내 이름이 아니어도 관심 주제로 뉴스 받기

이 봇은 "내 이름 뉴스"뿐 아니라, 평소 관심 있는 주제(예: "재정경제", "지역 개발", "AI 정책")도 함께 알림받을 수 있습니다. 방법은 간단합니다 — 이름 조합 쿼리와는 별도로, 관심 키워드 자체를 검색어로 쓰는 쿼리를 추가로 더 호출하면 됩니다. 이름 검색과 달리 관심 키워드 검색은 "동명이인을 걸러낼 이름"이 없으므로, 7단계의 필터와는 조금 다른 더 단순한 조건(키워드 포함 여부)만으로 판단합니다.

NAME_QUERIES = ["홍길동 더불어민주당", "홍길동 세종시의회"]      # 나에 대한 뉴스
INTEREST_KEYWORDS = ["재정경제", "지역 개발", "AI 정책"]         # 내가 관심있는 주제 뉴스

두 종류의 쿼리를 같은 방식으로 API에 호출하되, 이름 쿼리 결과에는 7단계의 is_target_person 필터를, 관심 키워드 결과에는 단순히 그 키워드가 포함돼 있는지만 확인하는 필터를 각각 적용합니다.

시간 범위 설정 — 몇 시간 이내 기사까지 가져올지 정하기

매번 검색할 때마다 "최근 몇 시간 이내에 나온 기사만" 가져오도록 기준을 정해두면, 오래된 기사가 계속 반복해서 잡히는 것을 줄일 수 있습니다. 이 기준은 pubDate(기사 발행 시각)를 파싱해서 지금 시각과 비교하는 방식으로 직접 구현합니다.

from datetime import datetime, timezone, timedelta
from email.utils import parsedate_to_datetime

RECENT_HOURS = 3  # 최근 3시간 이내 기사만 허용 (숫자를 줄일수록 더 "실시간"에 가까워집니다)

cutoff = datetime.now(timezone.utc) - timedelta(hours=RECENT_HOURS)
pub_dt = parsedate_to_datetime(item["pubDate"])
if pub_dt < cutoff:
    continue  # 너무 오래된 기사는 건너뜀
RECENT_HOURS 값을 얼마로 두느냐가 곧 "내가 뉴스를 얼마나 실시간에 가깝게 받고 싶은지"를 정하는 값입니다. 값을 1~2로 줄이면 거의 실시간에 가깝게 받고, 24로 두면 하루 단위로 몰아서 받습니다. 다만 사람이 스크립트를 직접 실행해야만 그 시점의 뉴스를 확인할 수 있다는 점은 동일합니다 — 실행 자체를 사람 없이 주기적으로 반복시키는 자동화는 다음 Day에서 다룹니다.

7. 필터링 — 동명이인 걸러내기 & 관심 키워드 판단

필수 키워드 · 제외 키워드 — 이름만으로는 부족한 이유

흔한 이름일수록 검색 결과에 배우·가수·다른 지역 인물 등 전혀 다른 사람의 기사가 섞여 들어옵니다. 이를 걸러내기 위해 실무에서 흔히 쓰는 방식은 "이름 + 필수 키워드 중 하나 이상 + 제외 키워드 없음"이라는 3단 조건입니다.

  1. 이름이 포함되는가: 제목이나 요약에 검색 대상 이름이 없으면 애초에 후보에서 제외합니다.
  2. 필수 키워드 중 하나 이상 포함되는가: 소속·지역·직함처럼 "진짜 나"를 특정하는 단어(예: 세종시, 세종시의회, 소속 정당명) 중 하나라도 있어야 통과시킵니다.
  3. 제외 키워드가 없는가: 배우·가수·교수 등 동명이인이 흔히 걸리는 직업군이나, 나와 무관한 지역명이 함께 나오면 제외합니다.
TARGET_NAME = "홍길동"
REQUIRED_KEYWORDS = ["세종", "세종시", "세종시의회", "더불어민주당"]
EXCLUDE_CONTEXTS = ["배우", "가수", "교수", "대전", "부산"]

def is_target_person(text: str) -> bool:
    if TARGET_NAME not in text:
        return False
    if not any(kw in text for kw in REQUIRED_KEYWORDS):
        return False
    if any(kw in text for kw in EXCLUDE_CONTEXTS):
        return False
    return True
키워드 목록은 한 번에 완벽하게 정하기 어렵습니다. 처음에는 넓게 잡아 시험 삼아 돌려보고, 실제로 걸러지는(또는 놓치는) 기사를 보면서 Claude Code에게 "이 기사는 왜 통과했는지/걸러졌는지 설명하고 키워드를 조정해줘"라고 요청하며 다듬어 가는 방식이 효율적입니다.

관심 키워드 필터 — 이름 없이 주제만으로 판단하는 더 단순한 조건

6단계에서 추가한 관심 키워드 기사는 애초에 "내가 관심있는 주제인지"만 확인하면 되므로, 동명이인 판단(is_target_person)만큼 조건이 복잡할 필요가 없습니다. 키워드 자체가 이미 검색어였으므로, 제목·요약에 그 키워드가 실제로 들어있는지 한 번 더 확인하는 정도로 충분합니다.

def is_interest_match(text: str, keyword: str) -> bool:
    return keyword in text

# 사용 예: 관심 키워드 쿼리로 받아온 기사마다
if is_interest_match(title + " " + desc, "재정경제"):
    ...  # 통과

8. 중복 발송 방지 — 이미 보낸 기사를 기억하기

seen_urls — 한 번 보낸 기사의 주소를 저장해두는 목록

봇을 여러 번 실행하다 보면 같은 기사가 검색 결과에 계속 다시 나타납니다. 이미 전송한 기사의 URL을 파일(예: seen_urls.txt)에 한 줄씩 저장해두고, 다음 실행에서는 이 목록에 있는 URL을 건너뛰면 같은 기사를 두 번 보내지 않습니다.

def load_seen(path="seen_urls.txt") -> set[str]:
    if not os.path.exists(path):
        return set()
    with open(path, encoding="utf-8") as f:
        return {line.strip() for line in f if line.strip()}

def save_seen(seen: set[str], path="seen_urls.txt"):
    with open(path, "w", encoding="utf-8") as f:
        f.write("\n".join(sorted(seen)))

같은 기사라도 utm_source, fbclid 같은 추적용 파라미터가 붙어 매번 URL이 조금씩 다르게 올 수 있습니다. 비교 전에 이런 추적 파라미터만 제거해 "정규화된 주소"로 맞춰두면 실제로는 같은 기사인데 다른 기사로 오인하는 일을 막을 수 있습니다.

seen_urls.txt 파일은 실행할 때마다 계속 남아 있어야 중복 방지가 의미가 있습니다. 로컬 PC에서만 실행한다면 파일이 그대로 남지만, 10단계처럼 GitHub Actions 자동화 환경에서 돌린다면 실행이 끝날 때마다 이 파일을 저장소에 다시 커밋해야 다음 실행에서도 유지됩니다.

9. 완성 — 새 기사를 텔레그램으로 실행해서 전송하기

지금까지 만든 조각들(이름·관심 키워드 검색 → 시간 범위·필터링 → 중복 제거)을 하나로 합치면, 오늘의 목표였던 "내 이름 + 관심 키워드 뉴스 알림 봇"의 핵심 로직이 완성됩니다. 전체 흐름은 다음과 같습니다.

1. 이름 조합 쿼리 + 관심 키워드 쿼리로 네이버 뉴스 API 호출 → 후보 기사 목록 수집
2. RECENT_HOURS 기준으로 너무 오래된 기사 제외
3. 이름 기사는 is_target_person(), 관심 키워드 기사는 is_interest_match()로 필터링
4. seen_urls와 대조해 이미 보낸 기사 제외
5. 남은 새 기사를 sendMessage로 텔레그램에 전송
6. 전송한 기사의 URL을 seen_urls에 추가하고 저장

Claude Code에게 "지금까지 만든 이름 검색·관심 키워드 검색·시간 필터·동명이인 필터·중복 제거 코드를 하나로 합쳐서, 새 기사가 있으면 텔레그램으로 보내고 seen_urls.txt를 갱신하는 스크립트로 만들어줘"라고 요청하면, 이 여섯 단계를 함수 하나로 엮은 실행 파일을 받을 수 있습니다. 이 스크립트를 로컬에서 직접 실행해보면, 오늘 만든 봇이 실제로 내 이름이 들어간 최신 기사와 관심 키워드 기사를 함께 찾아 텔레그램으로 보내주는 모습을 확인할 수 있습니다.

💡 오늘의 결과물: 지금까지의 6단계를 모두 마치면 "내 이름 + 관심 키워드 뉴스 알림 봇"이 그 자체로 완성되어, 실행할 때마다 실제로 동작합니다. RECENT_HOURS 값으로 얼마나 최신 기사만 받을지도 이미 내가 정할 수 있습니다. 다음 10단계에서는 이 스크립트를 사람이 직접 실행하지 않아도 저절로 주기적으로 실행되도록 GitHub에 올려 자동화합니다.

10. GitHub에 올려 자동으로 실행되게 만들기

지금까지는 사람이 스크립트를 직접 실행해야만 봇이 동작했습니다. GitHub Actions를 쓰면 이 스크립트를 정해진 시간마다 GitHub의 서버가 대신 실행해주므로, 내 컴퓨터를 켜두지 않아도 하루 2번씩(예: 아침 8시, 저녁 6시) 자동으로 뉴스를 확인해 보내줍니다.

저장소에 코드 올리기 — 비밀 값은 코드에서 완전히 분리하기

먼저 스크립트에서 BOT_TOKEN, NAVER_CLIENT_ID, NAVER_CLIENT_SECRET, CHAT_ID처럼 직접 적어뒀던 값들을 모두 환경 변수로 읽도록 바꿉니다. 이렇게 해야 코드 자체는 공개 저장소에 올려도 비밀 값이 함께 노출되지 않습니다.

import os

BOT_TOKEN = os.environ["BOT_TOKEN"]
NAVER_CLIENT_ID = os.environ["NAVER_CLIENT_ID"]
NAVER_CLIENT_SECRET = os.environ["NAVER_CLIENT_SECRET"]
CHAT_ID = os.environ["CHAT_ID"]
  1. GitHub에서 새 저장소 만들기: github.com 우측 상단 + → New repository에서 이름을 정하고 Private(비공개)로 생성합니다.
  2. 코드 push: Claude Code에게 "이 폴더를 방금 만든 GitHub 저장소에 올려줘"라고 요청하면 git init → git add → git commit → git remote add → git push 과정을 대신 실행해 줍니다.
.env 파일이나 토큰이 적힌 파일은 반드시 .gitignore에 추가해 저장소에 올라가지 않도록 합니다. 실수로 이미 올렸다면 그 토큰은 유출된 것으로 간주하고 BotFather의 /revoke로 즉시 재발급받아야 합니다.

Secrets 등록 — GitHub이 대신 보관해주는 비밀 값 저장소

저장소의 Settings → Secrets and variables → Actions → New repository secret에서 아래 값을 하나씩 등록합니다. 여기 등록된 값은 워크플로 실행 중에만 안전하게 불러와지고, 코드나 로그 어디에도 평문으로 노출되지 않습니다.

Secret 이름값
BOT_TOKEN3단계에서 발급받은 텔레그램 봇 토큰
NAVER_CLIENT_ID5단계에서 발급받은 네이버 Client ID
NAVER_CLIENT_SECRET5단계에서 발급받은 네이버 Client Secret
CHAT_ID4단계에서 확인한 내 chat_id

GitHub Actions 워크플로 — 정해진 시간마다 스크립트를 대신 실행해주는 설정 파일

저장소에 .github/workflows/news_check.yml 파일을 만들면, GitHub이 그 안의 cron 일정에 맞춰 스크립트를 자동으로 실행합니다. cron 시각은 UTC 기준이므로, 한국 시간(KST)에서 9시간을 빼서 적어야 합니다 (예: KST 오전 8시 → UTC 전날 23:00).

name: 내 이름 뉴스 알림 봇

on:
  schedule:
    - cron: '0 23 * * *'   # 매일 오전 8시 KST (UTC 23:00)
    - cron: '0 9 * * *'    # 매일 오후 6시 KST (UTC 09:00)
  workflow_dispatch:        # 수동 실행 버튼도 함께 활성화

concurrency:
  group: news-check
  cancel-in-progress: false

jobs:
  check:
    runs-on: ubuntu-latest
    permissions:
      contents: write

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-python@v5
        with:
          python-version: '3.12'

      - run: pip install requests

      - name: 뉴스 체크 및 전송
        env:
          BOT_TOKEN: ${{ secrets.BOT_TOKEN }}
          NAVER_CLIENT_ID: ${{ secrets.NAVER_CLIENT_ID }}
          NAVER_CLIENT_SECRET: ${{ secrets.NAVER_CLIENT_SECRET }}
          CHAT_ID: ${{ secrets.CHAT_ID }}
        run: python check_and_send.py

      - name: seen_urls.txt 변경사항 커밋
        run: |
          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git add seen_urls.txt
          if git diff --cached --quiet; then
            echo "변경 없음 - 커밋 스킵"
          else
            git commit -m "Update seen_urls"
            git pull --rebase -X theirs origin main
            git push origin main
          fi
구성 요소역할
schedule.cron정해진 시각마다 워크플로를 자동 실행 (하루 2회)
workflow_dispatchActions 탭에서 "Run workflow" 버튼으로 즉시 수동 실행 가능
concurrency이전 실행이 끝나기 전에 다음 실행이 겹쳐 중복 발송되는 것을 방지
permissions.contents: write워크플로가 seen_urls.txt 변경사항을 저장소에 다시 커밋할 수 있게 허용
마지막 "커밋" 스텝이번 실행에서 새로 채운 seen_urls.txt를 저장소에 반영해, 다음 자동 실행 때도 중복 방지가 이어지게 함
자동 실행 환경(GitHub Actions)에는 seen_urls.txt가 매번 새로 시작되는 것이 아니라 저장소에 계속 남아 있어야 중복 방지가 유지됩니다. 그래서 마지막 스텝에서 실행 결과를 다시 저장소에 커밋·푸시합니다. 이 파일을 사람이 직접 GitHub 화면에서 수정하면 다음 자동 실행과 충돌할 수 있으니 건드리지 않는 것이 좋습니다.

동작 확인 — Actions 탭에서 실행 로그 보기

  1. 수동 실행으로 먼저 테스트: 저장소의 Actions 탭 → 방금 만든 워크플로 선택 → Run workflow 버튼으로 cron 시각을 기다리지 않고 바로 실행해볼 수 있습니다.
  2. 로그 확인: 실행이 끝나면 각 스텝의 로그에서 몇 건을 검색했고 몇 건을 새로 보냈는지 확인할 수 있습니다.
  3. 실제 텔레그램 확인: 로그에 "전송 완료"가 찍혔다면, 내 텔레그램 대화방에도 실제로 메시지가 도착해 있어야 합니다.

💡 오늘 최종 완성: 이 10단계를 모두 마치면, 내 컴퓨터를 꺼두어도 GitHub의 서버가 하루 2번 대신 스크립트를 실행해 내 이름과 관심 키워드가 담긴 최신 뉴스를 텔레그램으로 보내주는 "완전 자동 뉴스 알림 봇"이 완성됩니다.
