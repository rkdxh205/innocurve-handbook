Day 2 · AI 애플리케이션 개념과 PRD 작성 — 실습

이론에서 익힌 RAG·PRD 개념을 바탕으로, 내가 만들 문서 Q&A 앱의 설계도(PRD)를 직접 완성하는 실습입니다. STEP 1부터 순서대로 진행하십시오.
개념이 헷갈리면 각 단계의 "관련 이론" 링크로 이론 문서를 참고하세요.

진행 순서STEP 1 개념 체험 → STEP 2 뼈대 → STEP 3 PRD 작성 → STEP 4 AI 검토·구체화
사용 도구Day 1에서 설치·구독한 Claude Desktop(Pro)
오늘의 결과물내 앱의 PRD.md 한 편 (다음 개발의 지도)
● 개념 이해 Understand 무엇을 만들지 감을 잡습니다
STEP 1 개념 RAG 챗봇을 직접 체험하고, 만들 앱 정하기
RAG는 글로만 보면 감이 잘 안 옵니다. Claude Desktop에 문서를 직접 올려 "문서 근거로 답하기"를 체험한 뒤,
그 경험을 바탕으로 내가 만들 문서 Q&A 앱의 주제를 정합니다.
관련 이론
AI 애플리케이션과 RAG 챗봇
단계별 실행
  1. 문서 하나 준비하기

    내가 잘 아는 내용의 문서를 하나 고릅니다. 사내 규정, 제품 매뉴얼, 회의록, 안내문 등 PDF나 텍스트 파일이면 됩니다.(개인정보·기밀이 없는 문서를 사용하세요. 없으면 공개된 안내문 PDF를 내려받아 써도 됩니다.)

  2. Claude Desktop에 문서 올려 질문하기

    Claude Desktop에서 새 대화를 열고, 준비한 문서를 첨부(＋ 또는 클립 아이콘)한 뒤 아래처럼 질문합니다.

    프롬프트 
    첨부한 문서 내용만 근거로 답해줘.
1. 이 문서의 핵심 내용을 3줄로 요약해줘.
2. (문서에 있을 법한 구체적 질문 하나)를 알려주고,
   어느 부분을 근거로 했는지도 함께 알려줘.
  3. 어떻게 답하는지 관찰하기

    답을 읽으며 아래를 확인합니다. 이것이 바로 RAG(문서 근거로 답하기)의 축소 체험입니다.

    1. 답이 문서 안의 내용을 근거로 하고 있는가?
    2. 문서에 없는 것을 물으면 어떻게 답하는가? (모른다고 하는지 확인)
    방금 한 것이 RAG의 핵심입니다 — Claude가 먼저 첨부 문서를 읽고(검색), 그 내용을 근거로 답(생성)했습니다. 우리가 만들 앱은 이 과정을 여러 문서에 대해 자동으로 해 주는 프로그램입니다.
  4. 내가 만들 앱 주제 정하기

    평소 "이 문서에서 뭐 좀 빨리 찾았으면" 싶던 업무를 하나 떠올립니다. 아래 예시를 참고해, 내 앱을 한 문장으로 적어 봅니다.

    주제 예시
    • 사내 규정 Q&A — 직원이 복무·경비 규정을 묻으면 관련 조항으로 답
    • 제품 매뉴얼 Q&A — 고객이 사용법을 물으면 매뉴얼에서 찾아 답
    • 회의록 Q&A — "지난달 예산 회의 결정사항"을 회의록에서 찾아 답
    내 앱 한 문장: "____(누구)____ 가 ____(무슨 문서)____ 에 대해 물으면 근거와 함께 답해 주는 Q&A 챗봇"
주제는 내가 실제로 겪는 불편에서 고를수록 뒤의 PRD가 잘 나옵니다. 남의 문제가 아니라 내 문제를 고르세요.
● PRD 작성 Design 만들 앱의 설계도를 글로 완성합니다
STEP 2 PRD PRD 뼈대 잡기 — 문제·핵심 기능·흐름
정식 문서를 쓰기 전, PRD의 핵심 세 기둥 — 문제 · 핵심 기능 · 사용자 흐름 — 을 짧은 메모로 먼저 잡습니다. 여기서 뼈대만 잘 잡아 두면 STEP 3의 작성이 빈칸 채우기처럼 쉬워집니다.
관련 이론
PRD란 무엇이고 왜 필요한가 좋은 PRD의 구성
단계별 실행
  1. 문제를 한 문장으로 적기

    STEP 1에서 정한 주제를 바탕으로 "누가, 어떤 불편을 겪는가"를 한 문장으로 적습니다.

    예시
    담당자가 수십 쪽짜리 사내 규정에서 원하는 조항을 찾느라 매번 10분 넘게 헤맨다.
  2. 핵심 기능 3~5개 적기

    꼭 필요한 기능(must)부터 적고, 있으면 좋은 기능(nice)은 따로 표시합니다.

    예시
    • (must) 문서 업로드
    • (must) 질문 입력 & 답변
    • (must) 답의 근거가 된 문서 위치 표시
    • (must) 문서에 없으면 "모른다"고 정직하게 답
    • (nice) 여러 문서 동시 검색
  3. 사용자 흐름을 화살표로 그리기

    사용자가 앱을 쓰는 순서를 화살표로 이어 봅니다. 그리다 보면 빠진 기능(예: 못 찾았을 때 처리)이 드러납니다.

    예시
    문서 업로드 → 질문 입력 → 문서에서 검색 → 답변 + 근거 표시
    └ (못 찾으면) "모른다"고 답
이 세 메모는 종이·메모장 어디에 적어도 됩니다. 혼자 막히면 Claude Desktop에 "문서 Q&A 챗봇의 핵심 기능 후보를 나열해 줘"라고 물어 아이디어를 얻은 뒤 내가 고르면 됩니다.
STEP 3 PRD 내 애플리케이션 PRD 직접 작성하기
STEP 2의 메모를 정식 문서로 옮깁니다. Claude Desktop으로 PRD.md 파일을 만들고, 다섯 항목을 내 앱에 맞게 채웁니다.
관련 이론
좋은 PRD의 구성
단계별 실행
  1. 프로젝트 폴더 만들고 작업 폴더로 지정

    Day 1과 같은 방식으로, 이 앱을 위한 새 폴더를 만들고 Claude Desktop의 작업 폴더로 지정합니다.

    1. 홈 디렉토리에 doc-qa-bot 폴더를 새로 만듭니다.(한글·공백 없는 영문 이름)
    2. Claude Desktop Settings에서 이 doc-qa-bot 폴더를 작업 폴더로 지정합니다.(방법이 기억나지 않으면 Day 1 STEP 2를 참고하세요.)
  2. PRD 템플릿 만들기

    Claude Desktop 대화창에 아래 프롬프트를 붙여넣어, 빈 PRD.md 템플릿을 만들게 합니다. 파일 생성을 물으면 "허용(Allow)" 합니다.

    프롬프트 
    doc-qa-bot 폴더에 PRD.md 파일을 만들어줘.
아래 다섯 항목의 빈 템플릿으로 만들어줘.
1) 문제 정의
2) 목표와 성공 기준
3) 핵심 기능 (must / nice로 구분)
4) 사용자 흐름
5) 범위와 비범위
  3. 내 내용으로 채우기

    STEP 2에서 잡은 메모를 불러 주며 각 항목을 채우게 합니다. 아래처럼 내 내용을 넣어 요청합니다.

    프롬프트 
    PRD.md를 아래 내용으로 채워줘.
- 문제: (STEP 2에서 적은 문제 한 문장)
- 핵심 기능: (STEP 2에서 고른 기능들)
- 사용자 흐름: (STEP 2에서 그린 흐름)
목표·성공 기준과 범위·비범위는 위 내용에 맞게
적절히 제안해서 채워줘.
  4. 결과 확인

    doc-qa-bot 폴더의 PRD.md를 열어 다섯 항목이 내 앱에 맞게 채워졌는지 확인합니다. 아래는 완성된 PRD의 예시입니다.

    PRD.md 작성 예시 — 사내 규정 Q&A 챗봇
    1) 문제 정의
    담당자가 수십 쪽 규정에서 원하는 조항을 찾느라 매번 10분 넘게 헤맨다.

    2) 목표와 성공 기준
    문서를 올린 뒤 질문하면, 관련 조항을 근거와 함께 30초 안에 답한다.

    3) 핵심 기능
    (must) 문서 업로드 · 질문/답변 · 근거 위치 표시 · 모르면 정직하게 답
    (nice) 여러 문서 동시 검색 · 자주 묻는 질문 저장

    4) 사용자 흐름
    업로드 → 질문 → 검색 → 답변+근거 → (못 찾으면) 모른다고 답

    5) 범위와 비범위
    (범위) 한국어 텍스트 문서 / (비범위) 이미지·표 인식, 로그인, 다국어
처음부터 완벽하지 않아도 됩니다. 지금은 다섯 항목이 다 채워진 초안이면 충분합니다. 다듬기는 STEP 4에서 AI와 함께 합니다.
STEP 4 PRD AI를 활용해 PRD 검토·구체화하기
초안을 Claude Desktop에 검토받아 빠진 부분과 모호한 표현을 메우고, 핵심 기능을 개발하기 좋은 단위로 쪼개 최종 PRD로 완성합니다.
관련 이론
AI를 활용한 PRD 검토·구체화
단계별 실행
  1. PRD 검토 요청

    같은 대화(또는 doc-qa-bot 폴더가 지정된 대화)에서 아래 프롬프트로 검토를 요청합니다.

    프롬프트 
    PRD.md를 읽고 검토해줘.
1) 빠졌거나 더 정해야 할 부분
2) 모호하게 적힌 문장
3) 사용자가 겪을 수 있는 예외 상황 5가지
를 짚어줘. 아직 파일은 고치지 말고 의견만 줘.
  2. 피드백 중 반영할 것 고르기

    Claude의 의견을 모두 반영할 필요는 없습니다. 이번 앱에 꼭 필요한 것만 골라, 반영을 지시합니다.

    프롬프트 
    방금 준 의견 중에서
(예: 1번, 3번의 첫째·둘째 항목)만 반영해서
PRD.md를 업데이트해줘.
  3. 핵심 기능을 개발 단위로 쪼개기

    다음 개발 단계에서 바로 쓸 수 있도록, 핵심 기능을 작은 작업 단위로 나눠 PRD에 덧붙입니다.

    프롬프트 
    핵심 기능을 만들기 좋은 작은 작업 단위로 나눠서,
PRD.md 맨 아래에 "개발 단위" 섹션으로 추가해줘.
각 단위는 한 줄로, 만들 순서대로 정렬해줘.
  4. 최종본 확인

    PRD.md를 다시 열어, 다섯 항목이 구체화되고 맨 아래에 "개발 단위"가 정리되었는지 확인합니다. 이 문서가 이후 개발의 지도가 됩니다.

검토를 두세 번 반복하면 문서가 눈에 띄게 단단해집니다. 단, AI가 제안한 기능을 전부 넣지 말고 이번에 꼭 필요한 것만 남기세요 — 범위를 좁히는 것도 실력입니다.
오늘의 마무리: "무엇을·왜 만들지"를 PRD 한 편으로 정리했습니다. 이 PRD.md가 다음 단계에서 실제 앱을 만들 때의 기준이 됩니다.
