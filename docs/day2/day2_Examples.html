Day 2 · AI 애플리케이션 개념과 PRD 작성 — 실습

이론에서 익힌 RAG·PRD 개념을 바탕으로, 내가 만들 문서 Q&A 앱의 설계도(PRD)를 직접 완성하는 실습입니다. STEP 1부터 순서대로 진행하십시오.
개념이 헷갈리면 각 단계의 "관련 이론" 링크로 이론 문서를 참고하세요.

STEP 1 개념 RAG 챗봇을 직접 체험하고, 내 손으로 직접 만들기
RAG는 글로만 보면 감이 잘 안 옵니다. Claude Desktop에 문서를 직접 올려 "문서 근거로 답하기"를 체험하고, 내가 만들 앱의 주제를 정합니다.
관련 이론
AI 애플리케이션과 RAG 챗봇
단계별 실행
  1. 문서 하나 준비하기

    내가 잘 아는 내용의 문서를 하나 고릅니다. 사내 규정, 제품 매뉴얼, 회의록, 안내문 등 PDF나 텍스트 파일이면 됩니다.(개인정보·기밀이 없는 문서를 사용하세요. 없으면 공개된 안내문 PDF를 내려받아 써도 됩니다.)

  2. Claude Desktop에 문서 올려 질문하기

    Claude Desktop에서 새 대화를 열고, 준비한 문서를 첨부(＋ 또는 클립 아이콘)한 뒤 아래처럼 질문합니다.

    프롬프트 
    첨부한 문서 내용만 근거로 답해줘.
1. 이 문서의 핵심 내용을 3줄로 요약해줘.
2. (문서에 있을 법한 구체적 질문 하나)를 알려주고,
   어느 부분을 근거로 했는지도 함께 알려줘.
  3. 어떻게 답하는지 관찰하기

    답을 읽으며 아래를 확인합니다. 이것이 바로 RAG(문서 근거로 답하기)의 축소 체험입니다.

    1. 답이 문서 안의 내용을 근거로 하고 있는가?
    2. 문서에 없는 것을 물으면 어떻게 답하는가? (모른다고 하는지 확인)
    방금 한 것이 RAG의 핵심입니다 — Claude가 먼저 첨부 문서를 읽고(검색), 그 내용을 근거로 답(생성)했습니다. 우리가 만들 앱은 이 과정을 여러 문서에 대해 자동으로 해 주는 프로그램입니다.
이어서 실행 — 헬로페이지에 문서 Q&A(RAG) 이어 붙이기
1️⃣ Day 1 헬로페이지 그대로 열기 — 새 폴더 없이 재사용
  1. hello-page 폴더에서 작업 시작

    Day 1에서 만든 hello-page 폴더를 그대로 씁니다. 이 폴더엔 이미 index.html(자기소개·시계)·.env(OpenAI 키)·재정경제부.svg 로고가 다 들어 있어, 새로 만들 게 없습니다.

    1. Claude Desktop Settings에서 작업 폴더가 hello-page로 되어 있는지 확인합니다.(바뀌어 있으면 Day 1 STEP 2처럼 다시 지정하세요.)
    2. .env에 OPENAI_API_KEY가 들어 있는지 확인합니다.(없으면 Day 1 STEP 4의 "방법 A"로 넣으세요 — 키가 채팅에 남지 않는 방법입니다.)
2️⃣ 챗봇 연결 파일 만들기 — api/chat.js, 키를 숨기고 문서를 근거로 답
  1. api/chat.js 요청

    화면(HTML)에서 OpenAI를 직접 부르면 API 키가 그대로 노출됩니다. 그래서 Day 1과 똑같이, 키를 대신 다뤄 주는 작은 연결 파일 api/chat.js 하나만 둡니다. 별도 설치 없이 Node 기본 기능(fetch)만 씁니다.

    프롬프트 
    hello-page 폴더에 api/chat.js 파일을 만들어줘.
- module.exports = async (req, res) => {...} 형식으로 작성
- 외부 패키지 설치 없이 Node 기본 fetch만 사용해줘.
- POST 요청 body에서 messages 배열과 docText(문서 전체 글자)를 받아줘.
- 시스템 메시지와 "다음 문서 내용만 근거로 답해줘:\n" + docText 를
  맨 앞에 넣고, 이어서 messages를 붙여
  https://api.openai.com/v1/chat/completions 에 전달해줘 (model: gpt-4o-mini).
- 시스템 메시지:
  "너는 매우 친절한 선생님이야. 답변은 중학생도 이해하도록 쉽게,
   그러나 내용은 빠뜨리지 말고 모두 답해줘. 첨부 문서에 없는 내용은
   지어내지 말고 '문서에 없습니다'라고 답해줘. 존댓말 한글로."
- Authorization 헤더용 키는 process.env.OPENAI_API_KEY 를 써줘.
  (배포하면 Vercel이 환경 변수 값을 자동으로 채워주니
   코드에서 .env를 직접 읽을 필요는 없어.)
- 키가 없으면 { error: "API 키가 없습니다" } 를 응답
- POST가 아닌 요청은 405로 응답
3️⃣ index.html에 "문서 Q&A" 섹션 이어 붙이기 — PDF는 브라우저에서 바로 읽기
  1. 기존 index.html에 섹션 추가 요청

    자기소개·시계는 그대로 두고, 그 아래에 문서 Q&A 섹션만 이어 붙입니다. PDF 글자 추출까지 브라우저 안에서 처리하므로 설치가 필요 없습니다. (PDF 읽기 도구 pdf.js는 인터넷 주소로 불러옵니다.)

    프롬프트 
    hello-page 폴더의 index.html에 새 섹션을 이어 붙여줘.
기존 자기소개·시계 내용은 절대 지우지 말고 그 아래에 추가해줘.
설치 없이 이 파일과 api/chat.js 만으로 도는 "문서 Q&A" 섹션이야.

- pdf.js를 CDN <script>로 불러와줘
  (cdnjs의 pdf.min.js + pdf.worker.min.js).
- 여러 개의 PDF를 선택해 올리는 영역
  · 올리면 pdf.js로 각 PDF의 글자를 브라우저에서 뽑아
    하나로 합쳐 docText 변수에 보관하고,
    끝나면 "문서 준비 완료 (N개)"만 표시
- 그 아래 채팅 UI
  · 지금까지의 대화가 쌓이는 메시지 목록 + 입력창 + 전송 버튼
  · 전송하면 fetch로 /api/chat 에 POST 해줘.
    body는 { messages: [...지금까지 대화...], docText } (JSON)
  · 답을 기다리는 동안 "생각 중..." 표시
  · 새로고침 전까지 이전 대화를 계속 이어서 기억
- API 키를 다루는 코드는 화면에 절대 넣지 말 것 (호출은 api/chat.js가 대신).
- 위쪽 시계 섹션과 어울리게 파란색 계열 카드로

    문서가 아주 길면 한 번에 보낼 수 있는 양에 한계가 있습니다. 실습은 짧은 문서 1~2개로 시작하세요.

4️⃣ 대화 세션 저장 — Supabase에 저장·불러오기

지금까지는 새로고침하면 대화가 사라집니다. Day 1 STEP 6(로그인)에서 만든 Supabase(계정·키·.env의 SUPABASE_URL·SUPABASE_ANON_KEY)를 그대로 재사용해, 대화와 올린 문서를 세션째 저장·복원합니다.

  1. ① 세션용 테이블 SQL 만들기

    대화와 문서를 담을 테이블 두 개를 만드는 SQL을 Claude에게 맡깁니다.

    프롬프트 
    hello-page 폴더에 sessions-ref.sql 파일을 만들어줘.
빈 Supabase 프로젝트라고 생각하고, 아래 두 테이블을 만드는 SQL을 담아줘.
- chat_sessions: id, title, doc_text(올린 문서 전체 글자), created_at
- chat_messages: id, session_id(chat_sessions 참조 FK), role, content, created_at
- 실습 편의를 위해 anon 키로 조회·저장·삭제가 되도록 두 테이블의 RLS는 꺼줘
  (또는 anon에 select/insert/delete를 허용하는 정책을 넣어줘).
이미 같은 이름의 객체가 있으면 지우고 다시 만드는 형태로 써줘.
  2. ② Supabase에서 실행 (사람이 직접)

    Supabase 대시보드 왼쪽 SQL Editor를 열고, 방금 만든 sessions-ref.sql 내용을 전체 복사해 붙여넣고 Run 합니다. 초록색 성공 메시지가 뜨면 준비 끝입니다.(.env에 SUPABASE_URL·SUPABASE_ANON_KEY가 없으면 Day 1 STEP 6에서 쓴 값을 그 두 줄로 추가하세요.)

  3. ③ 세션 저장·로드·삭제 붙이기 (api/sessions.js + 사이드바)

    대화·문서를 저장하고 목록에서 다시 불러오는 기능을 붙입니다. 설치 없이 fetch로 Supabase에 연결합니다.

    프롬프트 
    hello-page에 api/sessions.js를 만들고, index.html의 "문서 Q&A" 섹션에 세션 사이드바를 붙여줘.
설치 없이 Node 기본 fetch로 Supabase REST API를 직접 호출하는 방식으로 해줘.
접속 정보는 process.env.SUPABASE_URL / SUPABASE_ANON_KEY 를 써줘.

[api/sessions.js — 한 파일에서 아래 동작 처리]
- 세션저장: 지금까지의 대화(messages)와 올린 문서(docText)를 저장해줘.
  · 첫 질문·답을 gpt-4o-mini로 짧게 요약해 세션 제목을 만들고,
    chat_sessions에 INSERT(title, doc_text) → 반환된 session_id로
    chat_messages에 각 메시지(role, content)를 INSERT.
  · 기존 세션은 그대로 두고 새로 쌓이게 해줘.
- 세션목록: chat_sessions를 최신순(created_at desc)으로 가져와줘.
- 세션로드: 선택한 session_id의 doc_text와 chat_messages를 가져와,
  화면의 docText와 대화 목록을 그 내용으로 되살려줘.
- 세션삭제: 선택한 session_id의 chat_messages·chat_sessions를 지워줘.

[index.html — 문서 Q&A 섹션에 왼쪽 사이드바 추가]
- 저장된 세션 목록(선택 가능) + 버튼 [세션저장][세션로드][세션삭제][새 대화]
- 세션을 고르면 그 세션의 문서·대화가 화면에 다시 나오게 해줘.
- API 키·DB 키를 다루는 코드는 화면에 절대 넣지 말 것(호출은 api/가 대신).
- 위쪽 시계·문서 Q&A와 어울리게 파란색 계열로.

    답변이 끝날 때마다 자동 저장되게 하려면 이어서 "답변이 끝날 때마다 방금 대화를 현재 세션에 자동 저장해줘(세션이 없으면 새로 만들고)"라고 요청하면 됩니다.

방금 Claude에 문서를 첨부해 물어본 것을, Day 1 헬로페이지에 이어 붙여 내 웹앱으로 만든 것이 이 문서 Q&A입니다. 여기에 대화를 저장·불러오는 멀티세션(Supabase)까지 갖췄습니다. 세부사항(화면 문구 등)이 마음에 안 들면 이어서 말로 요청해 다듬으면 됩니다.
OPENAI_API_KEY·SUPABASE_URL·SUPABASE_ANON_KEY는 .env에만 두고 index.html·api/*.js에 직접 적지 마세요. GPT 호출은 API 사용료가 조금씩 과금됩니다(문서를 근거로 함께 보내므로 문서가 길수록 비용이 늘어납니다). 이 앱을 인터넷에 공개 배포할 때는 .gitignore에 .env를 추가하고, Vercel 프로젝트 설정의 환경 변수에 세 키를 모두 등록해야 합니다.
마지막으로 — 내가 만들 앱 주제 정하기
  1. 내가 만들 앱 주제 정하기

    평소 "이 문서에서 뭐 좀 빨리 찾았으면" 싶던 업무를 하나 떠올립니다. 아래 예시를 참고해, 내 앱을 한 문장으로 적어 봅니다.

    주제 예시
    • 사내 규정 Q&A — 직원이 복무·경비 규정을 묻으면 관련 조항으로 답
    • 제품 매뉴얼 Q&A — 고객이 사용법을 물으면 매뉴얼에서 찾아 답
    • 회의록 Q&A — "지난달 예산 회의 결정사항"을 회의록에서 찾아 답
    내 앱 한 문장: "____(누구)____ 가 ____(무슨 문서)____ 에 대해 물으면 근거와 함께 답해 주는 Q&A 챗봇"
주제는 내가 실제로 겪는 불편에서 고를수록 뒤의 PRD가 잘 나옵니다. 남의 문제가 아니라 내 문제를 고르세요. 이 주제가 STEP 3부터의 my-app이 됩니다.
● 배포 Deploy 만든 결과물을 인터넷에 공개합니다
STEP 2 실습 🚀 hello-page를 인터넷에 공개(배포)하기
방금 만든 RAG 챗봇까지 더해진 hello-page(자기소개·시계·로그인·문서 Q&A 챗봇)를 누구나 접속할 수 있는 인터넷 주소로 공개합니다. 터미널 없이 GitHub와 Vercel 사이트에서 클릭만으로 진행합니다. 로그인은 Supabase, 챗봇은 OpenAI를 쓰므로 api/ 폴더까지 함께 올리고 — 비밀 키가 담긴 .env는 올리지 말고 Vercel 환경 변수에 키를 등록해 안전하게 공개합니다.
관련 이론
배포(deploy)의 개념 Vercel의 이해
1️⃣ GitHub에 올리기 — .env는 제외
  1. GitHub에 새 저장소(repository) 만들기

    Day 1 STEP 1에서 만든 GitHub 계정으로 로그인한 뒤, 코드를 올려 둘 빈 저장소를 하나 만듭니다.

    1. github.com 우측 상단의 "+" → "New repository"를 선택합니다.
    2. Repository name에 hello-page를 입력하고, 공개 범위는 "Public"으로 둡니다.(다른 항목은 비워 둡니다.)
    3. "Create repository"를 클릭합니다.
  2. 결과물을 GitHub에 업로드하기 (.env 제외)

    터미널 없이 GitHub 사이트에서 파일을 바로 올립니다.

    1. 저장소 화면의 "Add file" → "Upload files"를 클릭합니다.
    2. hello-page 폴더에서 index.html·재정경제부.svg·api/ 폴더·package.json을 올립니다.(.gitignore에 .env가 있으면 자동으로 빠집니다. 목록에 .env가 보이면 반드시 제외하세요.)
    3. 아래 "Commit changes"로 업로드를 확정합니다.
2️⃣ Vercel Import + 환경 변수 등록 — 키는 Vercel에
  1. 저장소 연결 · 키 등록 · 배포

    키를 코드가 아니라 Vercel 설정에 넣습니다. 로컬 .env와 같은 값을 등록하면 됩니다.

    1. vercel.com에 로그인하고 "Add New… → Project"로 hello-page 저장소를 "Import"합니다.(목록에 없으면 "Adjust GitHub App Permissions"로 접근을 허용합니다.)
    2. Deploy 누르기 전에 "Environment Variables"를 펼쳐 아래 세 개를 이름·값으로 추가합니다. 
      SUPABASE_URL        = (Supabase Project URL)
SUPABASE_ANON_KEY   = (Supabase anon public 키)
OPENAI_API_KEY      = (Day 1 STEP 1의 sk- 키)
    3. 이제 "Deploy"를 클릭합니다. 잠시 뒤 공개 주소가 나타납니다.
  2. 공개 주소에서 확인
