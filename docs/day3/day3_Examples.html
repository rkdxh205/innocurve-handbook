Day 3 · Claude Code 심화 — 스킬·플러그인·하네스·PDCA (실습)

이론에서 익힌 스킬·플러그인·하네스·PDCA를 Claude Code에서 직접 설정하고 적용하는 실습입니다. STEP 1부터 순서대로 진행하십시오.
개념이 헷갈리면 각 단계의 "관련 이론" 링크로 이론 문서를 참고하세요.

진행 순서STEP 1 확장 → STEP 2 하네스 → STEP 3·4 계획·설계·검토
사용 도구Claude Desktop의 Code 탭 — Day 1에서 설치(앱 상단 Chat / Cowork / Code 중 "Code")
오늘의 결과물CLAUDE.md(규칙) · PLAN.md · DESIGN.md
작업 폴더는 PRD.md가 있는 Day 2의 my-app을 이어서 씁니다.

STEP 1 스킬·플러그인 스킬·플러그인·서브에이전트 확인하고 설치하기
Code 탭에서 지금 쓸 수 있는 명령·스킬을 살펴보고, 이 과정의 핵심 도구인 bkit 플러그인을 설치합니다. bkit은 Day 3 STEP 2~4·Day 4에서 쓸 PDCA 하네스 엔진입니다 — "매번 설명하던 것을 한 번 정의해 재사용"하는 플러그인의 힘을 이걸로 체감합니다.
관련 이론
스킬과 플러그인
단계별 실행
  1. Code 탭에서 새 세션 시작 (작업 폴더 지정)

    Claude Desktop을 실행하고 상단 Code 탭을 클릭합니다. + New session(또는 Ctrl+N)으로 새 세션을 열고, 프롬프트 영역에서 Environment: Local · Project folder: Day 2의 my-app 선택을 지정한 뒤 프롬프트를 입력합니다.이후 모든 프롬프트·명령은 이 Code 탭 화면에서 진행합니다 — 검은 터미널을 따로 열 필요가 없습니다.

  2. bkit 플러그인 설치

    이 과정의 핵심 도구인 bkit을 설치합니다. 개발자용 명령을 외울 필요 없이, 아래처럼 자연어로 부탁하면 됩니다 — Claude가 마켓플레이스 추가 → 설치를 순서대로 진행합니다. 중간에 권한을 물으면 본인이 승인합니다.

    프롬프트 (자연어로 설치 요청) 
    bkit 플러그인을 설치하고 싶어.
아래 주소의 마켓플레이스를 추가한 뒤
bkit을 설치해줘.

주소: popup-studio-ai/bkit-claude-code

    ▲ popup-studio-ai/bkit-claude-code는 bkit의 설치 주소입니다 — 이 값만 있으면 Claude가 알아서 /plugin marketplace add → /plugin install bkit 을 순서대로 실행합니다.

    직접 하고 싶으면 — 위 두 명령을 프롬프트에 한 줄씩 차례로 입력하거나, + 버튼 → Plugins 브라우저에서 bkit을 검색해 GUI로 설치할 수도 있습니다.
  3. 새 세션을 시작해 bkit 켜기

    설치만으로는 바로 켜지지 않습니다. + New session(Ctrl+N)으로 새 세션을 시작하면 방금 설치한 bkit이 활성화됩니다 — 새 세션에서도 작업 폴더를 Day 2의 my-app으로 그대로 지정합니다.이후 STEP 2~4·Day 4는 이 세션에서 진행합니다.

  4. 설치 확인 — bkit이 데려온 스킬·서브에이전트 보기

    프롬프트에 /를 입력해 bkit이 추가한 새 슬래시 명령이 보이는지 확인한 뒤, 아래 두 프롬프트를 차례로 물어 bkit이 함께 데려온 스킬과 서브에이전트(특정 작업을 위임받는 보조 에이전트)를 확인합니다 — 지금은 "이런 게 있다"는 것만 눈으로 확인하면 됩니다. 이 명령·스킬·에이전트들이 Day 3 STEP 2~4·Day 4의 PDCA를 돌리는 도구가 됩니다.

    프롬프트 ① 스킬 확인 
    지금 이 프로젝트에서 내가 쓸 수 있는 스킬에는
어떤 것들이 있어? 각각 무슨 일을 하는지 한 줄로 설명해줘.
    프롬프트 ② 서브에이전트 확인 
    지금 이 프로젝트에서 쓸 수 있는 서브에이전트에는
어떤 것들이 있어? 각각 무슨 일을 하는지 한 줄로 알려줘.
방금 설치한 bkit에는 "이렇게 일해라" 하는 작업 방식·순서·점검 방법이 미리 담겨 있습니다. 그래서 Day 3의 다음 단계들과 Day 4가 모두 이 위에서 편하게 진행됩니다 — 플러그인 하나 깔았을 뿐인데, 앞으로 쓸 준비물이 한 번에 갖춰진 셈입니다.
스킬은 대부분 상황에 맞을 때 Claude가 알아서 꺼내 씁니다. 억지로 외워 부르기보다, "이런 작업엔 이런 스킬이 있다"는 지도만 익혀 두면 됩니다.
🧭 직접 불러 쓰고 싶다면 — 스킬·에이전트 사용 가이드 (펼치기)

① 먼저 "뭐가 있는지" 본다
위 STEP 1의 두 프롬프트(스킬 확인 · 서브에이전트 확인)를 물으면, Claude가 지금 쓸 수 있는 목록과 각각이 하는 일을 한 줄씩 알려줍니다. 여기서 이름을 확인한 뒤 아래처럼 부르면 됩니다.

② 부르는 방법은 3가지

  • 이름 대며 말하기 — "○○ 스킬로 이거 해줘", "이 작업은 ○○ 에이전트한테 맡겨줘"
  • 슬래시 목록에서 고르기 — 프롬프트에 / 만 입력하면 쓸 수 있는 명령이 뜹니다. 그중 골라 실행합니다.
  • 그냥 하고 싶은 일만 말하기 — 이름을 몰라도 "PLAN.md 만들어줘"처럼 말하면, Claude가 맞는 스킬·에이전트를 알아서 골라 씁니다. 평소엔 이 방식이면 충분합니다.

③ 이 과정에서는 이런 상황에 이렇게

  • 규칙을 정하고 싶다 → "이 프로젝트 규칙서(CLAUDE.md) 초안 만들어줘" — 스킬이 처리
  • 작업 순서를 짜고 싶다 → "PRD 읽고 만들 순서대로 PLAN.md 만들어줘" — 스킬이 처리
  • 설계를 하고 싶다 → "PLAN을 바탕으로 DESIGN.md 만들어줘" — 스킬이 처리
  • 문서끼리 안 맞는 곳을 찾고 싶다 → "PRD·PLAN·DESIGN을 함께 읽고 어긋난 곳을 찾아줘" — 검토 서브에이전트가 처리
  • (Day 4) 코드를 만들고 싶다 → "DESIGN대로 구현해줘" — 스킬·에이전트가 처리
  • (Day 4) 설계대로 됐는지 확인 → "설계와 실제 코드가 다른 곳(Gap)을 찾아줘" — 검증 서브에이전트가 처리

스킬·에이전트의 이름과 개수는 설치된 버전에 따라 다를 수 있습니다. 그러니 헷갈릴 땐 항상 ①로 실제 목록을 먼저 확인하세요.

📖 bkit 스킬 전체 목록 (카탈로그 · 펼치기)
처음엔 다 외울 필요 없습니다. 이 과정(Day 3~4)에서 주로 만나는 건 몇 개뿐입니다 — pdca(계획→검증까지 전체 사이클) · plan-plus(계획 세우기) · starter(정적 웹 만들기) · code-review(검증). 나머지는 "이런 것도 있구나" 하고 필요할 때 찾아 쓰는 사전으로 보세요.

🧭 PDCA / Sprint 핵심 워크플로우

  • bkit — bkit 사용법을 한눈에 보여 주는 안내판 — 무엇부터 시작할지 알려 줍니다
  • pdca — 기능 하나를 '계획→설계→만들기→점검→고치기→보고'까지 처음부터 끝까지 챙겨 주는 진행 도우미
  • pdca-batch — 여러 기능을 서로 안 섞이게, 따로따로 동시에 진행 관리
  • pdca-fast-track — 결과가 충분히 믿을 만하면 중간 확인을 건너뛰고 빠르게 진행하는 모드
  • pdca-watch — 지금 어디까지 진행됐는지 실시간으로 보여 주는 현황판
  • sprint — 여러 기능을 한 묶음(하나의 목표·분량)으로 잡고 한꺼번에 진행 관리
  • plan-plus — "진짜 필요한 것만" 추려 가며 아이디어를 정리해 계획을 세워 줌
  • control — Claude에게 얼마나 알아서 맡길지(자동화 수준)와 안전장치를 조절
  • rollback — 중간중간 저장해 두고, 문제가 생기면 이전 상태로 되돌리기
  • audit — 무슨 결정을 왜 했는지, 지난 작업 기록을 되돌아보기

🏗️ 9단계 개발 파이프라인

  • development-pipeline — 앱 만들기의 처음(구조 잡기)부터 끝(공개)까지 9단계 전체 길잡이
  • phase-1-schema — 앱에서 쓸 용어와 정보 구조를 먼저 정리 (예: '회원'에 어떤 정보가 들어가나)
  • phase-2-convention — 코드를 어떤 규칙으로 통일해 쓸지 정하기
  • phase-3-mockup — 실제로 만들기 전에 화면 밑그림(초안)을 먼저 만들어 보기
  • phase-4-api — 화면 뒤에서 정보를 주고받는 부분(서버 기능)을 설계·구현
  • phase-5-design-system — 버튼·색·글꼴 같은 디자인 요소를 한 벌로 통일해 정리
  • phase-6-ui-integration — 눈에 보이는 화면을 만들고, 뒤의 서버 기능과 연결
  • phase-7-seo-security — 검색에 잘 뜨게(SEO) + 보안을 튼튼하게 보강
  • phase-8-review — 완성된 코드가 규칙대로·앞뒤 맞게 됐는지 점검
  • phase-9-deployment — 실제 사용자에게 공개(배포)

📋 기획 / QA

  • pm-discovery — 기획 도우미 여럿이 함께 "무엇을, 왜 만들지"를 정리하고 기획서(PRD)까지 작성
  • qa-phase — 기능 하나를 여러 수준으로 테스트할 계획을 세우고 실제로 돌려 결과 보고
  • code-review — 만든 코드를 검사해 문제(버그)를 찾고 개선점을 알려 줌
  • zero-script-qa — 따로 코드를 짜지 않고, 앱이 남기는 기록을 보며 문제를 찾는 테스트

📱 앱 유형별 가이드

  • dynamic — 로그인·데이터 저장 같은 기능이 있는 본격 웹앱 만들기

🛠️ bkit 자체 관리 도구

  • bkit-templates — 계획서·설계서·보고서 등 문서 서식(빈 양식) 모음

🤖 bkit 서브에이전트 전체 목록 (카탈로그 · 펼치기)
서브에이전트는 특정 단계를 대신 맡는 "전담 보조원"입니다. 스킬처럼 대부분 Claude가 필요할 때 알아서 부릅니다. 이 과정(Day 3~4)에서 특히 마주치는 건 — starter-guide(초보자 안내) · product-manager(계획 문서) · design-validator(설계 검증) · gap-detector(설계↔구현 격차) · report-generator(완료 리포트) 정도입니다.

  • self-healing — 오류가 나면 원인을 찾아 스스로 고치기

📋 PM / 기획

  • pm-lead — 기획 도우미 팀 전체를 이끄는 팀장
  • pm-discovery — 제품 아이디어를 단계별로 파고들어 "무엇을 만들지"를 발굴

● 하네스 Harness 결과를 반복 가능하고 안정적으로 만듭니다
STEP 2 하네스 구축 규칙(CLAUDE.md)과 검증 루프 만들기
같은 지시라도 결과가 들쭉날쭉한 이유(불안정성)와 하네스가 왜 필요한지는 이론 3장에서 다룹니다. 여기서는 그 불안정성을 줄이는 규칙 → 작업 → 절차 → 검증의 하네스를 실제로 세웁니다. 프로젝트 규칙서(CLAUDE.md)를 만들고, 변경 뒤 스스로 검증하는 검증 루프를 절차로 심습니다.
관련 이론
하네스 ① — 왜 필요한가 하네스 ② — 규칙·검증 루프
단계별 실행
  1. CLAUDE.md 초안 만들기

    아래처럼 자연어로 부탁하면 Claude가 프로젝트를 살펴 CLAUDE.md 초안을 만들어 줍니다.

    프롬프트 
    이 프로젝트를 살펴보고, 프로젝트 규칙서인
CLAUDE.md 초안을 만들어줘.

    ▲ 같은 일을 하는 명령: /init — 프롬프트에 이렇게 입력해도 됩니다.

  2. 규칙 추가하기

    내 프로젝트에 맞는 규칙을 덧붙입니다. 아래처럼 요청해 CLAUDE.md에 규칙을 채웁니다. trash-can 규칙은 Claude가 파일을 바로 지우지 않고 휴지통 폴더에 모아만 두게 해, 실수로 삭제되는 일을 막는 안전장치입니다. 맨 아래 서브에이전트 규칙은 보조 에이전트가 함부로 늘어나지 않도록, 사용자 허락이나 직접 요청이 있을 때만 만들게 하는 안전장치입니다.

    프롬프트 
    CLAUDE.md에 아래 규칙을 추가해줘.
- 모든 설명과 주석은 한국어로 작성한다.
- 새 파일은 my-app 폴더 안에만 만든다.
- 코드를 바꾸면 반드시 무엇을 왜 바꿨는지 한 줄로 알려준다.
- .env 등 비밀 정보 파일은 절대 커밋하지 않는다.
- 파일을 지워야 할 때는 바로 삭제하지 말고, trash-can 폴더를 만들어
  그 안으로 옮겨만 둔다. 작업이 끝난 뒤 내가 직접 확인하고 삭제하겠다.
- 서브에이전트는 메인 에이전트가 필요하다고 판단해 사용자 허락을 받았을 때,
  또는 사용자가 직접 요청했을 때만 만든다.
  3. 검증 루프 만들기 — "루프 엔지니어링"

    변경할 때마다 결과를 스스로 확인하고, 문제가 있으면 고치는 것까지 한 묶음으로 돌게 만듭니다. 이렇게 변경 → 검증 → 수정을 하나의 반복 고리로 설계하는 것을 루프 엔지니어링(loop engineering) — 곧 "검증을 루프로" 만드는 일 — 이라고 합니다. 아래처럼 요청해 이 검증 루프를 CLAUDE.md의 작업 절차로 심어 둡니다.

    프롬프트 
    CLAUDE.md에 "작업 절차(검증 루프)" 섹션을 추가해줘.
아래를 매번 반복(루프)하도록 적어줘.
1) 변경한다 → 2) 결과를 직접 확인한다(브라우저로 열기/실행)
→ 3) 스스로 코드 리뷰한다 → 4) 문제가 있으면 고치고 다시 1)로
→ 통과하면 무엇을 왜 바꿨는지 한 줄로 요약한다.

    ▲ 1)~4)를 반복해 돌다가, ② 결과 확인·③ 코드 리뷰에서 문제가 없으면 그때 루프를 "통과"합니다. 통과하면 한 줄 요약으로 마무리하고, 문제가 남아 있으면 고쳐서 다시 1)로 돌아갑니다.

  4. 규칙·검증 루프가 지켜지는지 확인

    간단한 변경을 하나 요청해, 이제 CLAUDE.md의 규칙(한국어·폴더·요약)과 검증 루프가 매번 자동으로 지켜지는지 확인합니다. 규칙과 루프가 결과를 일정하게 잡아 줍니다. (여기서 만든 임시 파일은 STEP 4에서 정리합니다.)

    프롬프트 
    my-app 폴더에 test.html을 만들어줘.
"안녕하세요"를 보여주는 간단한 안내 페이지야.
하네스는 한 번에 완성하지 않습니다. 작업하다 "또 이 실수를 하네" 싶은 것을 그때그때 CLAUDE.md에 규칙으로 추가하면, 프로젝트가 진행될수록 점점 안정됩니다.
● PDCA Plan & Design PRD를 계획과 설계로 좁혀 갑니다
STEP 3 Plan·Design PRD 기반 Plan·Design 문서 작성하기
PDCA 첫 바퀴의 Plan(계획)과 Design(설계)을 실제 문서로 남깁니다. Day 2의 PRD.md를 입력으로, PLAN.md와 DESIGN.md를 만듭니다.
관련 이론
PDCA 방법론 PDCA의 Plan·Design
단계별 실행
  1. Plan 만들기 — 작업을 순서로

    PRD의 기능·개발 단위를 만들 순서가 있는 작업 목록으로 바꿔 PLAN.md에 저장합니다. 이 과정의 PRD.md는 이미 MVP 규모로 좁혀 둔 것이므로, 범위를 더 잘라내지 말고 PRD 전체를 한 번에 계획으로 옮깁니다 — 대신 만들 순서만 "먼저 되면 나머지가 쉬워지는 것부터"로 잡습니다.

    프롬프트 
    PRD.md를 읽고, PRD에 정의된 기능 전체를 만들기 위한
PLAN.md를 만들어줘.
(이 PRD는 이미 MVP 범위라 따로 잘라내지 말고 전체를 담아줘.)
- 해야 할 작업을 만들 순서대로 번호로 정리
- 각 작업은 한 줄, 먼저 되면 나머지가 쉬워지는 것부터
- 맨 위에 이번 사이클 목표와 성공 기준을 적어줘
  2. Design 만들기 — 어떻게 구현할지

    각 작업을 어떻게 만들지(화면 구성, 데이터 흐름, 기술 선택)를 DESIGN.md로 정리합니다. 스택에 정답은 없습니다 — 내 PRD에 맞게, 다음 날(Do) 실제로 작업할 기술 스택을 고릅니다.

    프롬프트 
    PLAN.md를 바탕으로 DESIGN.md를 만들어줘.
- 화면 구성(무엇이 어디에 보이는지)
- 데이터 흐름(입력 → 처리 → 출력)
- 기술 선택: 내 PRD에 어울리는 기술을 자유롭게 제안해줘.
  제안은 구현 난이도별로, 비개발자인 내가 이해할 수 있는
  쉬운 설명 1줄로 3가지 옵션을 제시해줘.
핵심은 PRD 범위를 넘어 새 기능을 더하지 않는 것입니다. 이 PRD는 이미 하루 안에 만들 수 있는 MVP 크기로 좁혀 둔 것이니, Plan·Design에서는 범위를 늘리지 말고 만들 순서와 구현 방법에만 집중하면 다음 날 Do(만들기)가 3시간 안에 끝납니다.
STEP 4 정합성 검토 PRD·Plan·Design 정합성 검토하기
지금까지 만든 PRD.md · PLAN.md · DESIGN.md 세 문서가 서로 어긋나는 곳 없이 일치하는지 검토하고, 오늘 산출물을 정리합니다. 참고: PDCA의 Check(계획대로 동작하는지 검증)는 실제로 만들어 본 뒤라야 가능하므로 Day 4의 Do 다음에 진행합니다. 오늘은 세 문서의 정합성만 맞춥니다.
관련 이론
PDCA의 Plan·Design
단계별 실행
  1. 세 문서 정합성 검토

    PRD.md가 정한 "무엇을·왜"와, PLAN.md의 "순서", DESIGN.md의 "어떻게"가 서로 어긋나거나 빠진 곳이 없는지 함께 검토합니다.

    프롬프트 
    PRD.md · PLAN.md · DESIGN.md 세 문서를 함께 읽고,
서로 어긋나는 부분이나 빠진 부분이 있는지 검토해서 알려줘.
반영할 것을 고르면 파일을 업데이트해줘.
  2. 오늘 산출물 확인하고 정리하기

    CLAUDE.md · PLAN.md · DESIGN.md가 모두 있는지 확인하고, 실습 중 만든 임시 파일(예: STEP 2의 test.html)은 지워 작업 폴더를 깨끗이 둡니다.

    프롬프트 
    작업 폴더에 CLAUDE.md, PLAN.md, DESIGN.md가 모두 있는지 확인해줘.
그리고 실습 중 만든 test.html 같은 임시 파일이 있으면 지워서
폴더를 깨끗하게 정리해줘.
오늘의 마무리: Claude Code를 하네스로 다잡고, PRD를 PLAN.md·DESIGN.md로 좁힌 뒤 세 문서의 정합성까지 맞췄습니다. 이제 다음은 PDCA의 Do(만들기) — 계획대로 실제 앱을 구현하고 Check로 검증하는 단계입니다.
