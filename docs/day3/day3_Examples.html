Day 3 · Claude Code 심화 — 스킬·플러그인·하네스·PDCA (실습)

이론에서 익힌 스킬·플러그인·MCP·하네스·PDCA를 Claude Code에서 직접 설정하고 적용하는 실습입니다. STEP 1부터 순서대로 진행하십시오.
개념이 헷갈리면 각 단계의 "관련 이론" 링크로 이론 문서를 참고하세요.

진행 순서STEP 1·2 확장 → STEP 3·4 하네스 → STEP 5·6 PDCA 계획·설계
사용 도구Claude Code(CLI) — 설치되어 있다고 가정합니다
오늘의 결과물CLAUDE.md(규칙) · PLAN.md · DESIGN.md
Day 3은 Day 1·2에서 쓴 Claude Desktop 대신 Claude Code(터미널)를 사용합니다. 설치되어 있다고 가정하며, 미설치라면 Claude Code 공식 문서를 참고해 먼저 설치하세요. 작업 폴더는 Day 2의 doc-qa-bot을 이어서 씁니다.
● 확장 기능 Extend Claude Code의 능력을 넓힙니다
STEP 1 스킬·플러그인 스킬을 확인하고 플러그인 설치해 보기
Claude Code를 실행해 지금 쓸 수 있는 명령·스킬을 살펴보고, 마켓플레이스에서 플러그인을 하나 설치해 봅니다.
"매번 설명하던 것을 한 번 정의해 재사용"하는 감을 잡는 단계입니다.
관련 이론
스킬과 플러그인
단계별 실행
  1. 작업 폴더에서 Claude Code 실행

    터미널에서 Day 2의 doc-qa-bot 폴더로 이동한 뒤 claude를 실행합니다.

    cd doc-qa-bot
claude
  2. 쓸 수 있는 명령·스킬 살펴보기

    대화가 시작되면 /help로 명령 목록을 보고, 지금 쓸 수 있는 스킬을 Claude에게 직접 물어봅니다.

    /help
    프롬프트 
    지금 이 프로젝트에서 내가 쓸 수 있는 스킬과
슬래시 명령에는 어떤 것들이 있어? 간단히 설명해줘.
  3. 플러그인 마켓플레이스 열기

    /plugin을 입력해 플러그인 마켓플레이스를 엽니다. 어떤 플러그인이 있는지 목록을 둘러봅니다.

    /plugin
  4. 플러그인 하나 설치해 보기

    마음에 드는 플러그인을 하나 골라 안내에 따라 설치합니다. 설치 후 어떤 슬래시 명령·스킬이 새로 생겼는지 /help로 다시 확인합니다.(플러그인은 명령·스킬·MCP·훅을 한 번에 추가하므로, 설치 전 무엇이 추가되는지 설명을 확인하세요.)

스킬은 대부분 상황에 맞을 때 Claude가 알아서 꺼내 씁니다. 억지로 외워 부르기보다, "이런 작업엔 이런 스킬이 있다"는 지도만 익혀 두면 됩니다.
STEP 2 MCP MCP로 외부 도구·데이터 연결하기
Claude Code에 MCP 서버를 하나 연결해, Claude가 바깥 도구·데이터에 접근하게 합니다. 여기서는 안전한 예로 파일시스템 서버를 연결해 봅니다.
관련 이론
외부 도구·데이터 연결 (MCP)
단계별 실행
  1. 현재 연결된 MCP 서버 확인

    Claude Code 대화에서 /mcp를 입력하면 지금 연결된 서버 목록과 상태를 볼 수 있습니다.

    /mcp
  2. 파일시스템 MCP 서버 추가

    새 터미널(또는 Claude Code를 잠시 종료)에서 아래 명령으로 doc-qa-bot 폴더에 접근하는 파일시스템 서버를 추가합니다.(경로는 본인 doc-qa-bot의 전체 경로로 바꿉니다.)

    claude mcp add filesystem \
  -- npx -y @modelcontextprotocol/server-filesystem C:\Users\사용자명\doc-qa-bot

    ▲ macOS는 경로를 /Users/사용자명/doc-qa-bot 로 씁니다.

  3. 연결 확인

    다시 claude를 실행하고 /mcp로 filesystem 서버가 연결됐는지 확인합니다. 그 뒤 아래처럼 서버를 통해 파일을 다루게 해 봅니다.

    프롬프트 
    filesystem 서버로 이 폴더의 파일 목록을 보여주고,
PRD.md의 핵심 기능 부분을 요약해줘.
MCP 서버는 Claude에게 실제 권한을 줍니다. 실습 폴더처럼 꼭 필요한 범위만 연결하고, 출처가 불분명한 서버는 연결하지 마십시오.
● 하네스 Harness 결과를 반복 가능하고 안정적으로 만듭니다
STEP 3 개념 체험 불안정성을 직접 관찰하기
하네스가 왜 필요한지 몸으로 느끼는 단계입니다. 같은 요청을 두 번 실행해, 결과가 조금씩 달라지는 것을 직접 확인합니다.
관련 이론
하네스 엔지니어링 ① — 왜 필요한가
단계별 실행
  1. 같은 요청을 두 번 하기

    새 대화를 두 번 열어(또는 /clear 후) 아래 같은 요청을 각각 실행하고, 결과 파일을 비교합니다.

    프롬프트 (두 번 실행) 
    doc-qa-bot 폴더에 sample.html을 만들어줘.
"안녕하세요"를 보여주는 간단한 안내 페이지야.
  2. 차이 관찰하기

    두 결과가 구조·스타일·문구·파일 구성 면에서 어떻게 다른지 봅니다. 큰 틀은 같아도 세부는 매번 다릅니다 — 이것이 규칙 없는 상태의 불안정성입니다.

    결과가 달라지는 것은 고장이 아니라 LLM의 본래 성질입니다. 그래서 "매번 같은 기준"이 필요하고, 그 기준을 문서로 고정하는 것이 다음 단계(하네스 구축)입니다.
STEP 4 하네스 구축 규칙(CLAUDE.md)과 검증 절차 만들기
불안정성을 줄이기 위해 규칙 → 작업 → 절차 → 검증의 하네스를 세웁니다. 프로젝트 규칙서(CLAUDE.md)를 만들고, 변경 뒤 검증하는 절차를 정합니다.
관련 이론
하네스 엔지니어링 ② — 구축
단계별 실행
  1. CLAUDE.md 초안 만들기

    Claude Code에서 /init을 실행하면 프로젝트를 살펴 CLAUDE.md 초안을 만들어 줍니다.

    /init
  2. 규칙 추가하기

    내 프로젝트에 맞는 규칙을 덧붙입니다. 아래처럼 요청해 CLAUDE.md에 규칙을 채웁니다.

    프롬프트 
    CLAUDE.md에 아래 규칙을 추가해줘.
- 모든 설명과 주석은 한국어로 작성한다.
- 새 파일은 doc-qa-bot 폴더 안에만 만든다.
- 코드를 바꾸면 반드시 무엇을 왜 바꿨는지 한 줄로 알려준다.
- .env 등 비밀 정보 파일은 절대 커밋하지 않는다.
  3. 검증 절차 정하기

    변경 뒤 항상 거칠 확인 절차를 정해 규칙으로 적어 둡니다. 지금은 코드가 많지 않으니 "브라우저로 열어 동작 확인 + 리뷰"를 기본 절차로 삼습니다.

    프롬프트 
    CLAUDE.md에 "작업 절차" 섹션을 추가해줘.
1) 변경 → 2) 결과를 직접 확인(브라우저/실행) →
3) 스스로 코드 리뷰 → 4) 무엇을 왜 바꿨는지 요약
이 순서를 항상 따르도록 적어줘.
  4. 하네스 효과 확인

    STEP 3의 요청을 다시 실행해 봅니다. 이제 CLAUDE.md 규칙(한국어·폴더·요약)이 매번 지켜지는지 확인합니다. 규칙이 결과를 일정하게 잡아 줍니다.

하네스는 한 번에 완성하지 않습니다. 작업하다 "또 이 실수를 하네" 싶은 것을 그때그때 CLAUDE.md에 규칙으로 추가하면, 프로젝트가 진행될수록 점점 안정됩니다.
● PDCA Plan & Design PRD를 계획과 설계로 좁혀 갑니다
STEP 5 PDCA 내 프로젝트로 PDCA 한 사이클 설계하기
PDCA(Plan-Do-Check-Act)를 내 앱(doc-qa-bot)에 대입해, 첫 한 바퀴를 어떻게 돌릴지 정리합니다. 이 과정을 bkit(PDCA 기반 개발 흐름)이라 부릅니다.
관련 이론
PDCA 방법론
단계별 실행
  1. PDCA 4단계를 내 앱에 대입

    PRD의 핵심 기능 중 가장 작은 동작 버전(MVP) 하나를 골라, 그것을 한 사이클로 돌린다고 보고 아래 표를 채웁니다. Claude에게 도움을 받아도 됩니다.

    프롬프트 
    PRD.md를 참고해서, 첫 PDCA 한 사이클 계획을 세워줘.
가장 작은 동작 버전(MVP) 하나를 정하고,
Plan / Do / Check / Act 각각에서 무엇을 할지
표로 정리해줘. 아직 파일은 만들지 말고 표만 보여줘.
  2. 한 사이클의 목표 정하기

    이번 바퀴에서 "무엇이 되면 성공"인지 한 문장으로 정합니다. 너무 크게 잡지 말고, 하루 안에 확인 가능한 크기로 좁힙니다.

    예시 — 첫 사이클 목표
    MVP: 문서 1개를 올리면, 그 문서 내용으로 질문에 답하는 최소 화면.
    성공 기준: 샘플 문서를 올려 질문 3개에 근거와 함께 답하면 성공.
PDCA의 핵심은 작게 돌리는 것입니다. "앱 전체"가 아니라 "기능 하나"를 한 바퀴로 잡아야 Check·Act가 빨리 돌아 개선이 쌓입니다.
STEP 6 Plan·Design PRD 기반 Plan·Design 문서 작성하기
PDCA의 Plan(계획)과 Design(설계)을 실제 문서로 남깁니다. Day 2의 PRD.md를 입력으로, PLAN.md와 DESIGN.md를 만듭니다.
관련 이론
PDCA의 Plan·Design
단계별 실행
  1. Plan 만들기 — 작업을 순서로

    PRD의 기능·개발 단위를 만들 순서가 있는 작업 목록으로 바꿔 PLAN.md에 저장합니다.

    프롬프트 
    PRD.md를 읽고, STEP 5에서 정한 MVP를 만들기 위한
PLAN.md를 만들어줘.
- 해야 할 작업을 만들 순서대로 번호로 정리
- 각 작업은 한 줄, 가장 작은 것부터
- 맨 위에 이번 사이클 목표와 성공 기준을 적어줘
  2. Design 만들기 — 어떻게 구현할지

    각 작업을 어떻게 만들지(화면 구성, 데이터 흐름)를 DESIGN.md로 정리합니다.

    프롬프트 
    PLAN.md를 바탕으로 DESIGN.md를 만들어줘.
- 화면 구성(무엇이 어디에 보이는지)
- 데이터 흐름(문서 업로드 → 저장 → 검색 → 답변)
- 사용할 기술의 큰 선택
초보자가 이해할 수준으로 간단히 적어줘.
  3. 검토·정리

    PLAN.md와 DESIGN.md를 열어, PRD와 어긋나는 곳이 없는지 확인합니다. 애매한 부분은 아래처럼 다시 다듬습니다.

    프롬프트 
    PLAN.md와 DESIGN.md가 PRD.md와 어긋나는 부분,
또는 빠진 부분이 있는지 검토해서 알려줘.
반영할 것을 고르면 파일을 업데이트해줘.
오늘의 마무리: Claude Code를 하네스로 다잡고, PRD를 PLAN.md·DESIGN.md로 좁혔습니다. 이제 다음은 PDCA의 Do(만들기) — 계획대로 실제 앱을 구현하는 단계입니다.
