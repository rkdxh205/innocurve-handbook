Day 3 · Claude Code 심화 — 스킬·플러그인·하네스·PDCA (실습)

이론에서 익힌 스킬·플러그인·하네스·PDCA를 Claude Code에서 직접 설정하고 적용하는 실습입니다. STEP 1부터 순서대로 진행하십시오.
개념이 헷갈리면 각 단계의 "관련 이론" 링크로 이론 문서를 참고하세요.

진행 순서STEP 1 확장 → STEP 2 하네스 → STEP 3·4 계획·설계·검토
사용 도구Claude Desktop의 Code 탭 — Day 1에서 설치(앱 상단 Chat / Cowork / Code 중 "Code")
오늘의 결과물CLAUDE.md(규칙) · PLAN.md · DESIGN.md
작업 폴더는 PRD.md가 있는 Day 2의 my-app을 이어서 씁니다.
● 확장 기능 Extend Claude Code의 능력을 넓힙니다
STEP 1 스킬·플러그인 스킬·플러그인·서브에이전트 확인하고 설치하기
Code 탭에서 지금 쓸 수 있는 명령·스킬을 살펴보고, 이 과정의 핵심 도구인 bkit 플러그인을 설치합니다. bkit은 Day 3 STEP 2~4·Day 4에서 쓸 PDCA 하네스 엔진입니다 — "매번 설명하던 것을 한 번 정의해 재사용"하는 플러그인의 힘을 이걸로 체감합니다.
관련 이론
스킬과 플러그인
단계별 실행
  1. Code 탭에서 새 세션 시작 (작업 폴더 지정)

    Claude Desktop을 실행하고 상단 Code 탭을 클릭합니다. + New session(또는 Ctrl+N)으로 새 세션을 열고, 프롬프트 영역에서 Environment: Local · Project folder: Day 2의 my-app 선택을 지정한 뒤 프롬프트를 입력합니다.이후 모든 프롬프트·명령은 이 Code 탭 화면에서 진행합니다 — 검은 터미널을 따로 열 필요가 없습니다.

  2. bkit 플러그인 설치

    이 과정의 핵심 도구인 bkit을 설치합니다. Code 탭 프롬프트에 아래 세 명령을 한 줄씩 차례로 입력합니다 — ① bkit 마켓플레이스 추가 → ② bkit 설치 → ③ 새로고침으로 활성화. 중간에 권한을 물으면 본인이 승인합니다.

    /plugin marketplace add popup-studio-ai/bkit-claude-code
/plugin install bkit
/reload-plugins
    참고: 다른 플러그인은 + 버튼 → Plugins 브라우저에서 GUI로도 둘러보고 설치할 수 있습니다.
  3. 설치 확인 — bkit이 데려온 스킬·서브에이전트 보기

    프롬프트에 /를 입력해 bkit이 추가한 새 슬래시 명령이 보이는지 확인한 뒤, 아래 두 프롬프트를 차례로 물어 bkit이 함께 데려온 스킬과 서브에이전트(특정 작업을 위임받는 보조 에이전트)를 확인합니다 — 지금은 "이런 게 있다"는 것만 눈으로 확인하면 됩니다. 이 명령·스킬·에이전트들이 Day 3 STEP 2~4·Day 4의 PDCA를 돌리는 도구가 됩니다.

    프롬프트 ① 스킬 확인 
    지금 이 프로젝트에서 내가 쓸 수 있는 스킬에는
어떤 것들이 있어? 각각 무슨 일을 하는지 한 줄로 설명해줘.
    프롬프트 ② 서브에이전트 확인 
    지금 이 프로젝트에서 쓸 수 있는 서브에이전트에는
어떤 것들이 있어? 각각 무슨 일을 하는지 한 줄로 알려줘.
방금 설치한 bkit이 이 과정의 하네스입니다 — 규칙·PDCA·검증 절차를 담고 있어 Day 3 STEP 2~4와 Day 4가 이 위에서 돌아갑니다. "플러그인 하나를 설치했을 뿐인데" 과정 전체의 뼈대가 깔린 셈입니다.
스킬은 대부분 상황에 맞을 때 Claude가 알아서 꺼내 씁니다. 억지로 외워 부르기보다, "이런 작업엔 이런 스킬이 있다"는 지도만 익혀 두면 됩니다.
● 하네스 Harness 결과를 반복 가능하고 안정적으로 만듭니다
STEP 2 하네스 구축 규칙(CLAUDE.md)과 검증 루프 만들기
같은 지시라도 결과가 들쭉날쭉한 이유(불안정성)와 하네스가 왜 필요한지는 이론 3장에서 다룹니다. 여기서는 그 불안정성을 줄이는 규칙 → 작업 → 절차 → 검증의 하네스를 실제로 세웁니다. 프로젝트 규칙서(CLAUDE.md)를 만들고, 변경 뒤 스스로 검증하는 검증 루프를 절차로 심습니다.
관련 이론
하네스 ① — 왜 필요한가 하네스 ② — 규칙·검증 루프
단계별 실행
  1. CLAUDE.md 초안 만들기

    Claude Code에서 /init을 실행하면 프로젝트를 살펴 CLAUDE.md 초안을 만들어 줍니다.

    /init
  2. 규칙 추가하기

    내 프로젝트에 맞는 규칙을 덧붙입니다. 아래처럼 요청해 CLAUDE.md에 규칙을 채웁니다. 맨 아래 서브에이전트 규칙은 보조 에이전트가 함부로 늘어나지 않도록, 사용자 허락이나 직접 요청이 있을 때만 만들게 하는 안전장치입니다.

    프롬프트 
    CLAUDE.md에 아래 규칙을 추가해줘.
- 모든 설명과 주석은 한국어로 작성한다.
- 새 파일은 my-app 폴더 안에만 만든다.
- 코드를 바꾸면 반드시 무엇을 왜 바꿨는지 한 줄로 알려준다.
- .env 등 비밀 정보 파일은 절대 커밋하지 않는다.
- 서브에이전트는 메인 에이전트가 필요하다고 판단해 사용자 허락을 받았을 때,
  또는 사용자가 직접 요청했을 때만 만든다.
  3. 검증 루프 만들기 — "루프 엔지니어링"

    변경할 때마다 결과를 스스로 확인하고, 문제가 있으면 고치는 것까지 한 묶음으로 돌게 만듭니다. 이렇게 변경 → 검증 → 수정을 하나의 반복 고리로 설계하는 것을 루프 엔지니어링(loop engineering) — 곧 "검증을 루프로" 만드는 일 — 이라고 합니다. 아래처럼 요청해 이 검증 루프를 CLAUDE.md의 작업 절차로 심어 둡니다.

    프롬프트 
    CLAUDE.md에 "작업 절차(검증 루프)" 섹션을 추가해줘.
아래를 매번 반복(루프)하도록 적어줘.
1) 변경한다 → 2) 결과를 직접 확인한다(브라우저로 열기/실행)
→ 3) 스스로 코드 리뷰한다 → 4) 문제가 있으면 고치고 다시 1)로
→ 통과하면 무엇을 왜 바꿨는지 한 줄로 요약한다.
  4. 규칙·검증 루프가 지켜지는지 확인

    간단한 변경을 하나 요청해, 이제 CLAUDE.md의 규칙(한국어·폴더·요약)과 검증 루프가 매번 자동으로 지켜지는지 확인합니다. 규칙과 루프가 결과를 일정하게 잡아 줍니다. (여기서 만든 임시 파일은 STEP 4에서 정리합니다.)

    프롬프트 
    my-app 폴더에 test.html을 만들어줘.
"안녕하세요"를 보여주는 간단한 안내 페이지야.
하네스는 한 번에 완성하지 않습니다. 작업하다 "또 이 실수를 하네" 싶은 것을 그때그때 CLAUDE.md에 규칙으로 추가하면, 프로젝트가 진행될수록 점점 안정됩니다.
● PDCA Plan & Design PRD를 계획과 설계로 좁혀 갑니다
STEP 3 Plan·Design PRD 기반 Plan·Design 문서 작성하기
PDCA 첫 바퀴의 Plan(계획)과 Design(설계)을 실제 문서로 남깁니다. Day 2의 PRD.md를 입력으로, PLAN.md와 DESIGN.md를 만듭니다.
관련 이론
PDCA 방법론 PDCA의 Plan·Design
단계별 실행
  1. Plan 만들기 — 작업을 순서로

    PRD의 기능·개발 단위를 만들 순서가 있는 작업 목록으로 바꿔 PLAN.md에 저장합니다. 이 과정의 PRD.md는 이미 MVP 규모로 좁혀 둔 것이므로, 범위를 더 잘라내지 말고 PRD 전체를 한 번에 계획으로 옮깁니다 — 대신 만들 순서만 "먼저 되면 나머지가 쉬워지는 것부터"로 잡습니다.

    프롬프트 
    PRD.md를 읽고, PRD에 정의된 기능 전체를 만들기 위한
PLAN.md를 만들어줘.
(이 PRD는 이미 MVP 범위라 따로 잘라내지 말고 전체를 담아줘.)
- 해야 할 작업을 만들 순서대로 번호로 정리
- 각 작업은 한 줄, 먼저 되면 나머지가 쉬워지는 것부터
- 맨 위에 이번 사이클 목표와 성공 기준을 적어줘
  2. Design 만들기 — 어떻게 구현할지

    각 작업을 어떻게 만들지(화면 구성, 데이터 흐름, 기술 선택)를 DESIGN.md로 정리합니다. 스택에 정답은 없습니다 — 내 PRD에 맞게, 다음 날(Do) 실제로 작업할 기술 스택을 고릅니다.

    프롬프트 
    PLAN.md를 바탕으로 DESIGN.md를 만들어줘.
- 화면 구성(무엇이 어디에 보이는지)
- 데이터 흐름(입력 → 처리 → 출력)
- 기술 선택: 내 PRD에 어울리는 기술을 자유롭게 제안해줘.
  제안은 구현 난이도별로, 비개발자인 내가 이해할 수 있는
  쉬운 설명 1줄로 3가지 옵션을 제시해줘.
핵심은 PRD 범위를 넘어 새 기능을 더하지 않는 것입니다. 이 PRD는 이미 하루 안에 만들 수 있는 MVP 크기로 좁혀 둔 것이니, Plan·Design에서는 범위를 늘리지 말고 만들 순서와 구현 방법에만 집중하면 다음 날 Do(만들기)가 3시간 안에 끝납니다.
STEP 4 정합성 검토 PRD·Plan·Design 정합성 검토하기
지금까지 만든 PRD.md · PLAN.md · DESIGN.md 세 문서가 서로 어긋나는 곳 없이 일치하는지 검토하고, 오늘 산출물을 정리합니다. 참고: PDCA의 Check(계획대로 동작하는지 검증)는 실제로 만들어 본 뒤라야 가능하므로 Day 4의 Do 다음에 진행합니다. 오늘은 세 문서의 정합성만 맞춥니다.
관련 이론
PDCA의 Plan·Design
단계별 실행
  1. 세 문서 정합성 검토

    PRD.md가 정한 "무엇을·왜"와, PLAN.md의 "순서", DESIGN.md의 "어떻게"가 서로 어긋나거나 빠진 곳이 없는지 함께 검토합니다.

    프롬프트 
    PRD.md · PLAN.md · DESIGN.md 세 문서를 함께 읽고,
서로 어긋나는 부분이나 빠진 부분이 있는지 검토해서 알려줘.
반영할 것을 고르면 파일을 업데이트해줘.
  2. 오늘 산출물 확인하고 정리하기

    CLAUDE.md · PLAN.md · DESIGN.md가 모두 있는지 확인하고, 실습 중 만든 임시 파일(예: STEP 2의 test.html)은 지워 작업 폴더를 깨끗이 둡니다.

    프롬프트 
    작업 폴더에 CLAUDE.md, PLAN.md, DESIGN.md가 모두 있는지 확인해줘.
그리고 실습 중 만든 test.html 같은 임시 파일이 있으면 지워서
폴더를 깨끗하게 정리해줘.
오늘의 마무리: Claude Code를 하네스로 다잡고, PRD를 PLAN.md·DESIGN.md로 좁힌 뒤 세 문서의 정합성까지 맞췄습니다. 이제 다음은 PDCA의 Do(만들기) — 계획대로 실제 앱을 구현하고 Check로 검증하는 단계입니다.
