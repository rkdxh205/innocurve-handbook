1. 스킬과 플러그인

Claude Code는 기본 상태로도 강력하지만, 스킬과 플러그인으로 능력을 넓힐 수 있습니다. 둘 다 "Claude에게 새로운 능력·절차를 얹는" 장치이지만 규모가 다릅니다.

Skill — 반복 절차를 담은 능력 꾸러미

스킬(Skill)은 특정 작업을 하는 방법을 정리한 설명서 묶음입니다. "코드 리뷰는 이런 순서로", "배포는 이런 절차로" 같은 정형 절차를 담아 두면, Claude Code가 상황에 맞을 때 알아서 그 스킬을 꺼내 쓰거나, 사용자가 직접 불러 쓸 수 있습니다. 매번 길게 설명하지 않아도 같은 품질의 절차가 재현된다는 것이 핵심입니다.

Plugin — 능력을 묶은 확장 패키지

플러그인(Plugin)은 슬래시 명령·서브에이전트·스킬·MCP 서버·훅을 하나로 묶은 패키지입니다. 마켓플레이스에서 골라 설치하면 여러 능력이 한 번에 추가됩니다. 스킬이 "레시피 카드 한 장"이라면, 플러그인은 "레시피북 + 도구 세트"에 가깝습니다.

용어한 줄 정의비유
Skill정형 절차를 담은 능력 꾸러미레시피 카드
Plugin명령·에이전트·스킬·MCP·훅 묶음레시피북 + 도구 세트
Slash 명령/이름으로 부르는 정해진 작업단축 버튼
Subagent특정 일을 위임받는 보조 에이전트전담 담당자

💡 핵심 한 줄: 스킬·플러그인은 "매번 설명하던 것을 한 번 정의해 재사용"하게 해 주는 장치입니다 — 하네스의 재료가 됩니다.

Claude Code에서 사용 가능한 스킬을 확인하고 플러그인을 설치해 보는 과정은 실습 STEP 1에서 진행합니다.

2. 외부 도구·데이터 연결 — MCP

MCP(Model Context Protocol)는 Claude를 내 컴퓨터·외부 서비스의 도구와 데이터에 잇는 표준 규격입니다. Day 1에서 Claude Desktop에 "로컬 폴더"를 지정한 것도 사실은 파일시스템 MCP의 한 형태였습니다.

MCP 서버 — 능력을 제공하는 연결 상대

MCP는 "서버"가 도구(tool)와 자원(resource)을 제공하고, Claude가 필요할 때 그것을 호출하는 방식으로 동작합니다. 덕분에 Claude 혼자서는 못 하는 일 — 내 파일 읽고 쓰기, GitHub 저장소 다루기, 데이터베이스 조회, 외부 API 호출 — 을 연결된 서버에 위임할 수 있습니다.

MCP 서버 예연결하면 할 수 있는 일
filesystem지정한 폴더의 파일 읽기·쓰기
github이슈·PR·저장소 내용 다루기
데이터베이스사내 DB에 질의하고 결과 받기
웹·검색최신 정보를 찾아 답에 반영
MCP 서버는 Claude에게 실제 권한을 넘겨줍니다. 신뢰할 수 있는 서버만, 그리고 꼭 필요한 범위(폴더·권한)만 연결하십시오.

💡 핵심 한 줄: MCP는 "Claude가 바깥세상과 연결되는 표준 콘센트"입니다 — 플러그인이 이 서버들을 함께 담아 오기도 합니다.

MCP 서버를 하나 연결하고 /mcp로 상태를 확인하는 과정은 실습 STEP 2에서 진행합니다.

3. 하네스 엔지니어링 ① — 왜 필요한가

불안정성 — 한 번 잘 됐다고 매번 잘 되진 않는다

LLM은 그럴듯한 다음 문장을 매번 새로 생성합니다. 그래서 같은 지시를 줘도 결과가 조금씩 달라지고, 한 번 잘 나온 결과가 다음에도 나온다는 보장이 없습니다. 혼자 쓸 때는 넘어갈 수 있지만, 남에게 보여 주거나 반복 운영하는 앱에서는 이 들쭉날쭉함이 문제가 됩니다.

Harness — 모델을 둘러싸는 틀

하네스(harness)는 원래 "말이나 장비를 붙들어 매는 마구·안전벨트"를 뜻합니다. 하네스 엔지니어링은 똑똑하지만 변덕스러운 모델 주위에 규칙·맥락·도구·검증이라는 틀을 둘러, 결과를 반복 가능하고 안정적으로 만드는 일입니다. 아무리 좋은 요리사도 계량·레시피·간 보기 없이는 매번 맛이 다르듯, 좋은 모델도 하네스 없이는 결과가 흔들립니다.

하네스 없음:   지시 → [모델] → 매번 다른 결과 (운에 맡김)

하네스 있음:   지시 → [규칙 + 도구 + 절차 + 검증]로 둘러싼 모델
                        → 반복 가능한 안정적 결과

💡 핵심 한 줄: 하네스 엔지니어링의 목표는 "천재적인 한 방"이 아니라 "믿을 수 있는 반복"입니다.

같은 요청을 두 번 실행해 결과가 달라지는 것을 직접 관찰하고, 왜 하네스가 필요한지 체감하는 과정은 실습 STEP 3에서 진행합니다.

4. 하네스 엔지니어링 ② — 규칙·작업·절차·검증

하네스는 네 가지 기둥으로 세웁니다. 이 넷을 갖추면 "매번 설명 → 매번 다른 결과"가 "한 번 정의 → 반복되는 결과"로 바뀝니다.

기둥무엇을 하나Claude Code의 도구
① 규칙(Rules)프로젝트 규칙·컨벤션·금지사항을 명문화CLAUDE.md
② 작업(Tasks)자주 하는 일을 정형화해 재사용슬래시 명령 · 스킬
③ 절차(Procedure)일의 순서를 정함(계획→구현→검증)워크플로 · 서브에이전트
④ 검증(Verification)결과가 규칙·의도대로인지 확인테스트 · 리뷰 · 훅(hook)

CLAUDE.md — 프로젝트의 규칙서

CLAUDE.md는 Claude Code가 매 작업에서 참고하는 프로젝트 규칙 문서입니다. 사용할 언어·코드 스타일·폴더 구조·해서는 안 되는 일 등을 적어 두면, 매번 지시하지 않아도 그 규칙 위에서 작업합니다.

Hook — 정해진 순간에 자동으로 실행되는 검증

훅(hook)은 특정 사건이 일어날 때 자동으로 실행되는 명령입니다. 예를 들어 "코드를 고친 뒤에는 항상 자동으로 검사를 돌린다"처럼, 사람이 잊어도 검증이 빠지지 않게 만들어 줍니다.

💡 핵심 한 줄: 규칙으로 방향을, 작업·절차로 반복을, 검증으로 신뢰를 만듭니다.

CLAUDE.md로 규칙을 세우고 변경 뒤 검증 절차를 갖추는 과정은 실습 STEP 4에서 진행합니다.

5. PDCA 방법론

하네스가 "틀"이라면, PDCA는 그 틀 위에서 일을 굴리는 방식입니다. PDCA(Plan-Do-Check-Act)는 계획하고, 실행하고, 확인하고, 개선하는 것을 한 바퀴씩 반복하며 품질을 높이는 오래된 개선 사이클입니다. 이 과정에서는 이 PDCA를 Claude Code 개발에 적용한 흐름을 bkit이라 부릅니다.

        ┌──────────────────────────────┐
        ▼                              │
   [Plan] 무엇을·왜·어떻게 → [Do] 만든다 →
   [Check] 계획대로인지 검증 → [Act] 고치고 표준화 ┘
        (한 바퀴가 끝나면 다음 바퀴로)
단계하는 일산출물(예)
Plan (계획)무엇을 어떤 순서로 만들지 정함작업 계획·범위
Do (실행)계획대로 실제로 만듦동작하는 코드
Check (검증)계획·의도대로인지 확인테스트·리뷰 결과
Act (개선)문제를 고치고 규칙으로 정착수정·표준화, 다음 계획

PDCA와 하네스는 짝을 이룹니다. 하네스(규칙·검증)가 갖춰져 있어야 Check·Act가 매번 같은 기준으로 돌아가고, PDCA를 반복할수록 하네스도 함께 단단해집니다.

💡 핵심 한 줄: PDCA는 "한 번에 완성"이 아니라 "작게 돌리고 매 바퀴 나아지는" 개발 리듬입니다.

PDCA 네 단계를 내 프로젝트(doc-qa-bot)에 대입해 한 사이클을 설계하는 과정은 실습 STEP 5에서 진행합니다.

6. PDCA의 Plan·Design — PRD 기반 계획·설계

Day 3에서는 PDCA의 첫 바퀴 중 Plan(계획)과 그 안의 Design(설계)까지 다룹니다. 출발점은 Day 2에서 만든 PRD입니다. PRD가 "무엇을·왜"라면, Plan은 "어떤 순서로", Design은 "어떻게"를 정합니다.

단계답하는 질문산출물
PRD (Day 2)무엇을·왜 만드는가PRD.md
Plan어떤 작업을, 어떤 순서로 할 것인가작업 목록·순서(PLAN.md)
Design그 작업들을 어떻게 구현할 것인가구조·데이터 흐름·화면(DESIGN.md)

Plan — PRD를 작업 순서로 풀기

PRD의 "핵심 기능"과 "개발 단위"를 만들 순서가 있는 작업 목록으로 바꿉니다. 무엇을 먼저 만들면 나머지가 수월해지는지, 한 번에 다 하지 않고 가장 작은 동작 버전(MVP)부터 세우는 것이 요령입니다.

Design — 작업을 구현으로 잇기

각 작업을 어떻게 만들지 정합니다. 화면 구성, 데이터가 흐르는 순서(문서 업로드 → 저장 → 검색 → 답변), 사용할 기술의 큰 선택을 미리 그려 두면 Do(구현) 단계에서 헤매지 않습니다.

💡 핵심 한 줄: PRD(무엇/왜) → Plan(순서) → Design(어떻게) 순으로 좁혀 가면, 다음 날의 "만들기(Do)"가 훨씬 수월해집니다.

Day 2의 PRD.md를 바탕으로 PLAN.md와 DESIGN.md를 만드는 과정은 실습 STEP 6에서 진행합니다.
