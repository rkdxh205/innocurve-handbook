1. 스킬과 플러그인

Claude Code는 기본 상태로도 강력하지만, 스킬과 플러그인으로 능력을 넓힐 수 있습니다. 둘 다 "Claude에게 새로운 능력·절차를 얹는" 장치이지만 규모가 다릅니다.

Skill — 반복 절차를 담은 능력 꾸러미

스킬(Skill)은 특정 작업을 하는 방법을 정리한 설명서 묶음입니다. 실제로는 마크다운(.md) 파일 한 장에 "코드 리뷰는 이런 순서로", "배포는 이런 절차로" 같은 정형 절차를 적어 둔 것이며 — 사람이 읽는 업무 매뉴얼과 똑같이 생겼습니다. 이렇게 담아 두면 Claude Code가 상황에 맞을 때 알아서 그 스킬을 꺼내 쓰거나, 사용자가 직접 불러 쓸 수 있습니다. 매번 길게 설명하지 않아도 같은 품질의 절차가 재현된다는 것이 핵심입니다.

Plugin — 능력을 묶은 확장 패키지

플러그인(Plugin)은 슬래시 명령·서브에이전트·스킬·MCP 서버·훅을 하나로 묶은 패키지입니다. 마켓플레이스에서 골라 설치하면 여러 능력이 한 번에 추가됩니다. 스킬이 "레시피 카드 한 장"이라면, 플러그인은 "레시피북 + 도구 세트"에 가깝습니다.

용어한 줄 정의비유
Skill정형 절차를 담은 능력 꾸러미레시피 카드
Plugin명령·에이전트·스킬·MCP·훅 묶음레시피북 + 도구 세트
Slash 명령/이름으로 부르는 정해진 작업단축 버튼
Subagent특정 일을 위임받는 보조 에이전트전담 담당자

💡 핵심 한 줄: 스킬·플러그인은 "매번 설명하던 것을 한 번 정의해 재사용"하게 해 주는 장치입니다 — 하네스의 재료가 됩니다.

사용 가능한 스킬을 확인하고, 이 과정의 하네스인 bkit 플러그인을 설치하는 과정은 실습 STEP 1에서 진행합니다.

2. 외부 도구·데이터 연결 — MCP

MCP(Model Context Protocol)는 Claude를 내 컴퓨터·외부 서비스의 도구와 데이터에 잇는 표준 규격입니다. Day 1에서 Claude Desktop에 "로컬 폴더"를 지정한 것도 사실은 파일시스템 MCP의 한 형태였습니다. 이 규격 덕분에 Claude 혼자서는 못 하는 일 — GitHub 저장소 다루기, 데이터베이스 조회, 사내 시스템 연결 — 을 연결된 서버에 위임할 수 있습니다.

다만 이 과정의 rag-chatbot은 Code 탭이 이미 접근하는 작업 폴더 안에서 끝나므로, 실습에서 따로 MCP 서버를 붙이지는 않습니다(이미 되는 것을 다시 연결할 뿐이라 배울 게 없습니다). 나중에 GitHub·Slack처럼 실제 외부 시스템이 필요해지면, 그때 + 버튼 → Connectors에서 GUI로 연결하면 됩니다. MCP 서버는 Claude에게 실제 권한을 주므로 신뢰할 수 있는 서버만, 꼭 필요한 범위만 붙이는 것이 원칙입니다.

💡 핵심 한 줄: MCP는 "Claude가 바깥세상과 연결되는 표준 콘센트"입니다 — 지금 당장은 필요 없지만, 앱을 키울 때 꺼내 쓰는 확장 통로입니다.

3. 하네스 엔지니어링 ① — 왜 필요한가

불안정성 — 한 번 잘 됐다고 매번 잘 되진 않는다

LLM은 그럴듯한 다음 문장을 매번 새로 생성합니다. 그래서 같은 지시를 줘도 결과가 조금씩 달라지고, 한 번 잘 나온 결과가 다음에도 나온다는 보장이 없습니다. 혼자 쓸 때는 넘어갈 수 있지만, 남에게 보여 주거나 반복 운영하는 앱에서는 이 들쭉날쭉함이 문제가 됩니다.

Harness — 모델을 둘러싸는 틀

하네스(harness)는 원래 "말이나 장비를 붙들어 매는 마구·안전벨트"를 뜻합니다. 하네스 엔지니어링은 똑똑하지만 변덕스러운 모델 주위에 규칙·맥락·도구·검증이라는 틀을 둘러, 결과를 반복 가능하고 안정적으로 만드는 일입니다. 아무리 좋은 요리사도 계량·레시피·간 보기 없이는 매번 맛이 다르듯, 좋은 모델도 하네스 없이는 결과가 흔들립니다.

하네스 없음:   지시 → [모델] → 매번 다른 결과 (운에 맡김)

하네스 있음:   지시 → [규칙 + 도구 + 절차 + 검증]로 둘러싼 모델
                        → 반복 가능한 안정적 결과

💡 핵심 한 줄: 하네스 엔지니어링의 목표는 "천재적인 한 방"이 아니라 "믿을 수 있는 반복"입니다.

이 불안정성을 CLAUDE.md 규칙과 검증 루프로 잡는 실습은 실습 STEP 2에서 이어집니다.

4. 하네스 엔지니어링 ② — 규칙·작업·절차·검증

하네스는 네 가지 기둥으로 세웁니다. 이 넷을 갖추면 "매번 설명 → 매번 다른 결과"가 "한 번 정의 → 반복되는 결과"로 바뀝니다.

기둥무엇을 하나Claude Code의 도구
① 규칙(Rules)프로젝트 규칙·컨벤션·금지사항을 명문화CLAUDE.md
② 작업(Tasks)자주 하는 일을 정형화해 재사용슬래시 명령 · 스킬
③ 절차(Procedure)일의 순서를 정함(계획→구현→검증)워크플로 · 서브에이전트
④ 검증(Verification)결과가 규칙·의도대로인지 확인테스트 · 리뷰 · 훅(hook)

이 네 기둥을 실제로 채우는 재료가 바로 1장에서 배운 확장 기능입니다. 하네스는 새로 발명하는 것이 아니라, 그 부품들을 목적에 맞게 끼워 맞추는 일입니다.

하네스 부품정체맡는 기둥
CLAUDE.md프로젝트 규칙을 적은 마크다운 파일① 규칙
스킬(Skill)반복 절차를 적은 .md 파일 한 장② 작업
서브에이전트(Subagent)특정 단계를 위임받는 보조 에이전트③ 절차
플러그인(Plugin)위 셋과 훅·MCP를 한 벌로 묶은 패키지①~④ 전체

💡 핵심 한 줄: 스킬·서브에이전트는 .md 파일로 만드는 하네스의 부품이고, 플러그인은 그 부품 한 벌을 통째로 배포하는 그릇입니다 — 이 과정의 bkit이 바로 그런 플러그인입니다.

CLAUDE.md — 프로젝트의 규칙서

CLAUDE.md는 Claude Code가 매 작업을 시작할 때마다 자동으로 먼저 읽는 프로젝트 규칙 문서입니다. 프로젝트 폴더 맨 위에 두는 마크다운(.md) 파일 한 장으로, 스킬과 마찬가지로 특별한 문법이 아니라 사람이 읽는 글로 규칙을 적습니다. 사람으로 치면 새 직원에게 매번 말로 일러 주던 "우리 팀은 이렇게 일합니다"를 한 번 문서로 붙여 둔 사내 규정과 같아서, 다시 설명하지 않아도 Claude가 그 규정 위에서 작업합니다. 하네스의 ① 규칙(Rules) 기둥을 담당하는 핵심 파일입니다.

보통 다음 네 가지를 적어 둡니다. 처음부터 완벽하게 채울 필요 없이, 짧고 분명하게 시작해 작업하며 늘려 가는 것이 좋습니다.

적는 것예
프로젝트 개요"이 앱은 사내 문서에 답하는 Q&A 챗봇이다"
기술·스타일 규칙"HTML/JS만 쓴다", "주석은 한국어로", "새 라이브러리는 추가하지 않는다"
폴더·파일 구조"화면은 index.html, 서버 함수는 api/ 아래에 둔다"
금지·주의사항".env·API 키는 절대 코드에 넣지 않는다", "물어보지 않고 파일을 지우지 않는다"
 
# CLAUDE.md  (프로젝트 폴더 맨 위)

## 이 프로젝트
사내 문서에 근거를 들어 답하는 Q&A 챗봇(rag-chatbot)

## 규칙
- HTML/JS로만 만든다. 새 라이브러리를 함부로 추가하지 않는다.
- 답변은 반드시 문서의 근거와 함께 낸다.
- API 키는 코드가 아니라 환경변수로만 다룬다.

## 하지 말 것
- 확인 없이 파일을 지우거나 폴더 구조를 바꾸지 않는다.

중요한 것은 한 번 쓰고 끝이 아니라는 점입니다. 작업 중 "여기서 또 같은 실수를 하네" 싶은 것을 CLAUDE.md에 규칙 한 줄로 추가해 두면, 다음부터 그 실수가 줄어듭니다. 이렇게 PDCA를 한 바퀴 돌 때마다 규칙이 쌓이면서, 하네스가 프로젝트에 맞게 점점 단단해집니다.

검증 루프 — "루프 엔지니어링"으로 스스로 확인하게 하기

④ 검증은 한 번 하고 끝이 아니라 반복되는 고리(루프)로 만들 때 힘을 발휘합니다. 변경 → 결과 확인 → 문제가 있으면 수정 → 다시 확인을 통과할 때까지 돌게 하면, 사람이 일일이 점검하지 않아도 모델이 스스로 검증하며 고쳐 나갑니다. 이렇게 검증을 루프로 설계하는 일을 루프 엔지니어링(loop engineering)이라고 합니다. 이 절차를 CLAUDE.md에 적어 두면 매 변경마다 검증 루프가 자동으로 돕니다.

Hook — 정해진 순간에 자동으로 실행되는 검증

훅(hook)은 특정 사건이 일어날 때 자동으로 실행되는 명령입니다. 예를 들어 "코드를 고친 뒤에는 항상 자동으로 검사를 돌린다"처럼, 사람이 잊어도 검증이 빠지지 않게 만들어 줍니다.

💡 핵심 한 줄: 규칙으로 방향을, 작업·절차로 반복을, 검증으로 신뢰를 만듭니다.

CLAUDE.md로 규칙을 세우고 검증 루프를 절차로 심는 과정은 실습 STEP 2에서 진행합니다.

5. PDCA 방법론

하네스가 "틀"이라면, PDCA는 그 틀 위에서 일을 굴리는 방식입니다. PDCA(Plan-Do-Check-Act)는 계획하고, 실행하고, 확인하고, 개선하는 것을 한 바퀴씩 반복하며 품질을 높이는 오래된 개선 사이클입니다. 이 과정에서는 이 PDCA를 Claude Code 개발에 적용한 흐름을 bkit이라 부르며, STEP 1에서 설치하는 bkit 플러그인이 이 흐름(규칙·PDCA·검증)을 실제 도구로 제공합니다.

        ┌──────────────────────────────┐
        ▼                              │
   [Plan] 무엇을·왜·어떻게 → [Do] 만든다 →
   [Check] 계획대로인지 검증 → [Act] 고치고 표준화 ┘
        (한 바퀴가 끝나면 다음 바퀴로)
단계하는 일산출물(예)
Plan (계획)무엇을 어떤 순서로 만들지 정함작업 계획·범위
Do (실행)계획대로 실제로 만듦동작하는 코드
Check (검증)계획·의도대로인지 확인테스트·리뷰 결과
Act (개선)문제를 고치고 규칙으로 정착수정·표준화, 다음 계획

PDCA와 하네스는 짝을 이룹니다. 하네스(규칙·검증)가 갖춰져 있어야 Check·Act가 매번 같은 기준으로 돌아가고, PDCA를 반복할수록 하네스도 함께 단단해집니다.

💡 핵심 한 줄: PDCA는 "한 번에 완성"이 아니라 "작게 돌리고 매 바퀴 나아지는" 개발 리듬입니다.

PDCA의 Plan·Design을 내 프로젝트(rag-chatbot)에 적용해 계획·설계 문서를 만들고 검토하는 과정은 실습 STEP 3·4에서 진행합니다.

6. PDCA의 Plan·Design — PRD 기반 계획·설계

Day 3에서는 PDCA의 첫 바퀴 중 Plan(계획)과 그 안의 Design(설계)까지 다룹니다. 출발점은 Day 2에서 만든 PRD입니다. PRD가 "무엇을·왜"라면, Plan은 "어떤 순서로", Design은 "어떻게"를 정합니다.

단계답하는 질문산출물
PRD (Day 2)무엇을·왜 만드는가PRD.md
Plan어떤 작업을, 어떤 순서로 할 것인가작업 목록·순서(PLAN.md)
Design그 작업들을 어떻게 구현할 것인가구조·데이터 흐름·화면(DESIGN.md)

Plan — PRD를 작업 순서로 풀기

PRD의 "핵심 기능"과 "개발 단위"를 만들 순서가 있는 작업 목록으로 바꿉니다. 무엇을 먼저 만들면 나머지가 수월해지는지 만들 순서를 잡는 것이 요령입니다. 이 과정의 PRD는 이미 MVP 규모라 범위를 더 자르지 않고 전체를 계획으로 옮깁니다.

Design — 작업을 구현으로 잇기

각 작업을 어떻게 만들지 정합니다. 화면 구성, 데이터가 흐르는 순서(문서 입력 → 질문과 함께 서버로 → LLM 답변), 사용할 기술의 큰 선택을 미리 그려 두면 Do(구현) 단계에서 헤매지 않습니다.

💡 핵심 한 줄: PRD(무엇/왜) → Plan(순서) → Design(어떻게) 순으로 좁혀 가면, 다음 날의 "만들기(Do)"가 훨씬 수월해집니다.

Day 2의 PRD.md를 바탕으로 PLAN.md·DESIGN.md를 만들고(실습 STEP 3), 세 문서의 정합성을 검토하는(STEP 4) 과정으로 이어집니다.
