1. GitHub 저장소란 무엇인가 + 웹 화면으로 새 저장소 만들기

Repository — 코드를 담아두는 온라인 보관함

GitHub 저장소(repository)는 지금까지 Claude Code로 작성한 파이썬 코드를 온라인에 저장해 두는 공간입니다. 내 컴퓨터에만 있는 코드는 컴퓨터를 꺼두면 실행할 수 없지만, GitHub에 올려두면 GitHub의 서버가 대신 코드를 보관하고 있다가 정해진 시간에 실행해 줄 수 있습니다. Day 4의 목표는 이 저장소를 만들고, 그 안에서 자동 실행 규칙(Actions)을 설정하는 것입니다.

New repository — 웹 화면으로 저장소 생성하기

터미널 명령어를 몰라도 괜찮습니다. 아래 순서는 모두 github.com 화면에서 마우스 클릭만으로 진행합니다.

  1. 로그인: github.com에 접속해 본인 계정으로 로그인합니다.
  2. 새 저장소 시작: 화면 우측 상단의 + 아이콘을 클릭하고, 드롭다운 메뉴에서 New repository를 선택합니다.
  3. 이름 입력: Repository name란에 저장소 이름을 입력합니다. (예: news-alert-bot)
  4. 설명 입력(선택): Description란에 이 저장소가 무엇인지 한 줄로 적어둘 수 있습니다. 생략해도 됩니다.
  5. 공개 범위 선택: Public 또는 Private 중 하나를 선택합니다.
  6. README 추가(선택): 필요하면 Add a README file 체크박스를 선택합니다.
  7. 생성 완료: Create repository 버튼을 클릭하면 저장소가 만들어지고, 코드를 올릴 준비가 끝난 저장소 화면으로 이동합니다.

Public vs Private — 공개 범위 선택 기준

뉴스 알림 봇처럼 개인 이름을 검색어로 다루는 코드는 저장소 자체를 Private로 두는 편이 안전합니다. 다만 Secrets(3절)로 등록한 값은 저장소가 Public이어도 외부에 노출되지 않으므로, 공개 여부와 무관하게 실제 비밀값은 반드시 Secrets에만 저장해야 합니다.

구분공개 범위권장 상황
Public 누구나 코드를 볼 수 있음 공개해도 무방한 예제·학습용 코드
Private 본인과 초대한 사람만 볼 수 있음 개인 이름·검색어가 포함된 실제 운영 코드 (권장)

💡 핵심 한 줄: 저장소를 Private로 두더라도, 텔레그램 토큰이나 API 키 같은 실제 비밀값은 코드 안에 직접 적지 않고 반드시 Secrets에 등록합니다.

2. Secrets(비밀값) 등록하기

Settings → Secrets and variables → Actions — 비밀값 저장 위치

Secrets는 텔레그램 봇 토큰이나 네이버 API 키처럼 외부에 노출되면 안 되는 값을 GitHub 저장소 안에 암호화된 형태로 저장하는 기능입니다. 워크플로우 실행 시에는 값이 자동으로 주입되지만, 저장 이후에는 화면에서 값을 다시 확인할 수 없어 실수로 노출될 위험이 크게 줄어듭니다.

  1. 저장소 설정 진입: 저장소 화면 상단의 Settings 탭을 클릭합니다.
  2. Secrets 메뉴 이동: 좌측 메뉴에서 Secrets and variables → Actions를 클릭하고, Secrets 탭이 선택돼 있는지 확인합니다.
  3. 새 Secret 추가: New repository secret 버튼을 클릭합니다.
  4. 이름과 값 입력: Name란에 식별용 이름(예: BOT_TOKEN)을, Secret란에 실제 값을 입력합니다.
  5. 저장: Add secret 버튼을 클릭하면 등록이 완료됩니다.

Day 4에서는 아래 3개의 Secret을 이 순서대로 등록합니다.

Secret 이름값의 내용비고
BOT_TOKEN Day 2에서 BotFather로 발급받은 텔레그램 봇 토큰 이름만 공개되어도 안전 — 값은 절대 공개 금지
NAVER_CLIENT_ID Day 3에서 발급받은 네이버 검색 API Client ID Naver Developers 콘솔에서 확인
NAVER_CLIENT_SECRET Day 3에서 발급받은 네이버 검색 API Client Secret Naver Developers 콘솔에서 확인

${{ secrets.이름 }} — 워크플로우 안에서 참조하는 문법

등록한 Secret은 워크플로우 파일(.yml) 안에서 ${{ secrets.BOT_TOKEN }}처럼 secrets. 뒤에 등록한 이름을 붙여 참조합니다. 이렇게 참조된 값은 실행 중에만 환경 변수로 전달되며, 실행 로그 화면에도 실제 값 대신 ***로 가려져 출력됩니다.

env:
  BOT_TOKEN: ${{ secrets.BOT_TOKEN }}
  NAVER_CLIENT_ID: ${{ secrets.NAVER_CLIENT_ID }}
  NAVER_CLIENT_SECRET: ${{ secrets.NAVER_CLIENT_SECRET }}
Secret은 등록 후에는 값을 다시 볼 수 없고, 수정하려면 같은 이름으로 새 값을 덮어써야 합니다. 이름과 값을 입력할 때 오타가 없는지 저장 전에 한 번 더 확인하세요.

3. GitHub Actions 워크플로우 문법

.github/workflows/*.yml — 워크플로우 파일의 위치

GitHub Actions의 실행 규칙은 저장소 안의 .github/workflows/ 폴더에 있는 .yml 파일에 기록합니다. 이 폴더에 파일을 하나 추가하고 저장소에 커밋(commit)하면, GitHub가 자동으로 그 파일을 워크플로우로 인식합니다.

news-alert-bot/
├── check_and_send.py
├── seen_urls.txt
└── .github/
    └── workflows/
        └── news-check.yml

on.schedule.cron — 정해진 시간마다 자동 실행

cron 표현식은 "분 시 일 월 요일" 5개 필드로 실행 시각을 지정합니다. 여기서 중요한 점은 GitHub Actions의 cron은 기본적으로 UTC(협정 세계시) 기준으로 해석된다는 것입니다. 한국 시간(KST)은 UTC보다 9시간 빠르므로, 워크플로우 파일에는 한국시간을 UTC로 환산한 값을 적어야 합니다.

원하는 한국시간(KST)UTC 환산 값cron 표현식
오전 8시 전날 23시 0 23 * * *
오후 6시 같은 날 9시 0 9 * * * 
on:
  schedule:
    - cron: '0 23 * * *'   # 매일 오전 8시 KST (UTC 23:00)
    - cron: '0 9 * * *'    # 매일 오후 6시 KST (UTC 09:00)

workflow_dispatch — 수동 실행 버튼 추가하기

workflow_dispatch를 on 항목에 추가하면, 저장소의 Actions 탭에 "Run workflow" 버튼이 생겨 예약 시간을 기다리지 않고도 원할 때 직접 실행해볼 수 있습니다. 단, 이 버튼은 기본 브랜치(보통 main)에 올라가 있는 워크플로우 파일에서만 나타납니다.

on:
  workflow_dispatch:
예약 실행(cron)은 최소 5분 간격으로만 설정할 수 있고, GitHub 서버 부하 상황에 따라 실제 실행이 예정 시각보다 지연될 수 있습니다. 알림이 몇 분 늦게 와도 정상적인 동작이니 당황하지 않아도 됩니다.

4. 실행 확인과 로그 보기

Actions 탭 — 실행 목록과 상태 확인

저장소 상단의 Actions 탭을 클릭하면 지금까지 실행된(또는 예정된) 워크플로우 목록이 시간순으로 표시됩니다. 각 실행 항목 옆에는 성공(초록색 체크)과 실패(빨간색 X) 상태가 아이콘으로 표시되어, 굳이 내용을 열어보지 않아도 한눈에 성공 여부를 파악할 수 있습니다.

로그 보기 — 실패 원인 찾는 접근법

실행 항목을 클릭하면 각 단계(step)별로 걸린 시간과 출력 내용을 확인할 수 있습니다. 실패했다면 초록 체크 대신 빨간 X가 표시된 단계를 찾아 그 단계를 펼쳐보면, 어떤 이유로 멈췄는지(예: 패키지 설치 실패, 토큰 값 오류, API 응답 오류 등) 화면에 그대로 출력됩니다. 위에서부터 순서대로 읽어가며 가장 먼저 실패한 단계를 확인하는 것이 원인 파악의 기본 순서입니다.

  1. 어떤 실행인지 확인: Actions 탭에서 문제가 생긴 실행 항목을 클릭합니다.
  2. 실패한 단계 찾기: 단계 목록에서 빨간 X 표시가 붙은 단계를 찾습니다.
  3. 로그 펼쳐 읽기: 해당 단계를 클릭해 펼치고, 에러 메시지를 위에서부터 읽습니다.
  4. 원인에 맞춰 조치: Secret 값 오타, 코드 오류, API 키 만료 등 원인에 따라 해당 부분을 수정하고 다시 실행합니다.

💡 핵심 한 줄: 워크플로우가 실패하면 당황하지 말고 Actions 탭 → 실패한 실행 → 빨간 X가 붙은 단계 순서로 로그를 열어보면 대부분 원인을 바로 확인할 수 있습니다.
