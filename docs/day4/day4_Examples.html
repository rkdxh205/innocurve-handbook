Day 4 · bkit PDCA 완성 — 구현·검증·개선과 배포·발표 (실습)

Day 3에서 만든 PLAN.md·DESIGN.md를 실제로 굴려 앱을 완성합니다. Do → Check → Act 순으로 진행한 뒤, 배포·발표·수료로 마무리합니다.
개념이 헷갈리면 각 단계의 "관련 이론" 링크로 이론 문서를 참고하세요.

오늘의 결과물동작하는 내 앱 · 배포 주소 · 발표·시연
Day 4는 Day 3에서 작업하던 세션을 그대로 이어서 진행합니다. 이미 CLAUDE.md(규칙)·PLAN.md·DESIGN.md가 준비돼 있다고 가정합니다. 앱이 외부 API(예: OpenAI)를 쓰면 API Key가 필요하며, 키는 코드에 직접 쓰지 말고 .env 같은 파일로 관리하고 커밋하지 마세요.
● Do 구현 Plan·Design대로 실제로 만듭니다
STEP 1 Do PLAN·DESIGN대로 하나씩 구현하기
Day 3에서 작업하던 세션을 그대로 이어서 진행합니다 — 새 세션을 열거나 작업 폴더를 다시 지정할 필요가 없습니다. Day 3의 PLAN.md·DESIGN.md를 입력으로 내 앱을 구현합니다. 한 번에 다 만들지 말고 작업을 하나씩 시켜 그때그때 실행해 확인합니다. 먼저 핵심 흐름 하나를 끝까지 돌려 놓고 나머지를 얹으면 안전합니다.
관련 이론
Do — 실제 기능 구현
단계별 실행
  1. 계획 다시 확인하기

    구현을 시작하기 전에, Claude에게 PLAN.md·DESIGN.md를 읽고 가장 먼저 구현할 작업 하나가 무엇인지 짚게 합니다.

    프롬프트 
    bkit의 pdca 스킬을 사용해서,
PLAN.md와 DESIGN.md를 읽고
가장 먼저 구현할 작업 하나가 무엇인지 알려줘.
- 아직 코드는 만들지 않는다
- 그 작업만 한 문장으로 정리해줘
  2. 첫 작업 구현하기

    확인한 작업 하나를 구현하게 합니다. 큰 덩어리를 통째로 맡기지 말고, 한 조각씩 만들어 바로 실행해 봅니다.

    프롬프트 
    bkit의 pdca 스킬을 사용해서,
방금 정한 첫 작업을 DESIGN.md의 방식대로 구현해줘.
- 지금은 이 작업 하나만 만든다
- 다 만들면 어떻게 실행해서 확인하는지 알려줘
- CLAUDE.md 규칙을 지켜줘
  3. 실행해 확인하고 다음 작업으로

    구현한 것을 바로 실행해 눈으로 확인합니다. 잘 되면 아래 프롬프트로 PLAN.md의 다음 작업을 시키고, 다시 실행해 확인합니다. 반복의 기준은 PLAN.md의 작업 목록입니다 — 목록의 작업이 모두 구현되면 반복을 멈추고, STEP 2(Check)로 넘어가 설계대로 됐는지 검증합니다.(앱이 외부 API를 쓰면 그 API Key가 필요합니다. 키는 .env로 관리하고 커밋하지 마세요.)

    프롬프트 (반복용) 
    bkit의 pdca 스킬을 사용해서,
PLAN.md에서 아직 안 만든 다음 작업 하나를
DESIGN.md의 방식대로 구현해줘.
- 이번에도 이 작업 하나만 만든다
- 다 만들면 어떻게 실행해서 확인하는지 알려줘
- CLAUDE.md 규칙을 지켜줘

    ▲ 이 프롬프트를 PLAN.md의 목록이 끝날 때까지 반복해 씁니다.

막히면 통째로 다시 만들라고 하기보다, "방금 바꾼 부분에서 무엇이 문제인지"를 물어보세요. 작게 나눠 왔기 때문에 원인을 좁히기 쉽습니다.
● Check 검증 계획·설계대로 됐는지 확인합니다
STEP 2 Check Gap 분석과 성공 기준으로 검증하기
만든 결과가 DESIGN.md의 설계대로인지 확인합니다. 설계와 구현의 차이(Gap)를 목록으로 뽑고, Day 3에서 정한 성공 기준 시나리오를 실제로 돌려 봅니다.
관련 이론
Check — Gap 분석·검증
단계별 실행
  1. Gap 분석 요청하기

    설계 문서와 실제 구현을 대조해 어긋난 곳·빠진 곳을 목록으로 뽑게 합니다. pdca 스킬을 부르면 bkit이 이 일을 gap-detector(설계·구현 비교 전문가)에게 자동으로 맡깁니다.

    프롬프트 
    bkit의 pdca 스킬을 사용해서,
DESIGN.md에 적은 설계와 지금 실제 구현된 코드를 비교해
차이(Gap)를 목록으로 정리해줘.
- 빠진 기능 / 설계와 다르게 동작하는 부분 / 아직 처리 못 한 예외
- 각 항목을 한 줄로, 심각한 것부터 위에 적어줘
- 지금은 고치지 말고 목록만 보여줘
  2. 성공 기준 시나리오 돌리기 — Claude가 판정

    Day 3에서 정한 성공 기준을 시험지로 삼아, 그 시나리오를 실제로 실행하고 통과 여부를 Claude에게 판정하게 합니다.

    예시 — 성공 기준 시나리오 (앱마다 다름)
    성공 기준은 앱마다 다릅니다. Day 3에서 내가 정한 성공 기준을 그대로 쓰세요.
    예(문서 Q&A 봇): 샘플 문서 1개를 올리고 질문 3개를 던져, 셋 다 문서 근거와 함께 답하면 통과.
    프롬프트 
    이 프롬프트는 Day 3에서 정한 성공 기준을
지금 앱이 통과하는지 확인하기 위한 프롬프트야.

PLAN.md에 적힌 "이번 사이클 성공 기준"을 읽고,
지금 앱으로 그 시나리오를 실제로 실행해봐.
- 각 기준을 '통과/실패'로 판정해줘
- 실패한 건 왜 실패했는지 한 줄로 알려줘
    판정 기준 — 무엇이 되면 통과인가: 성공 기준을 하나씩 '통과/실패'로 봅니다. ① 모두 '통과' + ② 설계와 어긋난 치명적 Gap이 없으면 → 이번 Check 통과(다음 단계로). 하나라도 '실패'거나 치명적 Gap이 있으면 → STEP 3(Act)에서 고친 뒤 이 Check를 다시 돌립니다. 판정은 Claude가 1차로 하고, 최종 확인은 본인이 합니다. (작은 Gap은 통과를 막지 않되 CHECK.md에 남겨 Act에서 다듬습니다.)
  3. 보안 관점도 함께 점검하기 — "안전한가?"

    Gap 분석이 "계획대로인가"를 본다면, 여기서는 "안전한가"를 봅니다. AI가 만든 코드에는 본인도 모르는 보안 구멍이 있을 수 있어, 공개 배포 전에 한 번 훑어보게 합니다.

    프롬프트 
    bkit의 security-architect(보안 전문가)를 활용해서,
이 앱을 인터넷에 공개하기 전에 보안 관점으로 점검해줘.
- 비밀 키가 코드나 화면(브라우저)에 노출되는 곳은 없는지
- 사용자가 입력한 값을 그대로 믿고 쓰는 곳은 없는지
  (이상한 입력·스크립트가 끼어들 위험)
- (챗봇이면) "앞의 지시 무시하고…" 같은 프롬프트 인젝션에 대한 대비가 있는지
- 개인정보를 입력받거나 저장하는 부분이 있는지
발견한 위험을 심각한 것부터 목록으로 알려주고,
간단히 고칠 수 있는 건 어떻게 고칠지도 제안해줘.

    나온 항목은 CHECK.md에 함께 적어 두고, 심각한 것은 STEP 3(Act)에서 고칩니다. 실습 범위를 넘는 항목은 "실제 서비스로 만들 때 챙길 것"으로 메모만 해 둬도 됩니다.

  4. 검증 결과 기록하기 (CHECK.md 최신화)

    통과/실패와 Gap 목록을 CHECK.md로 남깁니다. Check는 개선하며 여러 번 돌리므로, 매번 새 파일을 만들지 말고 같은 CHECK.md를 최신 결과로 갱신(덮어쓰기)합니다.

    프롬프트 
    방금 나온 Gap 목록·성공 기준 판정·보안 점검 결과를
CHECK.md로 정리해줘.
- CHECK.md가 이미 있으면 새로 만들지 말고 그 파일을 최신 내용으로 덮어써줘
- 맨 위에 "통과 / 실패"를 적고
- 아래에 고쳐야 할 항목을 우선순위 순으로 정리해줘
Check의 목적은 "돌아가나?"가 아니라 "계획대로 돌아가나?"입니다. 동작하더라도 설계와 다르면 그것도 Gap입니다 — 반드시 목록으로 남기세요.
● Act 개선 고치고, 배운 것을 규칙으로 남깁니다
STEP 3 Act Gap 고치고 개선·고도화하기
Check에서 뽑은 CHECK.md의 Gap을 하나씩 고치고, 고칠 때마다 다시 검증합니다. 기본이 안정되면 우선순위 높은 기능 하나를 다듬고, 배운 규칙을 CLAUDE.md에 남깁니다.
관련 이론
Act — 개선·고도화
단계별 실행
  1. Gap 하나 고치기 → 다시 검증

    CHECK.md의 가장 위(심각한) 항목 하나부터 고치게 합니다. 한 번에 하나만 고치고, 고친 뒤 바로 다시 확인합니다. pdca 스킬을 부르면 bkit이 이 일을 pdca-iterator(반복 개선 전문가)에게 자동으로 맡깁니다.

    프롬프트 
    bkit의 pdca 스킬을 사용해서,
CHECK.md에서 가장 위에 있는 항목 하나만 고쳐줘.
- 이 항목 하나만 수정한다
- 고친 뒤 어떻게 다시 확인하는지 알려줘
- 무엇을 왜 바꿨는지 한 줄로 요약해줘

    ▲ 통과하면 다음 항목으로. CHECK.md의 항목이 빌 때까지 반복합니다.

  2. 기능 하나 고도화하기

    Gap이 정리되면, 여유가 있는 만큼 우선순위 높은 기능 하나를 다듬습니다. 욕심내 한 번에 많이 바꾸지 말고 작게 더합니다.

    프롬프트 (예시) 
    지금 앱에서 사용자 경험을 가장 크게 높일
작은 개선 한 가지를 제안하고, 그 하나만 구현해줘.
- 예: 잘못된 입력 안내 메시지, 로딩 표시, 결과를 보기 좋게 정리
- 이번에도 이 개선 하나만 만든다
- CLAUDE.md 규칙을 지켜줘
  3. 배운 것을 규칙으로 남기기

    이번 바퀴에서 반복된 실수·기준을 CLAUDE.md에 규칙으로 적어, 다음 PDCA 바퀴가 더 수월하게 합니다.

    프롬프트 
    이 프롬프트는 오늘 배운 것을 다음 작업에서도
지키게 만들기 위한 프롬프트야.

오늘 구현·검증하면서 반복된 실수나 새로 정한 기준이 있으면,
CLAUDE.md에 규칙으로 한두 줄 추가해줘.
남은 Gap이나 더 하고 싶은 기능은 다음 PDCA 한 바퀴로 넘기세요. Act의 목표는 완벽한 완성이 아니라 "고치고, 배운 것을 규칙으로 남기는" 것입니다.
여기까지: Plan·Design(Day 3) → Do·Check·Act(Day 4)로 첫 PDCA 한 바퀴를 완성했습니다. 이제 결과물을 배포하고 발표·수료로 마무리합니다.
● 마무리 배포 · 발표 · 수료 결과물을 배포하고 발표·수료로 마무리합니다
STEP 4 배포 GitHub에 올리고 Vercel로 배포하기
앱이 완성됐으니 인터넷 주소를 만들 차례입니다. 아래 프롬프트 하나로 Claude가 ① 프로젝트를 GitHub 저장소에 올리고(.env 제외) ② Vercel에 연결해 배포합니다. Day 2 실습에서 이미 한 번 해 본 과정입니다.
관련 이론 · 참고
배포 및 발표 준비 Day 2 실습 · 로그인(인증) 안내
단계별 실행
  1. 배포 요청하기

    프로젝트 전체를 올리되, 비밀 정보 파일만 빼는 것이 핵심입니다. 그래서 프롬프트에 .gitignore 확인 조항을 넣습니다.

    프롬프트 
    최종 산출물을 인터넷에 배포할 거야.
이 프로젝트를 GitHub 저장소에 올리고, Vercel로 배포해줘.
- .env 등 비밀 정보 파일이 .gitignore에 등록돼 있는지
  먼저 확인하고, 절대 올리지 않는다
- 배포가 끝나면 접속 주소를 알려줘

    ▲ 중간에 GitHub·Vercel 로그인(인증) 안내가 나오면 Day 2 실습의 로그인 안내를 그대로 따라 하면 됩니다.

  2. 접속 주소 확인하기

    Claude가 알려 준 https://…vercel.app 주소를 브라우저에서 열어 앱이 동작하는지 확인합니다. 스마트폰에서도 열어 보세요 — 발표 때 그대로 시연할 수 있습니다.

앱이 .env의 API Key를 쓴다면, 프롬프트에 - .env의 키들은 Vercel 환경 변수로 등록해줘 한 줄을 더하세요. 키를 GitHub에 올리지 않고도 배포된 앱이 쓸 수 있게 됩니다.
.env(API Key 등)는 GitHub에 절대 올라가면 안 됩니다. 한 번 올라간 키는 지워도 기록에 남기 때문에, 실수로 올렸다면 삭제가 아니라 재발급이 필요합니다.
STEP 5 진행 시간 결과물 발표 및 시연

각자 만든 앱을 배포된 주소로 직접 시연하며 무엇을·왜·어떻게 만들었는지 짧게 발표하는 시간입니다.

발표 준비 — 발표용 PPT 만들기 (선택)
발표 자료가 필요하면, my-app 폴더에서 아래 프롬프트를 실행하세요. Claude Code가 내 프로젝트 문서(PRD 등)를 읽고 발표용 PPT(.pptx)를 만들어 검수까지 해 줍니다 — 문서에 없는 내용은 지어내지 않습니다.

프롬프트 
이 프로젝트 폴더의 문서를 읽고 프로젝트 발표용 PPT(.pptx)를 만들어줘.

【0단계 — 문서 자동 수집】
아래 순서로 폴더에서 문서를 찾아 읽고 내용을 파악해:
1. PRD.md / prd_lite.md / 기획서·요구사항 문서 → 발표 내용의 근거 (필수)
2. CLAUDE.md → 디자인 가이드·프로젝트 규칙이 있으면 반영
3. README.md / DESIGN.md / PLAN.md → 보조 자료로 참고
- PRD류 문서가 하나도 없으면 만들지 말고, 어떤 문서를 근거로 할지 나에게 물어봐.
- 문서에 없는 내용(수치·기능·주장)을 지어내서 슬라이드에 넣지 마.

【1단계 — 확인 질문】
문서를 읽은 뒤, 아래 중 문서만으로 판단이 안 되는 것만 나에게 물어봐:
- 용도 (기본값: 프로젝트 소개 발표, 5~10분)
- 청중 (기본값: 개발을 잘 모르는 일반 청중도 이해 가능하게)
- 분량 (기본값: 10장 내외)
문서와 기본값으로 충분하면 질문 없이 바로 진행해.

【2단계 — 구성】
기본 흐름 (프로젝트 성격에 맞게 조정 가능):
표지 → 문제 정의 → 기존 방식의 한계 → 해결책(이용 흐름) →
핵심 기능 소개(기능당 1장) → 신뢰/보안 원칙 → 성공 기준 → 기술 스택 → 마무리
- 성공 기준·목표 수치는 큰 숫자 카드로 시각화 (예: "80%", "3분")
- PRD에 해당 섹션이 없으면 그 슬라이드는 빼고, 있는 내용으로 재구성해.

【3단계 — 디자인】
- CLAUDE.md에 디자인 가이드(색상·분위기)가 있으면 그대로 따라줘.
- 없으면: 프로젝트 주제에 어울리는 색 2~3개를 골라 제안하고 진행해.
- 공통 규칙:
  · 글머리표 나열 금지 — 아이콘·카드·흐름도·비교표 등 시각 요소 적극 활용
  · 표지와 마무리는 진한 배경으로 임팩트 있게, 본문은 밝은 배경
  · 한글 폰트는 맑은 고딕
  · 모든 슬라이드에 시각 요소 1개 이상

【4단계 — 품질 검수 (필수)】
- 완성 후 파일 구조를 검증하고, 슬라이드 전체를 이미지로 렌더링해서
  글자 잘림·요소 겹침·여백 문제를 직접 눈으로 검수해.
- 문제가 있으면 고치고 다시 렌더링해서 확인한 뒤 완성본을 줘.

【5단계 — 저장】
- 파일명: {프로젝트명}_소개.pptx (프로젝트명은 문서에서 파악한 한글 이름)
- 저장 위치: git 저장소 밖 (커밋에 섞이지 않게). 저장소가 아니면 프로젝트 폴더에.
- 완성 후: 슬라이드 구성 요약표 + 저장 경로를 알려줘.

완성된 .pptx는 커밋에 섞이지 않게 git 저장소 밖에 저장되며, 정확한 경로는 완성 후 Claude가 알려줍니다. 슬라이드 문구·색이 마음에 안 들면 이어서 "3번 슬라이드 제목을 ○○로 바꿔줘"처럼 말로 다듬으면 됩니다.
