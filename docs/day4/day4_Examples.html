Day 4 · bkit PDCA 완성 — 구현·검증·개선과 배포·발표 (실습)

Day 3에서 만든 PLAN.md·DESIGN.md를 실제로 굴려 앱을 완성합니다. Do → Check → Act 순으로 진행한 뒤, 배포·발표·수료로 마무리합니다.
개념이 헷갈리면 각 단계의 "관련 이론" 링크로 이론 문서를 참고하세요.

진행 순서STEP 1 Do → STEP 2 Check → STEP 3 Act → STEP 4~6 배포·발표·수료
사용 도구Claude Desktop의 Code 탭 · (외부 API를 쓰면) API Key · Vercel
오늘의 결과물동작하는 내 앱 · 인터넷에 배포된 주소 · 발표·시연
Day 4는 Day 3의 작업 폴더를 이어서 씁니다. 이미 CLAUDE.md(규칙)·PLAN.md·DESIGN.md가 준비돼 있다고 가정합니다. 앱이 외부 API(예: OpenAI)를 쓰면 API Key가 필요하며, 키는 코드에 직접 쓰지 말고 .env 같은 파일로 관리하고 커밋하지 마세요.
● Do 구현 Plan·Design대로 실제로 만듭니다
STEP 1 Do PLAN·DESIGN대로 하나씩 구현하기
Day 3의 PLAN.md·DESIGN.md를 입력으로 내 앱을 구현합니다. PRD가 이미 MVP 규모이니 그 전체를 만들되, 한 번에 다 만들지 말고 작업을 하나씩 시켜 그때그때 실행해 확인합니다. 먼저 핵심 흐름 하나를 끝까지 돌려 놓고 나머지를 얹으면 안전합니다.
구현 규약 — 서버·비밀 키를 쓰는 앱만 (정적 앱은 건너뛰기)
  • 서버리스 함수는 api/ 폴더에 둔다 (Vercel이 자동으로 인식)
  • 비밀 키는 코드에 두지 말고 환경변수(예: OPENAI_API_KEY)로만 읽고, 없으면 안내 메시지를 돌려준다
  • 화면(정적 HTML)은 그 함수를 /api/… 경로로 호출한다
내 앱이 서버에서 외부 API·비밀 키를 쓴다면 이 규약을 지켜야 Day 4 STEP 4 배포가 매끄럽습니다. 정적 화면만이라면 이 규약은 건너뛰어도 됩니다.

왜 api/ 함수가 필요할까요? 비밀 키를 화면(브라우저)에 두면 누구나 열어 볼 수 있어 위험합니다. 그래서 키를 감춰 줄 중간 서버(api/ 함수)를 두고, 브라우저는 그 함수만 부릅니다 — 키는 함수 안에서만 씁니다. 정리하면 .env·환경변수 = 키 보관, api/ 함수 = 그 키로 대신 일하는 서버라 대체재가 아니라 함께 씁니다.

관련 이론
Do — 실제 기능 구현
단계별 실행
  1. Code 탭에서 새 세션 시작 (작업 폴더 지정)

    Claude Desktop 상단 Code 탭에서 + New session(Ctrl+N)으로 새 세션을 열고, Environment: Local · Project folder: Day 3의 작업 폴더를 지정합니다. Day 3의 CLAUDE.md·PLAN.md·DESIGN.md가 그대로 보이는지 확인합니다.

  2. 계획 다시 확인하기

    구현을 시작하기 전에, Claude에게 PLAN.md·DESIGN.md를 읽고 가장 먼저 구현할 작업 하나가 무엇인지 짚게 합니다.

    프롬프트 
    PLAN.md와 DESIGN.md를 읽고,
가장 먼저 구현할 작업 하나가 무엇인지 알려줘.
아직 코드는 만들지 말고, 그 작업만 한 문장으로 정리해줘.
  3. 첫 작업 구현하기

    확인한 작업 하나를 구현하게 합니다. 큰 덩어리를 통째로 맡기지 말고, 한 조각씩 만들어 바로 실행해 봅니다.

    프롬프트 
    방금 정한 첫 작업을 DESIGN.md의 방식대로 구현해줘.
- 지금은 이 작업 하나만 만든다
- 다 만들면 어떻게 실행해서 확인하는지 알려줘
- CLAUDE.md 규칙을 지켜줘
  4. 실행해 확인하고 다음 작업으로

    구현한 것을 바로 실행해 눈으로 확인합니다(방법은 아래 팁 참고). 잘 되면 PLAN.md의 다음 작업으로 넘어가 3~4번을 반복하며 앱을 완성합니다.(앱이 외부 API를 쓰면 그 API Key가 필요합니다. 키는 .env로 관리하고 커밋하지 마세요.)

결과를 눈으로 확인하는 법 (Code 탭)
• 가장 쉬운 법 — 채팅에 뜬 index.html 같은 파일 경로를 클릭하면 오른쪽 브라우저 창에 바로 열립니다(단축키 Ctrl+Shift+B). 정적 화면은 서버 없이 이렇게 바로 보입니다.
• 서버(api/ 함수)가 있는 앱 — Claude에게 "로컬에서 띄워서 확인하게 해줘"라고 하면 개발 서버(vercel dev 등)를 실행해 localhost:3000을 브라우저 창에 자동으로 엽니다.
• 직접 명령을 치고 싶으면 — 내장 터미널을 Ctrl+`(컨트롤+백틱)로 열어 실행합니다.
막히면 통째로 다시 만들라고 하기보다, "방금 바꾼 부분에서 무엇이 문제인지"를 물어보세요. 작게 나눠 왔기 때문에 원인을 좁히기 쉽습니다.
● Check 검증 계획·설계대로 됐는지 확인합니다
STEP 2 Check Gap 분석과 성공 기준으로 검증하기
만든 결과가 DESIGN.md의 설계대로인지 확인합니다. 설계와 구현의 차이(Gap)를 목록으로 뽑고, Day 3에서 정한 성공 기준 시나리오를 실제로 돌려 봅니다.
관련 이론
Check — Gap 분석·검증
단계별 실행
  1. Gap 분석 요청하기

    설계 문서와 실제 구현을 대조해 어긋난 곳·빠진 곳을 목록으로 뽑게 합니다.

    프롬프트 
    DESIGN.md에 적은 설계와 지금 실제 구현된 코드를 비교해서,
차이(Gap)를 목록으로 정리해줘.
- 빠진 기능 / 설계와 다르게 동작하는 부분 / 아직 처리 못 한 예외
- 각 항목을 한 줄로, 심각한 것부터 위에 적어줘
지금은 고치지 말고 목록만 보여줘.
  2. 성공 기준 시나리오 돌리기 — Claude가 판정

    Day 3에서 정한 성공 기준을 시험지로 삼아, 그 시나리오를 실제로 실행하고 통과 여부를 Claude에게 판정하게 합니다.

    예시 — 성공 기준 시나리오 (앱마다 다름)
    성공 기준은 앱마다 다릅니다. Day 3에서 내가 정한 성공 기준을 그대로 쓰세요.
    예(문서 Q&A 봇): 샘플 문서 1개를 올리고 질문 3개를 던져, 셋 다 문서 근거와 함께 답하면 통과.
    프롬프트 
    PLAN.md에 적힌 "이번 사이클 성공 기준"을 읽고,
지금 앱으로 그 시나리오를 실제로 실행해봐.
각 기준을 통과했는지 '통과/실패'로 판정하고,
실패한 건 왜 실패했는지 한 줄로 알려줘.
    판정 기준 — 무엇이 되면 통과인가: 성공 기준을 하나씩 '통과/실패'로 봅니다. ① 모두 '통과' + ② 설계와 어긋난 치명적 Gap이 없으면 → 이번 Check 통과(다음 단계로). 하나라도 '실패'거나 치명적 Gap이 있으면 → STEP 3(Act)에서 고친 뒤 이 Check를 다시 돌립니다. 판정은 Claude가 1차로 하고, 최종 확인은 본인이 합니다. (작은 Gap은 통과를 막지 않되 CHECK.md에 남겨 Act에서 다듬습니다.)
  3. 검증 결과 기록하기 (CHECK.md 최신화)

    통과/실패와 Gap 목록을 CHECK.md로 남깁니다. Check는 개선하며 여러 번 돌리므로, 매번 새 파일을 만들지 말고 같은 CHECK.md를 최신 결과로 갱신(덮어쓰기)합니다.

    프롬프트 
    방금 Gap 목록과 성공 기준 판정 결과를 CHECK.md로 정리해줘.
- CHECK.md가 이미 있으면 새로 만들지 말고 그 파일을 최신 내용으로 덮어써줘
- 맨 위에 "통과 / 실패"를 적고
- 아래에 고쳐야 할 항목을 우선순위 순으로 정리해줘
Check의 목적은 "돌아가나?"가 아니라 "계획대로 돌아가나?"입니다. 동작하더라도 설계와 다르면 그것도 Gap입니다 — 반드시 목록으로 남기세요.
● Act 개선 고치고, 배운 것을 규칙으로 남깁니다
STEP 3 Act Gap 고치고 개선·고도화하기
Check에서 뽑은 CHECK.md의 Gap을 하나씩 고치고, 고칠 때마다 다시 검증합니다. 기본이 안정되면 우선순위 높은 기능 하나를 다듬고, 배운 규칙을 CLAUDE.md에 남깁니다.
관련 이론
Act — 개선·고도화
단계별 실행
  1. Gap 하나 고치기 → 다시 검증

    CHECK.md의 가장 위(심각한) 항목 하나부터 고치게 합니다. 한 번에 하나만 고치고, 고친 뒤 바로 다시 확인합니다.

    프롬프트 
    CHECK.md에서 가장 위에 있는 항목 하나만 고쳐줘.
- 이 항목 하나만 수정한다
- 고친 뒤 어떻게 다시 확인하는지 알려줘
- 무엇을 왜 바꿨는지 한 줄로 요약해줘

    ▲ 통과하면 다음 항목으로. CHECK.md의 항목이 빌 때까지 반복합니다.

  2. 기능 하나 고도화하기

    Gap이 정리되면, 여유가 있는 만큼 우선순위 높은 기능 하나를 다듬습니다. 욕심내 한 번에 많이 바꾸지 말고 작게 더합니다.

    프롬프트 (예시) 
    지금 앱에서 사용자 경험을 가장 크게 높일 작은 개선
한 가지를 제안하고, 그 하나만 구현해줘.
(예: 잘못된 입력 안내 메시지, 로딩 표시, 결과를 보기 좋게 정리 등)
  3. 배운 것을 규칙으로 남기기

    이번 바퀴에서 반복된 실수·기준을 CLAUDE.md에 규칙으로 적어, 다음 PDCA 바퀴가 더 수월하게 합니다.

    프롬프트 
    오늘 구현·검증하면서 반복된 실수나 새로 정한 기준이 있으면,
CLAUDE.md에 규칙으로 한두 줄 추가해줘.
남은 Gap이나 더 하고 싶은 기능은 다음 PDCA 한 바퀴로 넘기세요. Act의 목표는 완벽한 완성이 아니라 "고치고, 배운 것을 규칙으로 남기는" 것입니다.
여기까지: Plan·Design(Day 3) → Do·Check·Act(Day 4)로 첫 PDCA 한 바퀴를 완성했습니다. 이제 결과물을 배포하고 발표·수료로 마무리합니다.
● 마무리 배포 · 발표 · 수료 결과물을 인터넷에 배포하고, 발표·수료로 마무리합니다
STEP 4 배포 Vercel에 자동 배포하기 (Claude가 실행)
완성한 내 앱을 Claude Code가 Vercel에 직접 배포합니다. 사람은 최초 1회 vercel login(브라우저 승인)만 하고, 실제 배포 명령(vercel --prod)은 Claude가 대신 실행합니다 — 터미널을 직접 열거나 웹사이트에 들어갈 필요가 없습니다. 이후 수정할 때마다 "다시 배포해줘" 한마디면 됩니다.
관련 이론
배포 및 발표 준비
단계별 실행
  1. ① 배포 준비 점검 (Claude Code에게)

    배포 전에, 그대로 올라갈 구조인지 Claude Code에게 확인·정리시킵니다.

    프롬프트 
    이 프로젝트를 Vercel에 배포하려고 해. 아래를 점검·정리해줘.
- (서버리스 함수를 쓰면) api 폴더에 있고, 화면에서 /api/… 로 호출하는지
- 비밀 키를 코드에 직접 두지 않고 환경변수로만 읽는지
- .env 같은 비밀 파일이 코드에 섞여 올라가지 않는지
- 정적 파일만이라면 그대로 배포 가능한지
고쳤으면 무엇을 왜 바꿨는지 한 줄로 알려줘.
  2. ② Vercel 로그인 (최초 1회 · 브라우저 승인만)

    터미널을 직접 열 필요 없이, 아래처럼 Claude에게 시키면 Vercel 도구 설치·로그인을 Claude가 대신 실행합니다. 브라우저 창이 뜨면 본인이 승인하면 끝 — 이 승인이 사람이 하는 유일한 단계입니다.

    프롬프트 
    Vercel에 배포할 수 있게 준비해줘.
vercel 도구를 설치하고 로그인을 시작해줘.
(로그인 창이 뜨면 내가 승인할게.)

    ▲ Claude가 대신 실행하는 명령: npm install -g vercel → vercel login. 직접 보고 싶으면 내장 터미널(Ctrl+`)에서 같은 명령을 쳐도 됩니다.

  3. ③ Claude가 배포하기 (vercel --prod)

    로그인만 되면, 아래처럼 시켜 Claude가 직접 배포하게 합니다. 프로젝트 설정 질문은 기본값으로 넘어가고, 끝나면 https://…vercel.app 공개 주소가 나옵니다.

    프롬프트 
    이 프로젝트를 vercel --prod 로 배포해줘.
설정 질문은 기본값으로 진행하고,
완성된 공개 주소(https://...vercel.app)를 알려줘.
  4. ④ 비밀 키 등록 후 다시 배포 (키를 쓰는 앱만)

    앱이 외부 API(예: OpenAI)를 부른다면, 그 키를 Vercel 환경변수로 등록한 뒤 다시 배포해야 반영됩니다. 아래처럼 시키면 Claude가 등록·재배포하고, 키 값만 본인이 붙여넣습니다.

    프롬프트 
    OPENAI_API_KEY를 Vercel 환경변수로 등록해줘(값은 내가 입력할게).
적용 범위는 Production으로 하고, 등록되면 vercel --prod 로 다시 배포해줘.

    ▲ 정적 화면만인 앱은 이 단계를 건너뜁니다. 키 이름은 내 앱이 쓰는 API에 맞게 바꿔도 됩니다.

  5. ⑤ 이후 수정할 때마다 — "다시 배포해줘"

    코드를 고친 뒤에는 Claude에게 "다시 배포해줘"라고만 하면 vercel --prod를 다시 실행해 같은 주소에 갱신합니다. 발표 땐 이 공개 주소로 시연합니다.

비밀 키는 반드시 Vercel 환경변수로만. 코드나 정적 HTML(브라우저)에 키를 직접 넣으면 배포와 동시에 누구나 볼 수 있습니다. 키는 Vercel 환경변수에 두고 서버리스 함수가 그 값을 읽게 하며, .env 파일은 공유·커밋하지 않습니다.
배포 후 (키가 필요한) 기능이 안 되면 대개 환경변수를 등록한 뒤 다시 배포(vercel --prod)를 안 한 경우입니다. 키를 추가하면 반드시 다시 배포해야 반영됩니다.
완전 무인으로 하려면 vercel.com에서 토큰을 1회 발급해 Local 환경변수 VERCEL_TOKEN에 저장하면, 이후 로그인 없이 Claude가 배포합니다.
