Day 4 · bkit PDCA 완성 — 구현·검증·개선과 배포·발표 (실습)

Day 3에서 만든 PLAN.md·DESIGN.md를 실제로 굴려 앱을 완성합니다. Do → Check → Act 순으로 진행한 뒤, 배포·발표·수료로 마무리합니다.
개념이 헷갈리면 각 단계의 "관련 이론" 링크로 이론 문서를 참고하세요.

진행 순서STEP 1 Do → STEP 2 Check → STEP 3 Act → STEP 4~6 배포·발표·수료
사용 도구Claude Code(CLI) · OpenAI API Key(Day 1 발급) · Vercel
오늘의 결과물동작하는 doc-qa-bot · 인터넷에 배포된 주소 · 발표·시연
Day 4는 Day 3의 doc-qa-bot 폴더를 이어서 씁니다. 이미 CLAUDE.md(규칙)·PLAN.md·DESIGN.md가 준비돼 있다고 가정합니다. 답변 생성에는 OpenAI API Key가 필요하며, 키는 코드에 직접 쓰지 말고 .env 같은 파일로 관리하고 커밋하지 마세요.
● Do 구현 Plan·Design대로 실제로 만듭니다
STEP 1 Do PLAN·DESIGN대로 MVP 구현하기
Day 3의 PLAN.md·DESIGN.md를 입력으로, 문서 Q&A(RAG) 챗봇의 가장 작은 동작 버전(MVP)부터 만듭니다. 한 번에 다 만들지 말고, 작업을 하나씩 시켜 그때그때 실행해 확인합니다.
관련 이론
Do — 실제 기능 구현
단계별 실행
  1. 작업 폴더에서 Claude Code 실행

    터미널에서 doc-qa-bot 폴더로 이동한 뒤 claude를 실행합니다.

    cd doc-qa-bot
claude
  2. 계획 다시 확인하기

    구현을 시작하기 전에, Claude에게 PLAN.md·DESIGN.md를 읽고 이번에 만들 MVP의 첫 작업 하나가 무엇인지 짚게 합니다.

    프롬프트 
    PLAN.md와 DESIGN.md를 읽고, MVP를 만들기 위해
가장 먼저 구현할 작업 하나가 무엇인지 알려줘.
아직 코드는 만들지 말고, 그 작업만 한 문장으로 정리해줘.
  3. 첫 작업 구현하기

    확인한 작업 하나를 구현하게 합니다. 큰 덩어리를 통째로 맡기지 말고, 한 조각씩 만들어 바로 실행해 봅니다.

    프롬프트 
    방금 정한 첫 작업을 DESIGN.md의 방식대로 구현해줘.
- 지금은 이 작업 하나만 만든다
- 다 만들면 어떻게 실행해서 확인하는지 알려줘
- CLAUDE.md 규칙을 지켜줘
  4. 실행해 확인하고 다음 작업으로

    안내받은 방법으로 직접 실행해 동작을 확인합니다. 잘 되면 PLAN.md의 다음 작업으로 넘어가 3~4번을 반복하며 MVP를 완성합니다.(답변 생성 기능을 붙일 때 OpenAI API Key가 필요합니다. 키는 .env로 관리하고 커밋하지 마세요.)

막히면 통째로 다시 만들라고 하기보다, "방금 바꾼 부분에서 무엇이 문제인지"를 물어보세요. 작게 나눠 왔기 때문에 원인을 좁히기 쉽습니다.
● Check 검증 계획·설계대로 됐는지 확인합니다
STEP 2 Check Gap 분석과 성공 기준으로 검증하기
만든 결과가 DESIGN.md의 설계대로인지 확인합니다. 설계와 구현의 차이(Gap)를 목록으로 뽑고, Day 3에서 정한 성공 기준 시나리오를 실제로 돌려 봅니다.
관련 이론
Check — Gap 분석·검증
단계별 실행
  1. Gap 분석 요청하기

    설계 문서와 실제 구현을 대조해 어긋난 곳·빠진 곳을 목록으로 뽑게 합니다.

    프롬프트 
    DESIGN.md에 적은 설계와 지금 실제 구현된 코드를 비교해서,
차이(Gap)를 목록으로 정리해줘.
- 빠진 기능 / 설계와 다르게 동작하는 부분 / 아직 처리 못 한 예외
- 각 항목을 한 줄로, 심각한 것부터 위에 적어줘
지금은 고치지 말고 목록만 보여줘.
  2. 성공 기준 시나리오 돌리기

    Day 3 STEP 5에서 정한 성공 기준을 그대로 시험지로 삼아, 실제로 그 시나리오를 실행해 통과하는지 확인합니다.

    예시 — 성공 기준 시나리오
    시나리오: 샘플 문서 1개를 올리고, 그 문서로 답할 수 있는 질문 3개를 던진다.
    통과 조건: 세 질문 모두 문서 근거와 함께 답하면 통과.
  3. 검증 결과 기록하기

    통과/실패와 Gap 목록을 CHECK.md로 남겨, 다음 단계(Act)에서 무엇을 고칠지 분명히 합니다.

    프롬프트 
    방금 Gap 목록과 성공 기준 시나리오 결과를 CHECK.md로 정리해줘.
- 맨 위에 "통과 / 실패"를 적고
- 아래에 고쳐야 할 항목을 우선순위 순으로 정리해줘
Check의 목적은 "돌아가나?"가 아니라 "계획대로 돌아가나?"입니다. 동작하더라도 설계와 다르면 그것도 Gap입니다 — 반드시 목록으로 남기세요.
● Act 개선 고치고, 배운 것을 규칙으로 남깁니다
STEP 3 Act Gap 고치고 개선·고도화하기
Check에서 뽑은 CHECK.md의 Gap을 하나씩 고치고, 고칠 때마다 다시 검증합니다. 기본이 안정되면 우선순위 높은 기능 하나를 다듬고, 배운 규칙을 CLAUDE.md에 남깁니다.
관련 이론
Act — 개선·고도화
단계별 실행
  1. Gap 하나 고치기 → 다시 검증

    CHECK.md의 가장 위(심각한) 항목 하나부터 고치게 합니다. 한 번에 하나만 고치고, 고친 뒤 바로 다시 확인합니다.

    프롬프트 
    CHECK.md에서 가장 위에 있는 항목 하나만 고쳐줘.
- 이 항목 하나만 수정한다
- 고친 뒤 어떻게 다시 확인하는지 알려줘
- 무엇을 왜 바꿨는지 한 줄로 요약해줘

    ▲ 통과하면 다음 항목으로. CHECK.md의 항목이 빌 때까지 반복합니다.

  2. 기능 하나 고도화하기

    Gap이 정리되면, 여유가 있는 만큼 우선순위 높은 기능 하나를 다듬습니다. 욕심내 한 번에 많이 바꾸지 말고 작게 더합니다.

    프롬프트 (예시) 
    지금 앱에서 사용자 경험을 가장 크게 높일 작은 개선
한 가지를 제안하고, 그 하나만 구현해줘.
(예: 답변에 근거 문장 표시, 빈 입력 안내 메시지 등)
  3. 배운 것을 규칙으로 남기기

    이번 바퀴에서 반복된 실수·기준을 CLAUDE.md에 규칙으로 적어, 다음 PDCA 바퀴가 더 수월하게 합니다.

    프롬프트 
    오늘 구현·검증하면서 반복된 실수나 새로 정한 기준이 있으면,
CLAUDE.md에 규칙으로 한두 줄 추가해줘.
남은 Gap이나 더 하고 싶은 기능은 다음 PDCA 한 바퀴로 넘기세요. Act의 목표는 완벽한 완성이 아니라 "고치고, 배운 것을 규칙으로 남기는" 것입니다.
여기까지: Plan·Design(Day 3) → Do·Check·Act(Day 4)로 첫 PDCA 한 바퀴를 완성했습니다. 이제 결과물을 배포하고 발표·수료로 마무리합니다.
● 마무리 배포 · 발표 · 수료 별도 개념 없이 진행하는 시간입니다
STEP 4 진행 시간 결과물 Vercel 배포 및 발표 준비

완성한 doc-qa-bot을 Vercel에 배포해 인터넷 주소로 만들고, 시연·발표를 준비하는 시간입니다. Day 1에서 익힌 배포 방식을 이번 결과물에 그대로 적용합니다. (별도 학습 개념 없음)

STEP 5 진행 시간 결과물 발표 및 시연

각자 만든 앱을 배포된 주소로 직접 시연하며 무엇을·왜·어떻게 만들었는지 짧게 발표하는 시간입니다. (별도 학습 개념 없음)

STEP 6 진행 시간 전문가 피드백 및 종합 평가/수료

전문가의 피드백과 4일 과정에 대한 종합 평가를 받고 수료하는 시간입니다. (별도 학습 개념 없음)
