Day 4 · bkit PDCA 완성 — 구현·검증·개선과 배포·발표 (이론)

Day 3에서 세운 Plan·Design을 실제로 굴려, 앱을 완성하는 날입니다.
bkit PDCA의 나머지 세 단계 — Do(구현) · Check(검증) · Act(개선) — 의 개념을 정리한 이론 문서입니다. 실제 명령·프롬프트는 실습 문서에서 진행하며, 오후에는 배포·발표·수료로 과정을 마무리합니다.

1. Do — 실제 기능 구현

Do(실행)는 Plan·Design대로 실제로 만드는 단계입니다. Day 3에서 만든 PLAN.md(작업 순서)와 DESIGN.md(구현 방법)를 입력으로, 내 앱의 실제 기능을 하나씩 코드로 옮깁니다.

핵심 흐름부터 — 한 기능이 끝까지 돌게

우리 PRD는 이미 MVP 규모라, Day 4는 그 PRD 전체를 구현합니다. 다만 한 번에 다 만들려 하면 어디서 막혔는지 알기 어렵습니다. 먼저 핵심 기능 하나가 처음부터 끝까지 도는 상태를 만들어 놓고, 그 위에 나머지를 얹는 것이 요령입니다. 이렇게 한 줄기를 끝까지 완성해 두면 다음 단계(Check)에서 무엇을 확인할지도 분명해집니다.

작게 나눠 구현 — 한 번에 한 조각

Claude Code에는 PLAN.md의 작업을 한 번에 하나씩 시킵니다. 큰 덩어리를 통째로 맡기기보다, 작업 단위로 나눠 만들고 그때그때 실행해 확인하면, 문제가 생겨도 방금 바꾼 작은 부분만 보면 됩니다. Day 3에서 세운 하네스(CLAUDE.md 규칙)가 이 과정에서 결과를 일정하게 잡아 줍니다.

앱을 이루는 대표 구현 조각하는 일
화면(UI)사용자가 보고 입력·조작하는 부분
입력 처리사용자가 넣은 값을 받아 다룸
핵심 로직앱의 실제 일을 처리(계산·변환·외부 API 호출 등)
결과 출력·저장처리 결과를 화면에 보여 주거나 파일로 보관

💡 핵심 한 줄: Do는 "한 번에 완성"이 아니라 "작게 만들어 바로 확인"의 반복입니다 — 핵심 흐름 하나부터 끝까지 동작시키세요.

PLAN.md·DESIGN.md를 따라 내 앱을 작업 단위로 하나씩 구현하는 과정은 실습 STEP 1에서 진행합니다.

2. Check — Gap 분석과 설계 대비 검증

Check(검증)는 만든 것이 계획·설계대로인지 확인하는 단계입니다. "동작은 하는데 계획과 다르게 됐다"를 잡아내는 것이 목적이며, 핵심 도구가 Gap 분석입니다.

Gap 분석 — 설계와 구현의 차이 찾기

Gap(갭)은 "틈·차이"를 뜻합니다. Gap 분석은 DESIGN.md에 적은 "이렇게 만들기로 했다"와 실제 구현된 "이렇게 만들어졌다"를 나란히 놓고 어긋난 곳을 찾는 작업입니다. 빠진 기능, 설계와 다르게 동작하는 부분, 아직 처리 못 한 예외 상황이 여기서 드러납니다.

[DESIGN.md]  이렇게 만들기로 했다   ┐
                                   ├─→  차이(Gap) 목록
[실제 구현]  이렇게 만들어졌다      ┘     (빠진 것 · 다른 것 · 못 한 것)

검증 기준 — 무엇이 되면 통과인가

검증은 미리 정한 성공 기준이 있어야 흔들리지 않습니다. Day 3에서 정한 "이번 사이클의 성공 기준"(앱마다 다릅니다)을 그대로 시험지로 삼아, 실제로 그 시나리오를 돌려 봅니다. 규칙·의도대로인지 확인하는 이 과정이 하네스의 검증(Verification) 기둥과 이어집니다.

검증 항목확인 질문
기능 충족PLAN.md의 작업이 실제로 다 됐는가
설계 일치DESIGN.md의 흐름·화면대로 동작하는가
성공 기준정한 성공 시나리오를 통과하는가
예외 처리빈 입력·엉뚱한 입력에도 무너지지 않는가

💡 핵심 한 줄: Check는 "돌아가나?"가 아니라 "계획대로 돌아가나?"를 묻습니다 — 설계와 구현의 Gap을 목록으로 남기세요.

구현 결과와 DESIGN.md를 대조해 Gap 목록을 만들고 성공 시나리오로 검증하는 과정은 실습 STEP 2에서 진행합니다.

3. Act — 오류 수정·개선 반복·기능 고도화

Act(개선)는 Check에서 찾은 문제를 고치고, 더 나아지게 만드는 단계입니다. 그리고 이번 바퀴에서 배운 것을 규칙으로 정착시켜 다음 바퀴가 더 수월하게 합니다.

고치기 → 다시 검증 — 작은 개선의 반복

Gap 목록의 문제를 한 번에 하나씩 고치고, 고칠 때마다 다시 확인합니다. 한꺼번에 여러 곳을 바꾸면 무엇이 문제를 만들었는지 알기 어렵습니다. "고친다 → 검증한다 → 또 고친다"의 작은 반복이 Act의 리듬입니다.

고도화 — 되는 것을 더 낫게

기본 동작이 안정되면, 우선순위가 높은 것부터 기능을 더하거나 다듬습니다. 다만 욕심을 내 한 번에 많이 바꾸기보다, 다음 PDCA 한 바퀴로 넘겨 작게 굴리는 편이 안전합니다. "여기서 또 실수하네" 싶은 것은 CLAUDE.md에 규칙으로 적어 두면, 하네스가 함께 단단해집니다.

[Check의 Gap 목록]
   → 하나 고치기 → 다시 검증 → 통과? → 다음 항목
                                  └ 아니오 → 다시 고치기
   → (여유가 있으면) 우선순위 높은 기능 하나 고도화
   → 배운 규칙을 CLAUDE.md에 반영  → 다음 PDCA 바퀴로

💡 핵심 한 줄: Act는 "완벽하게 끝내기"가 아니라 "고치고, 배운 것을 규칙으로 남기기"입니다 — 나머지는 다음 바퀴로.

Gap을 하나씩 고치고 재검증하며 기능을 다듬는 과정은 실습 STEP 3에서 진행합니다.

4. 결과물 Vercel 배포 및 발표 준비

완성한 내 앱을 Vercel에 배포해 인터넷에서 접속되는 주소로 만듭니다. 배포 명령(vercel --prod)은 Claude Code가 내장 터미널에서 대신 실행하고, 사람은 최초 1회 vercel login(브라우저 승인)만 하면 됩니다 — 웹사이트에 들어갈 필요가 없습니다. 이후 수정할 때마다 "다시 배포해줘" 한마디로 갱신됩니다.

서버(api/ 함수) — 비밀 키를 브라우저에서 감추는 중간 다리

비밀 키는 절대 브라우저(사용자 화면)로 보내면 안 됩니다. 정적 HTML에서 외부 API를 직접 부르면 키가 화면 코드에 실려 누구나 열어 볼 수 있기 때문입니다. 그래서 중간에 내 서버(api/ 폴더의 서버리스 함수)를 두어, 브라우저는 내 함수만 부르고 키는 그 함수 안에서만 씁니다. 즉 .env·환경변수는 "키를 코드 밖에 보관"하는 곳이고, api/ 함수는 "그 키로 대신 일해 주는 서버"라 — 대체재가 아니라 함께 쓰는 별개의 장치입니다.

[브라우저]  ──질문──▶  [내 api/ 함수]  ──질문+키──▶  [외부 API]
 (키 없음)               (여기서만 키 사용)               │
      ◀──── 답변 ──────────◀──── 답변 ──────────────────┘

환경변수 — 배포본에 비밀 키를 안전하게 전달 (비밀 키를 쓰는 앱만)

.env 파일은 규칙대로 공유·커밋하지 않으므로 배포본에 키가 담기지 않습니다. 대신 Vercel에 비밀 키(예: OPENAI_API_KEY)를 환경변수로 등록하면(vercel env add 명령을 Claude가 실행하고 값만 본인이 입력), 서버리스 함수가 실행될 때 그 값을 안전하게 읽어 옵니다. 키를 코드나 정적 HTML에 직접 넣으면 노출되므로 절대 금지입니다.

배포 단계하는 일
① 준비 점검서버리스 함수는 api/에, 키는 환경변수로만 읽는지 확인·정리 (🤖 Claude Code)
② Vercel 로그인 (최초 1회)vercel login — 브라우저에서 본인이 승인 (🧑 1회)
③ 배포Claude가 vercel --prod 실행 → 공개 주소 (🤖 Claude Code)
④ 키 등록·재배포 (키 쓰는 앱만)vercel env add(값만 본인 입력) → 다시 vercel --prod (🤖 Claude, 값 🧑)

💡 핵심 한 줄: vercel login 한 번이면, 그다음부터는 "배포해줘" 한마디로 Claude가 배포 — 비밀 키만 코드가 아니라 Vercel 환경변수에 두세요.

Vercel CLI로 (Claude가) 배포하고 필요하면 키를 환경변수로 등록하는 구체적 순서는 실습 STEP 4에서 진행합니다.
