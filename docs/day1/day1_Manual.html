2. 온라인 서비스의 이해 — GitHub · Vercel · OpenAI · Supabase

GitHub — 코드를 보관하고 공유하는 원격 저장소

Git이 내 컴퓨터 안에서 이력을 관리한다면, GitHub는 그 코드를 인터넷 저장소(repository)에 올려 보관·공유·협업할 수 있게 해주는 서비스입니다. 뒤에서 사용할 Vercel이 GitHub 계정과 연동되기 때문에 함께 준비합니다.

Vercel — 만든 웹페이지를 인터넷에 공개하는 배포 서비스

내 컴퓨터에만 있는 웹페이지는 나만 볼 수 있습니다. Vercel은 그 파일을 자사 서버에 올려 누구나 접속할 수 있는 인터넷 주소(URL)로 만들어 주는 배포·호스팅 서비스입니다. "로컬(내 컴퓨터)"과 "배포(인터넷 공개)"의 차이는 5장에서 더 다룹니다.

API Key — 프로그램이 AI를 불러 쓰기 위한 비밀 열쇠

사람이 화면에서 로그인하듯, 프로그램이 외부 AI 서비스를 호출할 때는 "누가 요청했는지"를 증명해야 합니다. 그 증명 수단이 API Key(sk-로 시작하는 긴 문자열)입니다. 이후 실습에서 OpenAI 기능을 프로그램으로 불러 쓸 때 필요하므로 미리 발급해 둡니다.

API Key는 비밀번호와 같습니다. 코드·채팅·공개 저장소에 노출하면 타인이 내 계정으로 요금을 발생시킬 수 있습니다.
발급 화면을 벗어나면 다시 볼 수 없으므로 안전하게 보관하고, 유출이 의심되면 즉시 삭제(Revoke) 후 재발급해야 합니다.
서비스비유역할
GitHub코드 보관 창고코드를 인터넷에 저장·공유
Vercel공개 게시대웹페이지를 인터넷 주소로 배포
API Key출입 열쇠프로그램이 AI를 호출할 때 신원 증명
세 계정을 실제로 가입하고 API Key를 발급하는 과정은 실습 STEP 1에서 진행합니다.

Supabase — 데이터를 저장하는 클라우드 데이터베이스

위 세 서비스가 "코드 보관 · 배포 · AI 호출"을 맡는다면, Supabase는 회원 정보나 앱의 데이터를 인터넷 너머에 저장해 두는 클라우드 데이터베이스(백엔드) 서비스입니다. 내 컴퓨터를 꺼도 데이터가 남아 있어, 여러 사람이 각자 로그인해서 자기 데이터를 보고 쓰는 앱을 만들 수 있습니다. 무료로 시작할 수 있으며, 접속에는 Project URL과 anon public 키(환경변수 SUPABASE_URL·SUPABASE_ANON_KEY)를 사용합니다.

이 키들이 정확히 무엇인가요? 프로젝트를 만들면 대시보드 Project Settings → API 화면에 접속 정보가 모여 있습니다. 우리 실습에서 쓰는 건 앞의 두 개(SUPABASE_URL·SUPABASE_ANON_KEY)이고, 함께 보이는 나머지 키들은 아래 표로 정리합니다.

키위치성격설명 · 이 과정에서의 쓰임
Project URL
(SUPABASE_URL)		Settings → API공개내 저장소가 있는 인터넷 주소예요(예: https://xxxx.supabase.co). 앱이 이 주소를 보고 어디에 저장·조회할지 찾아갑니다. 실습에서 사용.
anon public
(SUPABASE_ANON_KEY)		Settings → API공개(제한적)저장소에 드나들 때 쓰는 일반 출입 열쇠. 권한이 제한돼 있어 비교적 안전하지만, 이 과정에서는 그래도 .env 파일 안에만 보관합니다. 실습에서 사용.
service_roleSettings → API비밀 · 노출 금지무엇이든 다 할 수 있는 관리자 열쇠. 이게 새어 나가면 남이 내 저장소를 통째로 열어보고 지울 수 있어 아주 위험합니다. 이 과정에서는 쓰지 않습니다.
JWT SecretSettings → API비밀로그인한 사람에게 주는 출입증(토큰)이 진짜인지 확인하는 열쇠예요. 서버가 "이 사람 로그인한 게 맞네" 하고 알아보는 데 씁니다. 이 과정에서는 직접 다루지 않습니다.
DB Password ·
Connection string		Settings → Database비밀저장소에 직접 로그인하는 비밀번호와 주소. 관리자 열쇠만큼 강력해서 남에게 알려주면 안 됩니다. 이 과정에서는 쓰지 않습니다.
service_role·JWT Secret·DB 비밀번호는 절대 브라우저나 GitHub에 올리지 마세요. 이 키들이 유출되면 남이 내 데이터베이스를 통째로 읽고 지울 수 있습니다. 공개용인 anon public조차도 이 과정에서는 로컬은 .env, 배포는 Vercel 환경 변수에만 보관합니다.

이 과정에서는 Supabase의 자체 로그인 기능(Auth) 대신, 직접 만든 user 테이블로 회원가입·로그인을 처리합니다. 이때 비밀번호는 반드시 해시(암호화)로 저장해, 데이터베이스가 유출되어도 원래 비밀번호를 알 수 없게 합니다.

서비스비유역할
Supabase회원·데이터 창고로그인 정보와 데이터를 클라우드에 저장
Supabase 키도 API Key처럼 코드·공개 저장소에 직접 적지 않습니다. 로컬에서는 .env, 인터넷 배포에서는 Vercel 환경 변수에 보관합니다.

로그인(멀티유저)은 왜 필요한가 — 배포된 챗봇과 API 비용

챗봇처럼 내 API Key로 동작하는 앱을 인터넷에 배포하면, 그 주소를 아는 누구나 그 챗봇을 사용할 수 있습니다. 그런데 다른 사람이 챗봇을 쓸 때마다 내 OpenAI 키가 호출되어 요금은 나에게 청구됩니다. 로그인 장치가 없으면 링크를 받은 사람은 물론 우연히 주소를 발견한 사람까지 내 비용으로 챗봇을 마음껏 쓸 수 있고, 누군가 악의적으로 대량 호출하면 요금이 순식간에 눈덩이처럼 불어날 수 있습니다.

여기서 헷갈리기 쉬운 두 가지를 나눠서 봐야 합니다.

  • ① 열쇠(키)를 도둑맞는 문제 — 뒤(5장)에서 배울 .env·.gitignore는 비밀 키가 코드에 그대로 적혀 남에게 보이는 것을 막아, 남이 내 키를 복사해 가져가지 못하게 해 줍니다.
  • ② 내 열쇠로 아무나 드나드는 문제 — 키를 아무리 잘 숨겨도, 인터넷에 공개한 챗봇을 아무나 쓸 수 있으면 그 사람들이 챗봇을 쓸 때마다 내 키로 요금이 빠져나갑니다. 누가 키를 훔친 게 아닌데도, 결국 내 돈으로 남이 AI를 쓰는 셈이죠.

즉, 키를 숨기는 것(.env)만으로는 ②를 막을 수 없습니다. 인터넷에 공개할 때는 "누가 쓸 수 있는지"까지 정하는 장치가 따로 필요합니다.

그래서 로그인(멀티유저)을 넣습니다. 가입·로그인한 사람만 챗봇을 쓰게 하면 아무나 내 API를 소모하지 못하도록 문을 잠글 수 있습니다. Supabase로 user 테이블을 만들어 회원을 관리하는 이유가 바로 이것입니다.

구분로그인 없이 배포로그인(멀티유저) 적용
누가 쓸 수 있나주소를 아는 누구나가입·로그인한 사람만
API 비용타인이 써도 내가 부담 · 폭탄 위험허용된 사용자로 한정
막는 것—무단 사용(비용 소모)
.env·.gitignore가 키를 훔쳐 가지 못하게 막는다면, 로그인은 아무나 내 키를 써서 비용을 발생시키지 못하게 막습니다. 배포하는 순간 "주소를 아는 누구나 접속 가능"해지므로, 내 API로 동작하는 앱에는 접근을 제한하는 로그인이 사실상 필수입니다.
Supabase 가입(GitHub 연동)은 실습 STEP 1에서, 프로젝트 생성과 로그인·멀티유저 붙이기는 STEP 6에서 진행합니다.

3. 개발 도구의 이해 — Node.js와 Git

Node.js — 개발 도구를 돌리는 실행 환경

Node.js는 원래 웹 브라우저 안에서만 동작하던 자바스크립트를, 컴퓨터에서 직접 실행할 수 있게 해주는 프로그램입니다. 이번 과정에서 사용하는 도구(Vercel 명령어 등)는 Node.js와 함께 설치되는 npm이라는 "도구 설치기"를 통해 내려받습니다. 그래서 Node.js가 미리 준비되어야 합니다.

npm(Node Package Manager)은 앱스토어처럼 개발 도구를 이름만으로 설치·관리해 주는 도구입니다. 실습에서 npm install 명령이 자주 나오는 이유가 바로 이것입니다.

Git — 코드의 변경 이력을 지키는 안전벨트

Git은 파일이 언제·무엇이 바뀌었는지 이력을 기록하고, 필요하면 과거로 되돌릴 수 있게 해주는 버전 관리 도구입니다. 하나의 저장 지점을 "커밋(commit)"이라 하며, 이 커밋 이력이 쌓여 프로젝트의 역사가 됩니다. 뒤에서 다룰 GitHub·Vercel 배포는 모두 이 Git을 바탕으로 동작합니다.

도구한 줄 정의이번 과정에서의 쓰임
Node.js자바스크립트 실행 환경Vercel CLI 등 도구 설치의 토대
npm도구 설치·관리기npm install로 도구를 내려받음
Git코드 변경 이력 관리기GitHub 업로드·Vercel 배포의 바탕
이 개념들을 실제로 설치하고 node -v·git --version으로 확인하는 과정은 실습 STEP 2에서 진행합니다.

4. Claude와 Claude Desktop의 이해

Free / Pro — LLM 구독이란

Claude는 무료(Free)로도 채팅과 데스크톱 앱을 쓸 수 있지만, 실습을 넉넉한 사용량으로 원활히 진행하려면 유료 구독(Pro 이상)을 권장합니다. 구독은 "더 많은 사용량과 추가 기능을 쓰기 위한 월 단위 이용권"이라고 이해하면 됩니다.

구분Free 플랜Pro 플랜

이용 범위웹·앱 채팅, 기본 기능채팅 + 넉넉한 사용량과 최신 모델 우선 이용
사용량기본Free 대비 대폭 확대, 혼잡 시 우선 접속

Claude Desktop — 대화만 하는 AI에서, 내 파일을 함께 다루는 AI로

Claude Desktop은 웹 브라우저 없이 내 컴퓨터에 설치해 쓰는 Claude 데스크톱 앱입니다. 기본적으로는 claude.ai 채팅처럼 "대화"를 하는 도구지만, 내 컴퓨터에 작업 폴더 하나를 만들어 그 폴더에서 작업해 달라고 하면 대화를 넘어 실제로 그 폴더 안의 파일을 만들고 수정하게 할 수 있습니다. 즉 "말로 설명해 주는 AI"에서 "직접 작업해 주는 AI"로 한 걸음 나아갑니다.

로컬 작업 폴더 — Claude가 내 폴더에서 직접 파일을 만들게 하기

내 컴퓨터에 작업 폴더 하나를 만들고 Claude에게 그 폴더에서 작업해 달라고 하면, Claude가 그 폴더 안에서 파일을 읽고 만들고 고칠 수 있습니다. 이번 과정에서는 실습 전용 폴더(예: hello-page) 하나를 만들어, 그 안에서 웹페이지 파일을 만들어 갑니다.

Claude Desktop내 컴퓨터에 설치하는 Claude 앱한국어로 지시해 코드를 만드는 작업 공간
로컬 작업 폴더Claude가 파일을 다루는 실습 전용 폴더hello-page 폴더에 파일 생성·수정
Claude에게 작업을 맡길 때는 실습 전용 폴더(예: hello-page)에 한해 파일을 다루게 하는 것이 안전합니다. 시스템 폴더나 개인 문서 폴더 전체에서 작업하게 하지 마십시오.

5. 바이브 코딩과 배포의 이해

Vibe Coding — 자연어로 소프트웨어를 만든다는 것

바이브 코딩(Vibe Coding)은 프로그래밍 문법을 직접 타이핑하는 대신, "이런 걸 만들어 줘"라고 자연어로 지시해 AI가 코드를 대신 작성하게 하는 방식입니다. 코드를 한 줄도 몰라도 결과물을 만들 수 있다는 것이 핵심이며, 이번 실습의 헬로페이지가 그 첫 경험이 됩니다.

Prompt — 좋은 결과를 부르는 지시문의 원리

같은 도구라도 어떻게 지시하느냐에 따라 결과가 크게 달라집니다. 맥락(context)을 구체적으로 줄수록 결과가 좋아진다는 원리가 여기에 그대로 적용됩니다.
좋은 프롬프트는 대체로 아래 세 요소를 담습니다.

요소의미예
역할어떤 입장에서 작업할지 지정"너는 웹 디자이너야"
구체성무엇을·어떤 형태로 원하는지 명시"제목, 소개 문단, 목록 3개를 넣어"
제약스타일·범위·조건을 한정"파란색 계열로 깔끔하게"

deploy — 로컬과 배포의 차이

내 컴퓨터 안의 파일을 브라우저로 여는 것은 "로컬(local)" 상태로, 나만 볼 수 있습니다. 이 파일을 Vercel 같은 서비스로 올려 인터넷 주소를 부여하는 것을 "배포(deploy)"라 하며, 그 순간부터 그 주소를 아는 누구나 접속할 수 있습니다. 실습의 마지막 단계에서 헬로페이지를 실제로 배포해, 로컬 → 인터넷 공개의 전 과정을 경험합니다.
"누구나 접속 가능"은 편리하지만, 내 API Key로 동작하는 챗봇이라면 아무나 써서 내 비용이 새어 나갈 수 있습니다 — 그래서 로그인으로 접근을 제한합니다(2장 "로그인은 왜 필요한가" 참고).

.env — 비밀 값을 코드와 분리해 보관하는 파일

API Key나 비밀번호 같은 민감한 값을 코드 안에 직접 적어 두면, 그 코드를 공유하거나 배포하는 순간 값도 함께 노출됩니다. 그래서 이런 값은 .env(environment, 환경변수)라는 별도 파일에 OPENAI_API_KEY=sk-...처럼 이름=값 형태로 따로 적어 두고, 코드는 그 파일에서 값을 읽어 쓰도록 만듭니다. 이렇게 하면 비밀 값과 코드가 분리되어, 코드를 고치거나 공유할 때 실수로 키가 딸려 나갈 위험이 줄어듭니다.
참고로 .env 파일은 서버 프로그램(예: Node.js)이 실행 중일 때 그 서버가 읽어옵니다. 그래서 이번 실습에서도 로컬 서버를 켜서(localhost) .env의 키를 읽어 챗봇을 동작시키고, 인터넷에 배포할 때는 .env 대신 Vercel의 환경변수(Environment Variables)에 같은 키를 등록합니다 — 형태만 다를 뿐 "비밀 값은 서버가 읽는다"는 원리는 같습니다.

.gitignore — "이 파일은 올리지 마"라고 알려 주는 목록

Git·GitHub은 폴더 안의 파일을 추적해서 저장소에 올립니다. 이때 .gitignore라는 파일에 이름을 적어 두면, Git이 그 파일·폴더를 추적 대상에서 제외해 저장소에 올리지 않습니다. 예를 들어 .gitignore에 .env 한 줄을 넣어 두면, 방금 만든 비밀 값 파일이 GitHub 같은 공개 저장소에 실수로 올라가는 것을 막아 줍니다.

정리하면 — ① 비밀 값은 .env에 따로 두고 → ② .gitignore에 .env를 적어 올라가지 않게 하는 두 단계가, 키를 안전하게 다루는 기본 원칙입니다.

API Key가 공개 저장소(GitHub 등)나 배포된 파일에 그대로 올라가면, 그 주소를 아는 누구나 내 키로 요금을 발생시킬 수 있습니다. 비밀 값은 반드시 .env로 분리하고 .gitignore에 .env를 등록해, 공개 저장소나 배포 파일에 키가 딸려 올라가지 않게 하십시오.

💡 핵심 한 줄: 자연어로 만들고(바이브 코딩), 클릭 몇 번으로 세상에 공개하는(배포) 흐름 — 이 개념을 손으로 확인하는 것이 Day 1 실습의 목표입니다. 이때 비밀 값(.env)은 코드와 분리해 안전하게 다루는 것이 함께 지켜야 할 원칙입니다.
